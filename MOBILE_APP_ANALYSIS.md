# Аналіз 1С-конфігурацій L-TEX як основа ТЗ нового мобільного додатку (React Native / Expo)

> **Призначення документа.** Цей звіт є технічним аудитом двох 1С-конфігурацій (центральної облікової бази та існуючого мобільного клієнта на 1С:Підприємство 8.3 Мобільна платформа), що складають backend для роботи менеджерів L-TEX. Документ має бути основою для:
> 1. Написання OpenAPI-специфікації нового backend-gateway (Next.js/FastAPI).
> 2. Розробки story-map і скрін-флоу нового RN-додатка.
> 3. Прийняття рішень про межу між «дзеркалити поведінку існуючого 1С-додатку» та «переробити з нуля».
>
> **Що НЕ є метою документа.** Виправлення коду 1С, нормативна оцінка архітектури, бізнес-консалтинг, security-аудит. Усе позначене `Unclear / TBD:` потребує живого уточнення у власника бізнесу / 1С-програміста.
>
> **Скорочення.** ЦБ — центральна 1С-база; МД — мобільний 1С-додаток (`MobileAgentLTEX`); РН — новий React Native додаток; УТ — типова конфігурація «1С:Управление торговлей»; РИБ — Распределённая Информационная База (механізм реплікації 1С); ПКО — Приходный Кассовый Ордер; РТУ — Реализация Товаров и Услуг.

## 1. Огляд конфігурацій

У репозиторії `1c-export/` є два каталоги:

| Каталог | Конфігурація | Призначення |
|---|---|---|
| `1c-export/Мобільний додаток/` | **MobileAgentLTEX** v1.15.3 | Поточний мобільний 1С-додаток для менеджерів («торговий агент»). Працює як автономний клієнт на 1С:Підприємство 8.3 Mobile Platform. Синхронізується з ЦБ через SOAP. |
| `1c-export/Центральна 1с/` | На базі **1С:Управление торговлей** (УТ для України) — **точну редакцію визначити неможливо** (`Configuration.xml` у експорті відсутній, є лише його дочірні об'єкти). | Центральна облікова база L-TEX. Зберігає номенклатуру, контрагентів, замовлення, розрахунки, інтегрується з сайтом, бухгалтерією, роздрібом, Viber-ботом, Push-сервісом. |

### 1.1. Мобільний додаток `MobileAgentLTEX`

**Файл:** `1c-export/Мобільний додаток/Configuration.xml:1-141`

```xml
<Properties>
    <Name>MobileAgentLTEX</Name>
    <Synonym>Mobile agent LTEX</Synonym>
    <ConfigurationExtensionCompatibilityMode>Version8_3_14</ConfigurationExtensionCompatibilityMode>
    <DefaultRunMode>ManagedApplication</DefaultRunMode>
    <UsePurposes>
        <Value>PlatformApplication</Value>
        <Value>MobilePlatformApplication</Value>
    </UsePurposes>
    <ScriptVariant>Russian</ScriptVariant>
    <DefaultRoles><Item>Role.Пользователь</Item></DefaultRoles>
    <Vendor>Intrata</Vendor>
    <Version>1.15.3</Version>
    <DefaultLanguage>Language.Українська</DefaultLanguage>
    <DataLockControlMode>Managed</DataLockControlMode>
    <CompatibilityMode>Version8_3_14</CompatibilityMode>
    <DefaultSearchForm>CommonForm.ФормаПодбораНоменклатуры</DefaultSearchForm>
</Properties>
```

**Ключові факти:**
- Платформа: **1С:Підприємство 8.3.14** (мінімальна сумісність) — режим керованих форм (Managed), Taxi-інтерфейс. Підтримує і десктопний, і мобільний клієнт (`PlatformApplication + MobilePlatformApplication`).
- Мови: основна — **українська** (`Language.Українська`), вторинна — російська. Скрипт-варіант (мова коду 1С) — **Russian** (тобто ключові слова `Процедура`, `Функция`, `Если`, `Для Каждого`).
- Розробник: **Intrata** (зовнішній підрядник, не L-TEX). Версія додатку зашита в `Метаданные.Версия` і відправляється на сервер при кожному обміні.
- Default-роль для всіх користувачів — `Пользователь` (єдина «робоча» роль; див. §3).
- Default-форма пошуку — `CommonForm.ФормаПодбораНоменклатуры` (тобто будь-яке `F11`-подібне відкриття списку номенклатури з табличного поля одразу веде в кастомний пікер).
- **Дозволи на мобільні API** (`RequiredMobileApplicationPermissions`):
  - `Multimedia` = false (камера/галерея вимикнено в маніфесті, але в коді є виклики сканування ШК — див. §5).
  - `Location` = false, `BackgroundLocation` = false (хоча в коді є `Геопозиціювання.ЗаписатьКоординаты()` — див. §6).
  - `Telephony` = false, `Contacts` = false.
  - `PushNotification` = **true**, `LocalNotification` = **true**, `Print` = **true**, `BackgroundAudioPlayback` = **true**, `FileExchangeWithPersonalComputer` = **true**.

> **TBD:** `Multimedia=false` і `Location=false` виглядають дивно, бо обидві можливості активно використовуються у коді (сканер штрих-кодів у формі характеристики, гео-логування у формі робочого столу). Можливо, це просто застарілий маніфест — на реальному пристрої ці права запитуються динамічно. Уточнити у Intrata.

### 1.2. Центральна 1С

**`Configuration.xml` у експорті відсутній** — є тільки дочірні об'єкти. Ідентифікувати точну редакцію («УТ 11», «УТ для України 3.x» тощо) можна тільки за непрямими ознаками:

- Файл `1c-export/Центральна 1с/Ext/SessionModule.bsl:1-12`:
  ```bsl
  Процедура УстановкаПараметровСеанса()
      ПараметрыСеанса.ОбщиеЗначения = Новый ХранилищеЗначения(Новая Структура);
      ПолныеПрава.ОпределитьПараметрыСеансаДляОбменаДанными();
      ПолныеПрава.УстановитьПараметрыСеанса();
  КонецПроцедуры
  ```
  → стандартний для «Управление торговлей для Украины 8» патерн: загальний модуль `ПолныеПрава` як централізована точка налаштування сеансу.
- `1c-export/Центральна 1с/Ext/ExternalConnectionModule.bsl:1-29` — типова обгортка `глОбщиеЗначения` (стиль УТ для України 2.3).
- `1c-export/Центральна 1с/Ext/ParentConfigurations.bin` — двійковий файл, прочитати без 1С-Конфігуратора неможливо. **TBD:** в 1С відкрити «Конфігурація → Підтримка → Інформація про оновлення» і записати точне ім'я постачальника / редакцію.
- Стандартні підсистеми (див. нижче): `ПодсистемаПолученияАналитическойИнформации`, `УправлениеЗакупками`, `УправлениеПродажами`, `УправлениеРозничнойТорговлей` тощо — це лексикон **УТ 10.3 для України / УТ для України 8** (стара лінійка, не «УТ 11»).
- Україномовні поля: `УчастьВЧаті`, `КолірІмені`, `ВідправленоВВайбер` — отже конфіг або форк з локалізацією, або типова УТ для України з користувацькими доробками.
- Версії документів і регістрів виглядають як «1С 8.3 + керовані форми + сумісність 8.2» (перевірити окремо).

> **Висновок:** ЦБ — це або **«1С:Управление торговлей для Украины 8» (на платформі 8.3)**, або форк УТ 10.3 з накладанням рашоретного шару української локалізації. Точну редакцію та поточний реліз постачальника треба з'ясовувати в 1С Конфігураторі. Для нашого ТЗ достатньо знати, що це **типова УТ-подібна архітектура** з типовими патернами обміну.

## 2. Модель даних (для backend нового додатку)

> **Як читати цю секцію.** Назви типів — як в 1С: `String(N)`, `Boolean`, `Number(M,K)`, `Date`, `DateTime`, `UUID`, `CatalogRef.X`, `DocumentRef.X`, `EnumRef.X`. У RN/JSON їх треба мапити так:
> - `CatalogRef.X` / `DocumentRef.X` → JSON `string` (UUID v4) + опціонально вкладений об'єкт.
> - `EnumRef.X` → JSON `string` (значення енуму) або `enum`.
> - `Number(15,2)` → JSON `number` (з обережністю — у 1С це `decimal`, не `float`; для сум використовувати рядок або bigint).
> - `Date` (без часу) → ISO 8601 `YYYY-MM-DD`.
> - `DateTime` → ISO 8601 `YYYY-MM-DDTHH:mm:ssZ`.

### 2.1. Архітектура даних мобільного додатку

Мобільний додаток має **локальну SQLite-подібну базу 1С** (вмонтовану в платформу). На відміну від ЦБ він має **спрощену модель**: лише ті об'єкти, що потрібні менеджеру в полях; усе зайве (бухоблік, ПДВ, регістри собівартості тощо) видалено.

Деякі об'єкти **логічно одні й ті ж самі** в МД та ЦБ, але мають **різні набори реквізитів**:

| Логічна сутність | Об'єкт у МД | Об'єкт у ЦБ |
|---|---|---|
| Клієнт | `Catalog.Контрагенты` (МД) | `Catalog.Контрагенты` (ЦБ) |
| Товар | `Catalog.Номенклатура` (МД) | `Catalog.Номенклатура` (ЦБ) |
| Варіант / партія товару | `Catalog.ХарактеристикиНоменклатуры` | `Catalog.ХарактеристикиНоменклатуры` |
| Замовлення клієнта | `Document.Заказ` (МД) | `Document.ЗаказПокупателя` (ЦБ) |
| Реалізація товарів | `Document.РеализацияТоваровУслуг` | `Document.РеализацияТоваровУслуг` |
| Повернення від покупця | `Document.ВозвратОтПокупателя` | `Document.ВозвратТоваровОтПокупателя` |
| Грошовий ордер | `Document.КассовыйОрдер` (єдиний для приходу і витрати) | `Document.ПриходныйКассовыйОрдер` + `Document.РасходныйКассовыйОрдер` |
| Маршрутний лист | `Document.МаршрутныйЛист` | (відсутній — маршрут це фіча мобільного шару) |
| Презентація товару клієнту | `Document.Презентація` | (відсутній — «Презентация» — це задача-комунікація) |

> Це означає, що при синхронізації ЦБ → МД одне `ЗаказПокупателя` з ЦБ перетворюється на `Заказ` у МД (відкидаються ПДВ, валюта документа, договір контрагента, кур валюти і т. ін.), а при МД → ЦБ зворотна трансляція робиться на стороні веб-сервісу `MobileExchange`.

```bsl
Перем мЭтоОбмен Экспорт;

Процедура ПриЗаписи(Отказ)
    Если мЭтоОбмен <> Истина Тогда 
        Общий.ЗарегистрироватьИзменения(ЭтотОбъект.Ссылка);
    КонецЕсли;
КонецПроцедуры

Процедура ПередЗаписью(Отказ)
    // Установка дати створення для нових
    Если ЭтоНовый() И Не ЗначениеЗаполнено(ДатаСоздания) Тогда
        ДатаСоздания = ТекущаяДата();
    КонецЕсли;
    ДополнительныеСвойства.Вставить("ЭтоНовый", ЭтоНовый());

    // Освіжаємо «РаботаСКлиентом» — останнє повідомлення / коментар
    Если НЕ ЭтоНовый() Тогда  
        // зчитуємо останній зріз з регістру РаботаСКлиентом
        // ...
        КонтрольВерсийПерезапису(); // якщо ДатаОновлення на сервері > локальної — відкочуємо локальні зміни
    КонецЕсли;

    // Реєстрація факту перейменування для аудиту в РегистрацияИзмененияКонтрагента
    Общий.ЗарегистрироватьИзмененияКонтрагента(
        Ссылка, Ссылка.Наименование, Наименование, 
        Ссылка.ДатаОновлення, ДатаОновлення);
КонецПроцедуры
```

Ключове:
- **Прапорець `мЭтоОбмен`** — відключає реєстрацію змін під час прийому пакетів з сервера, щоб не було рекурсії «отримали → зареєстрували → відправили назад».
- **`КонтрольВерсийПерезапису()`** — реалізація **last-write-wins з пріоритетом серверу** (якщо `ДатаОновлення` на сервері новіша — локальні зміни тіла відкидаються, але CRM-поля типу «РаботаСКлиентомПериод», «КонтактВайбер», «ПодписанийВВайбере» зберігаються).
- Аудит зміни найменування пишеться в `РегистрСведений.РегистрацияИзмененияКонтрагента` (поля: `НаименованиеСтар/Нов`, `ДатаОновленняСтар/Нов`).

Структура близька до `Document.Заказ`:

- Header: ті самі поля + бізнесові ознаки (КурсEUR, КурсUSD, СумаОплатиНаложкою, МаршрутныйЛист, Сделка, Неплановая та інші — повний перелік потрібно отримати з XML).
- TabularSection `Товары`: ідентична до `Заказ.Товары` + поле `Знижка` (знижка по позиції).
- Numerator: 11 chars, yearly.
- Реєстри: **порожні** (рухи списання запасів і виручки виконуються після обміну на стороні ЦБ при проведенні `Реализации`).

**Бізнес-логіка:**

```bsl
Процедура ПередЗаписью(Отказ, РежимЗаписи, РежимПровідення)
    Если Не Архив Тогда Архив = ЭтотОбъект.ПометкаУдаления; КонецЕсли;
    
    // Авто-розрахунок ціни/суми
    Для Каждого Стр Из Товары Цикл
        Если Стр.Цена = 0 Тогда 
            Стр.Цена = Стр.ЦенаПродажиВес * Стр.ХарактеристикаНоменклатуры.Вага;
        КонецЕсли;
        Если Стр.Сумма = 0 Тогда 
            Стр.Сумма = Стр.Цена * Стр.Количество;
        КонецЕсли;
    КонецЦикла;
    Сума = Товары.Итог("Сумма");
КонецПроцедуры
```

> Це те, що бекенд має зробити сам, якщо менеджер у RN-додатку забув ввести ціну, але вказав «вага × ціна за кг».

### 2.8. Регістри накопичення мобільного додатку

> У 1С регістр накопичення = транзакційна таблиця руху ресурсів (типу General Ledger). У мобільному їх лише **3**.

#### 2.8.1. `AccumulationRegister.ОстаткиТоваров`

- **Тип:** Остатки (Balance).
- **Измерения (Dimensions):** `Номенклатура`, `ХарактеристикаНоменклатуры`, `ЕдиницаИзмерения`.
- **Ресурсы (Resources):** `Количество` (Number 12,2).
- **Хто пише:** `Document.Возврат` (Расход), `Document.Приход` (Приход), `Document.ВозвратОтПокупателя` (Расход), а також синхронізатор `ОновитиЗалишкиТаЦіни` (через створення «технічного» приходу/витрати).

#### 2.8.2. `AccumulationRegister.Продажи`

- **Тип:** Обороты (Turnover).
- **Измерения:** `Номенклатура`, `ХарактеристикаНоменклатуры`, `ЕдиницаИзмерения`.
- **Ресурсы:** `Количество` (12,2), `Цена` (12,2), `ЦенаПартнера` (12,2).
- **Хто пише:** `Document.ВозвратОтПокупателя` (мінусом).
- **Хто читає:** `Report.Продажи` (через `ФормаОтчетПродажи`).

#### 2.8.3. `AccumulationRegister.ДенежныеСредства`

- **Тип:** Обороты.
- **Измерения:** `ВидДвиженияДенежныхСредств`.
- **Ресурсы:** `Сумма` (Number 12,2).
- **Хто пише:** `Document.КассовыйОрдер`.

> **Висновок:** мобільний додаток **майже не веде складського обліку** — це просто кеш для відображення. Реальний облік ведеться в ЦБ. Регістр `Продажи` фактично «обірваний» (його заповнює лише повернення).

## 3. Авторизація і ролі

### 3.1. Як ідентифікуються користувачі у мобільному

У МД **немає звичайної автентифікації** — додаток працює під одним технічним користувачем `Пользователь` (роль `Role.Пользователь`):

**Файл:** `1c-export/Мобільний додаток/CommonModules/УправлениеПользователями/Ext/Module.bsl:1-19`

```bsl
Функция СоздатьПользователяНаСервере() Экспорт
    Если ПользователиИнформационнойБазы.НайтиПоИмени("Пользователь") = Неопределено Тогда 
        НовыйПользователь = ПользователиИнформационнойБазы.СоздатьПользователя();
        НовыйПользователь.Имя = "Пользователь";
        НовыйПользователь.ПолноеИмя = "Пользователь";
        НовыйПользователь.Роли.Добавить(Метаданные.Роли.Найти("Пользователь"));
        НовыйПользователь.Записать();
        Возврат Истина;
    Иначе
        Возврат Ложь;
    КонецЕсли;
КонецФункции
```

Викликається з `ManagedApplicationModule.ПередНачаломРаботыСистемы()`. Тобто **на стороні мобільного — анонімний доступ до локальної БД**.

**Реальна ідентифікація менеджера** відбувається на стороні сервера, через **`Константы.ПарольВхода`** — це по суті **shared secret пристрою**. Сервер шукає цього менеджера в ЦБ за паролем:

```bsl
// ЦБ → Module ОбменАндроид
Продавец = ОбменАндроид.ПолучитьПродавцаПоКодуПартнера(ПарольВхода);
```

`Продавец` тут — це конкретний `Catalog.Пользователи` (або пов'язаний `ФизЛицо`/`ТорговыйАгент`). **TBD:** з'ясувати, **де саме в ЦБ зберігається відповідність «password → Менеджер»** — найімовірніше в `Catalog.ТорговыеАгенты` чи в кастомному регістрі. Перевірити в коді ЦБ модуля `ОбменАндроид`.

#### 3.2.1. `Role.Пользователь`

**Файл:** `1c-export/Мобільний додаток/Catalogs/РабочийСтол/Forms/ФормаСписка/Ext/Form/Module.bsl:32-44`

> **Висновок про безпеку:** **МД не має ні справжньої автентифікації, ні справжньої авторизації**. Уся «авторизація» — це факт, що сервер віддає лише ті дані, які прив'язані до пароля цього пристрою. Для нового RN-додатка це треба переробити (див. §8).

### 3.5. Прив'язка користувача до інших сутностей

**У ЦБ:**

- `Catalog.Пользователи` має реквізит `ФизЛицо` → `Catalog.ФизическиеЛица`. Це база — усе ім'я користувача, дата народження, ДРФО зберігається там.
- `ЗаказПокупателя.Ответственный` → `CatalogRef.Пользователи` — **це поле визначає, до якого менеджера належить замовлення**. Саме за ним повинна йти RLS-фільтрація.
- `СкладГруппа` → склад/група складів — теоретично може використовуватись для warehouse-based RLS.

**Регістр відповідності User → Манагер не знайдено** в експорті явно. **TBD:** це може бути в ChartsOfCharacteristicTypes.НастройкиПользователей або у ВидыПрав/ОбщегоНазначения.

**У МД:**

- `Константы.ПродавецПартнера` (String) — кодове ім'я менеджера, синхронізується з сервера.
- `Константы.СкладПартнера` (String) — кодове ім'я складу.
- `Константы.Партнер` (String) — напрямок (?).
- `Константы.Пользователь1с` (CatalogRef.УчасникиЧата) — посилання на учасника чату 1С.

Усе це **константи**, не сутність користувача — тобто менеджер отримує свій «контекст» при синхронізації як набір рядкових констант.

### 4.4. Цикл повного обміну (manager flow)

`Обмен_УправлениеОбменом.ВыполнитьОбменДанными()` (1c-export/Мобільний додаток/CommonModules/Обмен_УправлениеОбменом/Ext/Module.bsl:38-249):

```
1. Очистити стани (ОшибкаОбмена="", ОбменВыполняется=true)
2. СоединениеСБазой.ПолучитьПрокси() — створити SOAPProxy
3. ПарольВхода = Константы.ПарольВхода.Получить() — якщо порожньо → відмова
4. КодМобильногоКомпьютера = СисИнфо.ИдентификаторКлиента — UUID пристрою
5. SOAP: Прокси.НачатьОбмен(Метаданные.Версия, КодМобильногоКомпьютера, ПарольВхода, ЭтоНеПервыйОбмен)
   - якщо у відповіді не "*N", то це текст помилки — завершити
   - інакше N = Число(Сред(Ответ, 2)) — кількість пакетів
6. ЯКЩО НЕ ЭтоНеПервыйОбмен:
   Обмен_АнализДанных.УдалитьВсеДанные() — повне очищення локальної БД (зі збереженням Нагадування, Номенклатура, Контрагенты, СкладыКонтрагентов)
7. Цикл: поки КоличествоДанныхВПакете > 0:
   a) SOAP: Прокси.СформироватьПакетДанных(КодМ, ПарольВхода) → серіалізована Структура
   b) Десериализовать(пакет) → ДанныеОбменаСтруктура
   c) Якщо ЭтоПолнаяВыгрузка → УдалитьВсеДанные() (повторно, на випадок переключення режиму)
   d) Обмен_АнализДанных.ПрочитатьПакетДанных(...) → запис у локальну БД
   e) ЯКЩО ТаблицаИД ≠ порожньо → SOAP: Прокси.УдалитьРегистрацию(Сериализовать(ТаблицаИД), ПарольВхода)
   f) Прогрес: НовПроцент = ТекПакетов * 100 / N → Константы.ПроцентОбмена.Установить()
   g) Лог("Кількість в пакеті: ...")
8. ВиплачуВихідні зміни (закоментовано в коді — нижче в окремій функції ВыполнитьОбменДаннымиВыгрузкаЗаказов)
9. Константы.ОшибкаОбмена.Установить("")

11. Константы.ДатаОстанньогоОбміну.Установить(ТекущаяДата())
```

### 4.7. Обмін чату (`ОновитьЧат`)

```bsl
// 1c-export/Мобільний додаток/CommonModules/Обмен_УправлениеОбменом/Ext/Module.bsl:896
```

- `Резервы` — оновлені резерви.

### 4.8. Push-сповіщення (FCM)

ЦБ:
- `Constant.ИдентификаторПроектаFirebase` (String 500) — Firebase Project ID.
- `Constant.ДаныеКлючаДоступаGoogleAPI` (ValueStorage) — Service Account JSON (зберігається у бінарному вигляді).
- CommonModule `fcm_Уведомлени_HTTP_v1_API` — обгортка над FCM API v1 (`https://fcm.googleapis.com/v1/projects/{projectId}/messages:send`).

МД:
- `Constant.КлючПодписчикаУведомлений` (ValueStorage) — поточний FCM token пристрою.
- `Обмен_УправлениеОбменом.ОбновитьИдентификаторПолучателя()` — відправляє token на сервер (через SOAP `ОбновитьКлиента`).
- `ManagedApplicationModule.ПриПолученииУведомления` — handler для вхідних push:
  - дія `НовоеСообщение` → `ОткрытьФорму("Обработка.Чат1с.Форма.Форма")`.
  - дія `Напоминание` → `ОткрытьФорму("Обработка.Нагадування.Форма.ФормаПерегляду", параметри)`.

### 4.9. Як саме поточний МД синхронізується з ЦБ — підсумок

| Аспект | Поведінка |
|---|---|
| **Транспорт** | SOAP over HTTPS, gzip-серіалізація. |
| **Аутентифікація** | HTTP Basic + per-device password (`Константы.ПарольВхода`). |
| **Періодичність** | • Ручна — кнопка «Обмін» з робочого столу.<br>• Авто — `АвтообменКурсВалют` кожні 10 хв (якщо `АвтоматичнийОбмінКурсВалют=true`).<br>• Авто — `ВечірнійОбмін` кожні 30 хв (якщо `АвтоматичнеВечірнєСкиданняДокументів=true` І час > `ЧасСкиданняЗамовлень`).<br>• Авто — `ПолучениеСообщенийВайбер` кожні 60 хв.<br>• Фоновий повний — `Общий.АвтообмінВключений()` тригер. |
| **Ініціатор** | Завжди клієнт (МД). Сервер пасивний. |
| **Вирішення конфліктів** | • Контрагент: `КонтрольВерсийПерезапису` — якщо серверна `ДатаОновлення` новіша, локальні зміни тіла відкидаються (зберігаються лише поля CRM-кеша).<br>• Документи: на сервер у `ИнфоМаршруты` повертається список того, що треба перезаписати (НеХватает рядки).<br>• Загалом — **Last-Write-Wins з пріоритетом сервера**. |
| **Що працює офлайн** | Усе — локальна 1С-БД працює як SQLite. Менеджер може створити/змінити Заказ, КассовыйОрдер, Презентацію, бачити Прайс, Залишки. Усі зміни попадають у `РегистрацияОбмена` і ждуть наступного синку. |
| **Що НЕ працює офлайн** | Запит до Viber, оновлення курсів валют, бронювання (треба сервер), отримання нових повідомлень чату, синхронізація прайсу. |
| **Розмір локальної БД** | Може бути великим (повна копія прайсу + всі історичні замовлення). У МД є `Обмен_АнализДанных.УдалитьВсеДанные()` — кнопка «повний скид». |

### 4.10. Інтеграційні точки із зовнішнім світом (з конфігурації)

1. **Наш Next.js сайт** (catalog-full repo): обмін через `ExchangePlan.ОбменССайтомТоварами` (товари + прайс) і `ОбменССайтомЗаказами` (замовлення з сайту → ЦБ). Реалізація — підсистема `WEBПриложения`. **Формат і протокол треба підтвердити окремо** (CommerceML 2 / 3 / кастомний JSON?). Регламентне завдання `ЗаданиеОбменССайтом` тригер.
2. **Viber-бот** (зовнішній сервер `ltex1c.com.ua/ltexbot/hs/bots/exchange`): обмін через HTTP POST. Сервер L-TEX отримує Webhook на `/bots/send` і `/bots/sendsecond`.
3. **1С:Бухгалтерія** (через `ОбменУправлениеТорговлейБухгалтерия` + Фоновый план): вивантаження документів проведених в обліку.
4. **1С:Роздрібна торгівля** (`ОбменУправлениеТорговлейРозничнаяТорговля`): синхронізація з POS-системою магазинів.
5. **Розподілена база (RIB)** через `ExchangePlan.Полный`: реплікація на додаткові філії.
6. **Firebase FCM**: пуш-сповіщення на мобільні (через `fcm_Уведомлени_HTTP_v1_API`).
7. **Pactum** (`Subsystem.Пактум_СервисКонтрагент`): онлайн-перевірка контрагентів через стороній сервіс.
8. **CommerceML / EDI** (`DataProcessor.ВыгрузкаДанныхCommerceML`): експорт даних у форматі CommerceML.
9. **1С:Бухгалтерія 7.7 і 8.1** (`DataProcessor.ВыгрузкаДанныхВ1СБухгалтерию77`, `81_11`): легасі-вивантаження.

> **Не з конфігурації, але треба перевірити:** інтеграції з банками (для виписок), чат-боти Telegram, API налаштування доставки (Нова Пошта, Укрпошта тощо). У XML-метаданих явно не видно.

- ЧасОбміну (DateTime, тільки час)

При збереженні записує всі константи. Викликається з `CommonCommand.ПарольВходаОткрыть`.

#### 5.4.6. `ФормаВстановленняКурсуВалют`

Два поля: USD, EUR. При збереженні:
1. Записує в `Константы.КурсUSD`, `Константы.КурсEUR`.
2. Викликає SOAP `ЗаписатиКурсВалют` для синхронізації з ЦБ.

Тільки якщо `Константы.ПравоМінятиКурсиВалют = true`.

#### 5.4.7. `ФормаОтправкиСообщенийВайбер` — масова розсилка

Поля: НовоеСообщение (multiline), ВыбратьФайл (file picker), СписокКонтрагентов (TS з контрагентами). Кнопка «ОтправитьВайбер» викликає `ViberОбновлениеКлиент.ОтправитьСообщенияСКлиента(СписокКонтактов, НовоеСообщение, ВыбраныйФайл)`.

#### 5.4.8. `ФормаОтчетОстатки` / `ФормаОтчетПродажи`

Дата-початок + Дата-кінець + кнопка «Сформувати». Будує СКД-звіт по `ОстаткиТоваров` / `Продажи`.

#### 5.4.9. `ФормаПросмотраПоследнейОшибки`

Показує останній запис з `РегистрСведений.Лог` для дебагу.

### 5.5. Обробки (DataProcessors)

#### 5.5.1. `DataProcessor.Оплата` — введення оплати

**Файл:** `1c-export/Мобільний додаток/DataProcessors/Оплата/Forms/Форма/Ext/Form/Module.bsl`

Параметри відкриття: `МаршрутныйЛист | ЗаказПокупателя | Реализация | Контрагент | СуммаКОплате`. Поля:
- `СуммаДокумента` (read-only)
- `ОплатаEUR`, `ОплатаUAH`, `ОплатаUSD`, `ОплатаUAH_Безнал`
- `СдачаEUR`, `СдачаUAH`, `СдачаUSD` (розраховуються)
- `ОстатокДокумента` (заборгованість після оплати; від'ємний = переплата)
- `ОплатаEURРек`, `ОплатаUAHРек`, `ОплатаUSDРек` (рекомендована розкладка)

Логіка `СуммаПриИзменении`:
```
ОплатаДокумента = ОплатаEUR + ОплатаUAH/КурсEUR + ОплатаUSD*КурсUSD/КурсEUR
СдачаДокумента   = ОплатаДокумента - СуммаДокумента (якщо > 0)
ОстатокДокумента = СуммаДокумента - ОплатаДокумента
```

При збереженні створює `Document.КассовыйОрдер` (ВидДвиженияДенежныхСредств=Приход) і прив'язує до основи (`Основание`).

#### 5.5.2. `DataProcessor.Чат1с` — внутрішній чат

Функціональність:
- Список груп чату (з `Catalog.ГруппыЧата`).
- Для кожної групи: останнє повідомлення, лічильник непрочитаних.
- Відкриття групи: список повідомлень з `РегистрСведений.ВходящиеСообщения` + `ИсходящиеСообщения`, з фото/файлами, реакціями.
- Введення нового повідомлення → `РегистрСведений.ИсходящиеСообщения` + push на сервер при наступному синку.

#### 5.5.3. `DataProcessor.Нагадування` — нагадування

Форми: `Форма` (створення), `ФормаОтриманихНагадувань` (список одержаних), `ФормаПерегляду` (перегляд одного).

Створення:
- Тип нагадування: разовий / періодичний (різні UI-сторінки)
- Дата/час
- Текст
- Прив'язка до товару (`УИД_Номенклатура`) або клієнта (`Контрагент`) — опціонально

Зберігається в `РегистрСведений.ЗапланированыеУведомления` + `Catalog.Нагадування`. На мобільному — створює `ДоставляемоеУведомление` (через 1С Mobile API), яке OS показує на пристрої.

#### 5.5.4. `DataProcessor.ЗакрытиеСтарыхЗаказов`

Менеджер вибирає клієнта, тисне «Заповнити» → SOAP `ПолучитьДанныеНезакрытыхЗаказов(пароль, контрагент_УИД)` → відображає список відкритих замовлень того клієнта в ЦБ → можна виділити рядки і натиснути «Закрити» → SOAP `ЗакрытьСтарыеЗаказы(пароль, товары, контрагент_УИД)`.

### 5.6. Workflow #1 — створення замовлення

```
1. РабочийСтол → [Заказы] (Document.Заказ.ФормаСписка)
2. [Створити] → відкривається ФормаДокумента
3. Менеджер обирає Контрагент (через ФормаВыбора Контрагенты)
   - Авто-підставляється ТипЦен з налаштувань клієнта (?TBD)
   - Підставляється СкладКонтрагента (за замовч. перший зі списку клієнта)
4. Натискає [+] над таблицею Товары → відкривається ФормаПодбораНоменклатуры
   - Користувач вводить пошук АБО сканує ШК камерою (метод ШК)
   - Якщо ШК знайдено → авто-вибір товару + характеристики
   - Натискає товар → з'являється поле кількості
   - Кількість ≤ Залишок (валідація)
   - Натискає [Додати] → товар попадає в Заказ.Товары
5. Повторює для всіх позицій
6. Опціонально: [Записати коментар] → ФормаВводуКоментаря
7. Натискає [Записати і закрити]:
   - ПередЗаписью: Сума = Товары.Итог("Сума"); Архив = ПометкаУдаления
   - ПриЗаписи: Общий.ЗарегистрироватьИзменения() → запис в РегистрацияОбмена
8. Активний сценарій з геолокацією: ПриЗаписи паралельно фіксує Широту/Довготу через РегистрСведений.РегистрацияКоординат
9. Документ потрапляє в чергу. При наступному синку (ручному або фоновому):
   - SOAP ОбработатьПакетДанных → ЦБ створює ЗаказПокупателя.Ответственный = поточний менеджер
   - У відповідь приходить ТаблицаИД → УдалитьРегистрацию → документ Архив=true, В_1с=true
```

### 5.7. Workflow #2 — реалізація + оплата + здача

```
1. Менеджер на маршруті (МаршрутныйЛист.Заказы є цей Заказ)
2. Відкриває Заказ, тисне [Реалізувати] → копія в РеализацияТоваровУслуг (з прив'язкою до Заказ через Сделка)
3. РТУ-форма: можна змінити кількість/ціни/знижки
4. [Записати і провести]:
   - ПередЗаписью: розраховує ціну з ваги характеристики
   - ПриЗаписи: реєстрація в РегистрацияОбмена
5. [Прийняти оплату] → DataProcessor.Оплата (з параметрами: МаршрутныйЛист, Реализация, Контрагент, СуммаКОплате)
   - Менеджер вводить готівку (UAH/EUR/USD)
   - Розраховується СдачаДокумента
6. [Зберегти оплату] → створюється КассовыйОрдер.ВидДвиженияДенежныхСредств=Приход
7. Якщо є здача — окремий КассовыйОрдер.ВидДвиженияДенежныхСредств=Расход з посиланням на основний (СдачаДокумента)
8. Уся група йде в РегистрацияОбмена → синк
```

### 5.8. Workflow #3 — повернення товару

```
1. Документи → Возврат (від клієнта на склад агента)
2. ФормаДокумента → [Додати товар]
3. Відкривається ФормаПодбораНоменклатурыДляВозврата (показує товари клієнта)
4. Вибір товару + кількість
5. [Записати]:
   - ОбработкаПроведения: Движения.ОстаткиТоваров (Расход)
6. Якщо повертають за гроші → паралельно ВозвратОтПокупателя (рухи Продажи- та ОстаткиТоваров-)
7. Синк
```

### 5.9. Workflow #4 — маршрутний лист

```
1. Документи → МаршрутныйЛист
2. Створити новий ML на сьогодні
3. Заповнити TS Заказы (вибрати з відкритих замовлень дня)
4. Можна сформувати порядок об'їзду (через CommonCommand.ВідкритиМаршрутНаКарті — TBD)
5. [Записати і провести]:
   - ПриЗаписи: для кожного Заказа з TS встановлює МаршрутныйЛист = Ссылка
   - Зворотні зв'язки прописуються в усіх Заказах
6. Менеджер їде, проводить продаж/оплату
7. Наприкінці дня — фіксує кінцевий кілометраж (ФормаВводаКілометражу)
8. ВечірнійОбмін автоматично відправляє все на сервер
```

### 5.10. Workflow #5 — презентація товару клієнту

```
1. Документи → Презентація → новий
2. Вибір контрагента
3. Додати товари (без кількості, з коментарем)
4. [Відправити у Viber] (з модуля Общий.ПолучитьТекстСообщенияВайберДляПрезентации)
   - Формується текст з картинками з ХарактеристикиНоменклатуры.СсылкаНаYouTube
   - Запускається Android intent у Viber
   - ВідправленоВВайбер = true
5. [Записати]
6. Синк → в ЦБ створюється запис, заноситься в РаботаСКлиентом для CRM
```

### 5.11. Робота зі сканером штрих-кодів

**Підтримка:** в формі `Catalog.ХарактеристикиНоменклатуры.Forms.ФормаЭлемента` (для встановлення сектору на складі) і у `ФормаПодбораНоменклатуры`:

### 5.12. Камера, друк, файли

- **Камера:** використовується тільки для сканера ШК (через `СредстваМультимедиа.ПоказатьСканированиеШтрихКодов`).
- **Камера для фото:** TBD — в коді не видно явного `СделатьФотографию`. Можливо, використовується через intent у `Document.Презентація` для прикріплення фото.
- **Друк:** дозвіл `Print=true` в маніфесті, але реальних викликів `ТабличныйДокумент.Печать()` у мобільному коді не виявлено. Скоріш за все, використовується сторонній додаток для друку чеків (через intent).
- **Файли:** дозвіл `FileExchangeWithPersonalComputer=true`. Використовується для бекапу локальної бази через 1С Mobile UI.

### 5.15. Конфлікти та повторні відправки

- **Документи:** мають `мЭтоОбмен Перем` — прапорець, що відключає реєстрацію змін під час прийому з сервера.
- **Контрагенти:** `КонтрольВерсийПерезапису` (див. §2.4.1) — last-write-wins з пріоритетом сервера, але CRM-кеш зберігається локально.
- **Документи з сервера, які локально були змінені:** механізм незрозумілий — можливо, документи на сервері просто перезаписуються.

> **TBD:** з'ясувати, як вирішується конфлікт «менеджер змінив документ → не встиг синхронізуватись → сервер прийшов з версією іншого менеджера». В коді явно не видно — імовірно, серверна версія перезаписує локальну.

### 6.3. Розрахунок ціни / знижки / кредитного ліміту

**Розрахунок ціни:**
- Базовий: `Документы.РеализацияТоваровУслуг.ПередЗаписью` — `Цена = ЦенаПродажиВес * Характеристика.Вага`, `Сумма = Цена * Кількість` (якщо їх не вказано).
- Точкова ціна: `Общий.ПолучитьЦенуНоменклатуры(Номенклатура, Характеристика, ТипЦен)` — лукап у `РегистрСведений.ЦеныНоменклатуры`.

**Знижки:**
- `Документы.РеализацияТоваровУслуг.Товары.Знижка` — поле в TS.
- В UI оплати: автоматично виводиться кнопка «Дать скидку», якщо залишок документа потрапляє у певний поріг (TBD: де визначається поріг — імовірно константа в ЦБ).

**Кредитний ліміт:**
- `Константы.ПорогЗадолженостиEUR` — глобальний поріг (~ EUR ліміт).
- `Константы.ДолгПартнера` — поточний борг (в EUR).
- В UI: при перевищенні ліміту менеджер бачить **червоний борг**: `Клієнтський.ИнфоБорг` (рядок 2 модуля `Клієнтський`).
- Конкретний борг клієнта: `Контрагенты.Борг`, `Контрагенты.ПросроченийБорг`, `Контрагенты.БоргТзОВ`, `Контрагенты.ПросроченийБоргТзОВ` — синхронізуються з ЦБ через `ОбмінБоргиТаЦіни`.

## 8. Recommendations for new mobile app (React Native)

### 8.1. Архітектурний підхід — gateway, не прямий 1С

**Не рекомендується** новому RN-додатку прямо ходити в `MobileExchange.1cws`:
- SOAP + ValueStorage важко парсити в JS.
- Логіка обміну (24+ кроки, гра з пакетами по N записів) дуже специфічна.
- Аутентифікація через shared password не підходить для довгоживучих токенів.
- Конфлікт-резолюшн зашитий у бінарну серіалізацію 1С.

**Рекомендується** проміжний шар-gateway: новий REST/JSON API в `apps/store/api/*` Next.js, який:
- Викликає `MobileExchange.1cws` під капотом (там, де треба бізнес-логіка УТ — проведення документів, проведення в регістри тощо).
- АБО викликає **новий HTTPService у ЦБ** (новий ендпойнт, що віддає JSON напряму, без ValueStorage). Цей варіант кращий — менше серверних викликів, але потребує співпраці з 1С-розробником.
- Тримає JWT/OAuth токени, refresh-логіку, RBAC.
- Кешує прайс і залишки (Redis / Postgres).
- Транслює 1С-структури в чисті REST-ресурси.

### 8.4. Real-time vs cached (стратегія мережевої роботи)

| Дані | Real-time | Кешувати локально | Підстава |
|---|---|---|---|
| Каталог товарів (метадані) | ❌ | ✅ (повний синк раз/день) | Великий обсяг, рідкі зміни. |
| Прайс | ❌ | ✅ (за `priceTypeId`, lazy при відкритті товару) | Зміни максимум кілька разів на день, можна inkrementно. |
| Залишки **загальні** | ❌ | ✅ + інкр. синк | Для UX-швидкості пошуку. |
| Залишки **конкретної характеристики** | ✅ (опц. за `Константы.ОновлюватиЗалишкиПриВиборі`) | — | При виборі товару — точний залишок з сервера. |
| Контрагенти (метадані) | ❌ | ✅ | Великий обсяг, переміщення менеджера = одні й ті ж клієнти. |
| Борги контрагентів | ❌ | ✅ + інкр. синк (10-30 хв) | Швидко змінюються, але не критично. |
| Резерви (мої) | ✅ | ❌ | Конкуренція з іншими менеджерами — не можна показувати застаріле. |
| Документи менеджера | ✅ запис → сервер | ✅ читання | Local-first з оптимістичним апдейтом. |
| Курси валют | ❌ | ✅ (раз/день) | Не змінюються вночі. |
| Чат | ✅ | ✅ (історія) | Push-нотифікації + WebSocket. |
| CRM-таймлайн клієнта | ❌ | ✅ + інкр. синк | Не критично real-time. |
| Маршрут на сьогодні | ✅ при зміні | ✅ | Менеджер бачить, що йому призначено. |

- Гео-логування (фон).

#### 8.5.3. Can be removed / переробити

- Виклик Android-intent для Viber (замінити на нативну React Native бібліотеку `react-native-share`).
- Власна реалізація черги обміну (`РегистрСведений.РегистрацияОбмена`) → замінити на TanStack Query mutations + offline queue (наприклад, `react-native-queue` або кастомний AsyncStorage queue).
- Власна серіалізація через ValueStorage → JSON у RN.
- Кнопки «Завантажити товари / контрагентів окремо» → авто-синк за дельтами (`modifiedSince` в API).
- Multi-mode `ФормаОбмена` → один бекенд-керований pull-to-refresh.
- Прапорець `мЭтоОбмен Перем` у документах → не потрібно (на бекенді просто не рекурситься).
- Дві ролі (Пользователь + ВидимістьВсіхКлієнтів) → стандартна RBAC (Manager / SeniorManager / Admin).

### 8.6. Архітектура RN-додатка (рекомендації)

#### 8.6.1. Стек

- **Framework:** Expo (managed) для перших місяців; пізніше — bare workflow якщо знадобляться нативні модулі (BLE-принтер, серйозний background-location).
- **Навігація:** `expo-router` (file-based).
- **Server state:** TanStack Query (з `persistQueryClient` на AsyncStorage для офлайн-кешу).
- **Local DB:** SQLite (`expo-sqlite`) для локальних чернеток замовлень + `op-sqlite` для виправлення продуктивності, або WatermelonDB якщо потрібна реактивність.
- **Mutation queue (offline):** власна реалізація в SQLite — таблиця `mutation_queue {id, endpoint, body, createdAt, attempts, status}`, sync-worker через `expo-task-manager`.
- **Auth:** JWT, refresh з `expo-secure-store`.
- **Push:** Expo Notifications (працює поверх FCM на Android + APNs на iOS).
- **Геолокація:** `expo-location` (background = `expo-task-manager` + `Location.startLocationUpdatesAsync`).
- **ШК-сканер:** `expo-barcode-scanner` або `expo-camera`.
- **Шарінг:** `expo-sharing`, `expo-linking`.
- **Аналітика помилок:** Sentry.
- **CI/CD:** EAS Build.

### 8.7. Чи писати окремий gateway чи використати наявні HTTPServices

**Рекомендація: писати окремий gateway** — наприклад, `apps/store/api/v1/*` у Next.js (App Router):

- ✅ Дає JSON-API без зміни 1С (1С лишається SOAP/ValueStorage за лаштунками).
- ✅ Дозволяє додати JWT-аутентифікацію з нормальним RBAC.
- ✅ Можна кешувати важкі запити (прайс, баланси) у Redis/Postgres.
- ✅ Дозволяє транслювати CRUD-моделі ЦБ → клієнтські JSON-моделі без знань 1С на стороні мобільного.
- ✅ Можна додати rate-limiting, audit log, telemetry.
- ✅ Можна підготувати функціонал для майбутнього публічного API.
- ❌ Дублювання шару (мап `1С-name → REST-name`).
- ❌ Додаткова інфра (Next.js + DB + Redis).

Альтернатива — **новий HTTPService у ЦБ**, що віддає JSON напряму. Плюси: менше latency, прямий доступ. Мінуси: складніше тестувати, важко версіонувати, треба 1С-розробника для будь-яких змін, важко додати middleware (rate limiting, telemetry).

**Гібрид** (рекомендую): gateway робить операції **читання** з кешу (Postgres-snapshot оновлюється фоновим воркером, що ходить у `MobileExchange.1cws` за дельтами); операції **запису** — проксі через `MobileExchange.1cws.ВыполнитьКоменду` (generic RPC), щоб бізнес-логіка проведення лишалась в одному місці.

## 9. Ризики і нерозв'язані питання

### 9.1. Архітектурні ризики

#### 9.1.1. Зашитий SOAP-пароль і його зміна

**Файл:** `1c-export/Мобільний додаток/CommonModules/СоединениеСБазой/Ext/Module.bsl:16-17`

```bsl
Пользователь = "SOAP";
Пароль = "wsBFtyg87GDR4kdsjKI8";
```

- Пароль захардкоджено. Зміна = пересборка APK і ре-роллаут на всі пристрої.
- В декомпільованому APK буде відразу видно. Якщо `192.168.0.10` доступний з публічної мережі — можна авторизуватись як SOAP-користувач.
- **Дія для нового RN-додатка:** не дублювати — отримувати креди з backend gateway або з env-файлу через EAS Secrets.

#### 9.1.2. Per-device password = identity

`Константы.ПарольВхода` одночасно є:
1. Аутентифікаційним секретом.
2. Логічним ID менеджера (сервер по ньому шукає `Catalog.Пользователи`).

Наслідки:
- Не можна змінити пароль без того, щоб «втратити» історію менеджера.
- Не можна мати того ж менеджера на двох пристроях (бо `СисИнфо.ИдентификаторКлиента` різний).
- Не можна виключити ярлик доступу одного менеджера, не торкаючи іншого.
- При компрометації пароля треба **створювати нового користувача** в ЦБ.

**Дія:** новий RN-додаток повинен мати JWT-аутентифікацію з рознесеними `userId` (UUID менеджера в ЦБ) і `accessToken`.

#### 9.1.3. RLS у ЦБ невидимі в експорті

В експорті ролей ЦБ немає файлів `Ext/Rights.xml`. Не можна підтвердити, які саме об'єкти бачить `МенеджерПоПродажам`, які поля приховані, які накладені обмеження по запису. Усе це треба з'ясувати у живому Конфігураторі.

Особливо важливо: **чи має `МенеджерПоПродажам` обмеження за `Ответственный = ТекущийПользователь`** на читання `ЗаказПокупателя`? Чи може він бачити чужі замовлення? Це критично для дизайну ендпойнтів `GET /orders?onlyMine=true`.

#### 9.1.4. Режим сумісності 8.3.14

Мобільний додаток компілюється в режимі сумісності 8.3.14. Це досить старий режим (актуальна 8.3.27+ на 2025/26). Якщо ЦБ оновлюється до новішої платформи, є ризик регресій у мобільному.

Для нового RN-додатку це не критично (RN не залежить від версії 1С), але треба врахувати при міграції з gateway → SOAP.

#### 9.1.5. Stateless SOAP без транзакційності

Виклики `НачатьОбмен` + `СформироватьПакетДанных` + `УдалитьРегистрацию` — це **3 окремі SOAP-виклики**, без розподіленої транзакції. Якщо мережа впала між викликами:
- Може втратитись пакет (отримали, але не очистили чергу на сервері).
- Може дублюватись запис (відправили, мережа впала, повторно відправили).

В коді МД є retry-логіка (рядок 200): «Unexpected end of stream» обробляється як «продовжити». Але інші помилки = abort.

**Дія для нового RN-gateway:** використати **idempotency keys** (передавати `X-Idempotency-Key: <uuid>`) для всіх write-операцій. На стороні backend дедуплікувати.

#### 9.1.6. Поліморфне поле `Объект: AnyRef` у `РегистрацияОбмена`

```xml
<Field>
    <Name>Объект</Name>
    <Type><v8:Type>v8:AnyRef</v8:Type></Type>
</Field>
```

Це дозволяє складати довільний об'єкт у одну чергу, але:
- Складно фільтрувати/індексувати.
- Складно автоматично знати, в який план обміну він має йти.
- Розв'язується через `ТипЗнч(Об'єкт) = Тип("ДокументСсылка.Заказ")` всюди (антипатерн `instanceof`).

**Дія для RN:** замість поліморфних таблиць — окремі mutation-черги для кожного типу ресурсу.

### 9.2. Бізнес-ризики

#### 9.2.1. «Магічні» статуси і константи без коментарів

- `Заказ.ПершаФорма`, `Заказ.ПризначитиПродажТорговомуКонтрагента` — невідомий бізнес-зміст.
- `Заказ.Архив` ставиться автоматично при `ПометкаУдаления` — може бути неочікуваним для нового персоналу.
- `КассовыйОрдер.КоректировкаДолга` (Number 10,2) — невідомо, що це за коректування і коли застосовується.
- `КассовыйОрдер.УИДUSD` і `УІДГРН` — невідомо, для якої системи звіряння.

#### 9.2.2. Дублювання сутностей (МД vs ЦБ)

- В МД є `Заказ`, в ЦБ — `ЗаказПокупателя`. Різна структура (МД спрощена).
- В МД є один `КассовыйОрдер` для приходу і витрати, в ЦБ — `ПриходныйКассовыйОрдер` + `РасходныйКассовыйОрдер`.
- В МД є `Возврат` і `ВозвратОтПокупателя`, в ЦБ — `ВозвратТоваровОтПокупателя`.

Кожна синхронізація — це **трансляція моделі**, виконується на стороні `WebService.MobileExchange`. Якщо колись треба буде підтримувати ще одну версію клієнта (RN), додасться ще один шар трансляції. Є ризик, що в ЦБ є поля (наприклад, `СуммаНДС`), які мобільний втрачає.

#### 9.2.3. Конфлікт-резолюшн «сервер виграє»

`Контрагенты.КонтрольВерсийПерезапису()` зберігає лише CRM-кеш при перезаписі сервером. Усі локальні правки тіла (адреса, телефон) **губляться без попередження**. Менеджер не побачить, що його зміна перезаписана.

**Дія для RN:** у gateway зробити merge-логіку (або хоча б показувати «локальні зміни перезаписані сервером») і логувати це у CRM-таймлайн.

#### 9.2.4. Локальна БД може розростатись

`Обмен_АнализДанных.УдалитьВсеДанные` чистить лише при першому або повному обміні. Між повними обмінами накопичуються:
- Усі прайси (`ЦеныНоменклатуры`).
- Усі залишки.
- Усі документи 6+ місяців.
- Уся історія Viber.

На дешевих Android-пристроях з обмеженою пам'яттю це може стати проблемою.

**Дія для RN:** використати чітку TTL-стратегію (lru cache на 30 днів для документів; повний скид прайсу щоночі).

### 9.4. Технічний борг (місця, де код виглядає крихким / застарілим)

- Дублювання патерну `НачатьОбмен → ПолучитиСеанс → Цикл → ОновитиПрогрес → ЗакінчитиОбмен` в **8 функціях** модуля `Обмен_УправлениеОбменом` (рядки 38, 251, 411, 525, 654, 768, 793, 896 — кожна з варіаціями). Спільну частину можна було винести в шаблон.
- Багато закоментованого коду (рядки 200-228, 304-311, 318-340 у різних модулях) — невідомо, чи це експерименти, чи реальна логіка, що тимчасово вимкнена.
- Таблиця `Лог` з `Дата` як єдиною ключовою колонкою (`MainFilter`) — конфлікти при швидких записах (одна секунда — один рядок).
- `Обмен_АнализДанных.СоздатьЭлементыСправочника` — гігантська процедура (≈ 200 рядків) з нізкорівневою роботою з реквізитами через ім'я (`Метаданные.Справочники[ИмяСправочника]`). Складна для тестування.
- Нумерація документів: `Numerator` зашитий у кожному документі окремо (немає спільного `DocumentNumerator`). При зміні правил — треба змінювати в кожному документі.
- `КассовыйОрдер.СдачаДокумента` — рекурсивний референс на той же документ. Може формувати замкнені цикли.
- `Заказ.ВивантажуватиВ1С` (default true) і `В_1с` (виставляється при синку) — два прапорці для одного концепту. Легко плутати.
- Запис `мЭтоОбмен Перем` як **публічна змінна** документа — ламає енкапсуляцію, легко не виставити, тоді буде нескінченний обмін.

### 9.5. Залежності від конкретної платформи

- **Платформа 1С:** 8.3.14 + Mobile Platform. Якщо 1С відмовиться від мобільної платформи (були чутки про депрекацію) — мобільний додаток не буде підтримуватись.
- **Платформа Android:** через 1С Mobile Platform працює тільки на Android (iOS — в теорії, але потребує підпису від Apple, що складно для 1С). Поточний додаток фактично **Android-only**.
- **Windows-only сервер:** SOAP-сервіс хоститься на Windows-сервері з IIS (типово для 1С). Linux-варіант 1С існує, але рідкісний у проді.
- **Viber API:** прив'язка до Rakuten Viber. Якщо Viber змінить API — треба адаптувати.
- **Firebase FCM:** залежність від Google. Альтернатив на українському ринку немає.

```
МД user → CommonCommand.ВыполнитьПолныйОбмен →
  Константы.ЭтоНеПервыйОбмен.Установить(Ложь)

    Обмен_АнализДанных.ОтправитьИзменения() → СтруктураВідправки

    Розпакувати РезРаспакованный
    ИнфоМаршруты → оновити МаршрутныйЛист.НеХватает

    Обмен_АнализДанных.УдалитьВсеДанные()  // повне очищення

      SOAP: СформироватьПакетДанных → пакет
      Десериализовать → ДанныеОбменаСтруктура
      Если ЭтоПолнаяВыгрузка → УдалитьВсеДанные() (повторно)

      ПринятьПакетДокументов
      Если ТаблицаИД → SOAP: УдалитьРегистрацию(serialize(ТаблицаИД))
      Прогрес++
  
  Обмен_УправлениеОбменом.ОновитиЗалишкиТаЦіни()
```
