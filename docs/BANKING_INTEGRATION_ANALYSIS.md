# Інтеграція банкінгу в CRM L-TEX — глибокий аналіз API та план

**Дата:** 2026-07-24 · **Гілка:** `claude/crm-banking-integration-ynoex1` · **Статус:** аналіз + план (коду ще немає)

Мета user (з ТЗ): підключити всі наші рахунки, щоб бухгалтер бачив усі рухи в системі; система сама створює касові ордери (прихід/розхід) за налаштованими статтями; дашборд залишків у реальному часі; авто/напівавто рознесення платежів клієнтів; чек у Checkbox на потрібних операціях (напр. зарахування з транзитного); облік готівкової каси (грн/євро/долар) зі щоденним підбиттям, звіркою, виявленням переліміту/недостачі та підказкою «який документ створити». Банки: **ПриватБанк, Монобанк, NovaPay**.

---

## 0. Головний висновок

- **Половина фундаменту вже в коді.** Є моделі `BankPaymentIncoming/Outgoing`, `CashTransfer` + двигун проведення `treasury-posting.ts` (ДДС + борг + draft/post/cancel), касові ордери `MgrCashOrder`, борг `MgrDebtMovement`, ДДС `CashFlowMovement`, Checkbox (поки лише ETTN-наложка), NovaPay авто-звірка (`novapay-reconcile.ts`), крони з `CRON_SECRET`, роль `bookkeeper` + гейт `canManageTreasury`. **Треба ДОБУДУВАТИ, не дублювати.**
- **Чого немає:** жодного клієнта зовнішнього банківського API (виписки Privat/Mono), поля валюти на рахунку, узагальненого (не-ETTN) чека Checkbox.
- **Реалізм по банках:** Monobank — найкращий (webhook = реальний час, read-only); PrivatBank — зрілі виписки, лише опитування (без webhook); NovaPay — **немає публічного API виписки транзитного рахунку**, реконсиляція через статуси Нової Пошти (те, що вже частково є).
- **Вихідні платежі з системи** — обмежені: Privat дозволяє створити, але підпис у кабінеті; Mono взагалі не дозволяє платити. Тобто «система сама платить постачальникам» — лише підготовка + ручний підпис.

## 1. PrivatBank (Автоклієнт / Приват24 для бізнесу)

- **Правильний API:** корпоративний REST «Автоклієнт» — база `https://acp.privatbank.ua/api`. (НЕ старий картковий `api.privatbank.ua/p24api`, НЕ LiqPay, НЕ Open Banking.)
- **Виписки (ядро):** `GET /api/statements/transactions?account=<IBAN>&startDate=DD-MM-YYYY&endDate=DD-MM-YYYY&limit=100&followId=`. Пагінація курсором `next_page_id`. Поля: `AUT_CNTR_CRF` (ЄДРПОУ контрагента), `AUT_CNTR_ACC`/`AUT_CNTR_NAM`, `CCY`, `SUM`, `SUM_E` (еквів. UAH), `TRANTYPE` (`C`=прихід/`D`=розхід), `OSND` (призначення платежу), `NUM_DOC`, `ID`/`TECHNICAL_TRANSACTION_ID` (ключ ідемпотентності).
- **Near-real-time:** `GET /api/statements/transactions/interim` — операційний день, що триває; опитувати кожні 1–5 хв. `.../final` — закритий день (авторитетно). `GET /api/statements/settings` дає межі опердня.
- **Баланси:** `/api/statements/balance`, `/balances`, `/balanceat`, `/balances/interim` — `balanceOut` (нативний) + `balanceOutEq` (UAH), `is_final_bal`.
- **Авторизація:** токен на кабінет (усі IBAN кабінету), генерується в Приват24 бізнес → Бухгалтерія → Інтеграція (Автоклієнт). Передається HTTP-заголовком `token`. ⚠️ Перевірити точну назву заголовка у своєму кабінеті (`token` vs `id`+`token`).
- **Webhooks:** НЕМАЄ — лише опитування.
- **Створення платежів:** існує, але зазвичай лягає чернеткою, яку треба **підписати в кабінеті** (OTP/ЕЦП). Не закладати авто-виплати без підпису → Фаза 2, потребує уточнення в підтримки Privat.
- **Мультивалюта:** 1 IBAN на валюту; токен покриває всі.
- **Ліміти:** `limit`≤100, офіційного req/sec немає — backoff + дедуп.

## 2. Monobank (Personal API для ФОП)

- **Правильний API:** безкоштовний Personal API `https://api.monobank.ua` з `X-Token` (генерується самим у застосунку/на api.monobank.ua). ФОП-рахунок повертається як звичайний account з `type:"fop"` і додатковими полями контрагента. Corporate/Provider API (ECDSA+згода) — лише щоб читати ЧУЖІ рахунки, нам не треба. Acquiring — окремий продукт (приймати оплати).
- **READ-ONLY по грошах:** читати рахунки/баланси/виписки + ставити webhook. **Платежі створювати НЕ можна.**
- **Виписки:** `GET /personal/statement/{account}/{from}/{to}` (unix-секунди, макс 31 день+год). Поля `StatementItem`: `id`, `time`, `amount` (мін. одиниці, знак), `operationAmount`, `currencyCode` (980 UAH/978 EUR/840 USD), `balance` (після операції), `comment`, `counterEdrpou`/`counterIban`/`counterName` (ФОП). Суми в копійках → ÷100.
- **Баланси:** `GET /personal/client-info` → усі accounts (кожна валюта = окремий account з власним `id`/`iban`/`balance`).
- **Webhook (ключова фіча — реальний час):** `POST /personal/webhook` з `{webHookUrl}`. Наш роут має відповідати 200 на GET-перевірку і на кожен POST `{type:"StatementItem", data:{account, statementItem}}`. Дедуп по `statementItem.id`. **Обходить ліміт 1 запит/60с.** ⚠️ Додати шлях у Cloudflare WAF skip (як для Viber/Telegram ботів).
- **Ліміти:** statement/client-info — 1 запит/60с (429 при перевищенні); webhook не лімітований. Опитування раз/хв — лише як резерв + оновлення балансу.

## 3. NovaPay + Нова Пошта (COD / контроль оплати)

- **Чесно:** «NovaPay» і «Nova Poshta API» — різні платформи. **Публічного REST API виписки/балансу бізнес- або транзитного рахунку NovaPay НЕМАЄ.** Виписки доступні лише через кабінет NovaPay Бізнес (електронні виписки) + партнерські клієнт-банк інтеграції (як у KeyCRM/Bookkeeper) — це не документований API, треба окремо домовлятися.
- **Що є в NovaPay API:** Acquiring (`api-ecom.novapay.ua` — приймати картки), Comfort (`contragent-api.novapay.ua` — масові виплати, має `/v1/balance` + `/v1/export-operations`, але це реєстр виплат, НЕ транзитний рахунок). Авторизація — `x-sign` (RSA) + `x-merchant-id`, потрібен договір.
- **Звідки реально беремо реконсиляцію COD:** з логістичного API Нової Пошти `api.novaposhta.ua` `TrackingDocument.getStatusDocuments` — `RedeliverySum` (сума наложки), `RedeliveryNum` (№ грошового переказу), `StatusCode` **9**(отримано)→**10**(переказ обробляється)→**11**(переказ видано). Це вже використовується в `novapay-reconcile.ts` + крон `refresh-np-shipments`.
- **Транзитний рахунок:** внутрішній леджер NovaPay, публічного API рухів/виводу немає.
- **Checkbox на зарахуванні з транзиту:** точний момент кредиту транзит→рахунок API не бачить. Два варіанти: (a) **приблизно** за `StatusCode 11` (+~30 хв — NovaPay платить швидко) — просто, збігається з поточним авто-Checkbox; (b) **точно за випискою** — лише якщо отримаємо клієнт-банк фід NovaPay Бізнес. Не обіцяти (b), доки NovaPay не підтвердить доступ.
- **Не переобіцяти:** «бачити баланси/рухи NovaPay у CRM» з документованих API **недосяжно** — лише через кабінет-фід (партнерське підключення) або показувати «очікувану» суму в дорозі, розраховану зі статусів НП.

## 5. Що вже є в коді (фундамент — розширювати)

- **Рахунки:** `MgrBankAccount` (kind account/card/cash, реквізити; **немає поля валюти** — валюта на документах).
- **Касові ордери:** `MgrCashOrder` + `cash-order.ts` (`createPaymentOrders`, `postCashOrder`, draft/post; проведення = ДДС + борг + архів; поля `source`, `verifiedAt` для звірки).
- **Банк-платежі + перекази:** `BankPaymentIncoming/Outgoing`, `CashTransfer` + `treasury-posting.ts` (`postBankPaymentIncoming` → ДДС-прихід + борг↓; `postBankPaymentOutgoing` → ДДС-розхід; `postCashTransfer` → 2 ноги). **Саме сюди вписується банківський фід.**
- **Борг:** `MgrDebtMovement` + `applyDebtMovementSafe` (ідемпотентно, +збільшує/−зменшує).
- **ДДС:** `CashFlowMovement` + `MgrCashFlowArticle` (статті, дерево, `direction`), звіт `cashflow-flex.ts`.
- **Checkbox:** `lib/fiscal/*` — **лише ETTN-наложка** (`/np/ettn`). Для чека на банк/касову операцію потрібен новий тип чека (`/receipts/sell`) + новий payload-білдер + env.
- **NovaPay:** `novapay-reconcile.ts` — шаблон «опитати статус → авто-створити+провести прихід → позначити `source` → сповістити бухгалтера → щоденна звірка `verifiedAt`». **Це точний патерн для банківського фіда.**
- **Крони:** `app/api/cron/*` з `CRON_SECRET` (взірець `refresh-np-shipments`).
- **Ролі:** `bookkeeper` (view/create/edit payments+finance), гейт `canManageTreasury` = {admin, owner, bookkeeper}.

## 6. Двигун, який будуємо

1. **Банківський фід** → Monobank webhook (миттєво) + PrivatBank `interim` опитування (1–5 хв).
2. **Архів транзакцій** (нова таблиця `BankTransaction`, сирий рядок, дедуп по банківському `id`) — незмінний, для аудиту.
3. **Нормалізатор** → єдина форма (рахунок, валюта, сума±, контрагент ЄДРПОУ/IBAN/назва, призначення, дата).
4. **Матчинг** → клієнт за ЄДРПОУ/IBAN/назвою + парсинг призначення на № замовлення (`L0000002477`).
5. **Рішення:**
   - прихід + знайдено клієнта → створити+провести `BankPaymentIncoming` → борг↓, ДДС+, дашборд оновивсь;
   - прихід без клієнта → чернетка + прапор **«надійшли гроші, не рознесено»** (закриває біль user);
   - розхід → `BankPaymentOutgoing` (ДДС-розхід).
6. **Дошка звірки бухгалтера** (як `novapay-check`) — підтвердити авто-проведення.

## 7. Готівкова каса + звірка рахунків

- **Каса грн/євро/долар:** біжучий залишок з проведених касових ордерів (дані вже є через ДДС по `CASH_DESK`).
- **Щоденне підбиття:** бухгалтер вводить фактично порахований залишок → система порівнює з обліковим → показує **переліміт/недостачу** + пропонує документ вирівнювання (внесення/вилучення/корекція).
- **Звірка рахунків:** обліковий залишок (сума проведень) vs балансом із фіда банку → різниця = не рознесені операції.
- **Головна цінність:** «гроші є, а менеджер не створив прихід → у клієнта борг» — фід бачить гроші й або авто-створює прихід, або підсвічує червоним на дошці звірки.

## 8. Мої ідеї (вдосконалення)

1. Розумний матчинг за призначенням платежу (авто-лінк оплата↔реалізація за № документа).
2. Дошка «нерознесені гроші» — кожна вхідна транзакція без клієнта = червона; драйвить щоденну рутину бухгалтера.
3. Сповіщення менеджеру (Viber/in-app): «клієнт оплатив, борг закрито» / «оплата без документа».
4. Валютний знімок на кожній транзакції (нативна + UAH/EUR еквів.) для коректної маржі.
5. Фінансовий пульс дня на дашборді власника (притік/відтік/прогноз).
6. Checkbox на зарахуванні — новий тип чека (не ETTN) для каси/зарахування.
7. Авто-виявлення дублів/розбіжностей оплат.
8. Незмінний архів виписки для аудиту (дедуп по банківському id).

## 10. Що потрібно від user (не блокує старт аналізу, блокує реалізацію)

- **Monobank:** згенерувати `X-Token` (api.monobank.ua, ФОП) — ми ставимо webhook + WAF-skip. Найшвидший старт.
- **PrivatBank:** увімкнути Автоклієнт у Приват24 бізнес → отримати токен; підтвердити назву заголовка та (для Фази 6) чи можливий підпис поза кабінетом.
- **NovaPay:** визнати, що публічного API виписки немає → або живемо на реконсиляції зі статусів НП (вже є), або запитуємо клієнт-банк фід у NovaPay Бізнес (партнерське підключення).
- **Checkbox:** підтвердити, що для чеків на банк/касові операції заводимо новий тип чека (каса L-TEX у Checkbox) — поточний код лише для наложки НП.
- **Статті ДДС + рахунки:** налаштувати довідник статей і прив'язки (яка стаття для якого типу надходження/витрати).

## 11. Ідентифікація платежів — «воронка впізнавання» (обговорення 2026-07-24)

**Контекст від user:** бухгалтера, який вручну обробляє кожну транзакцію, НЕМАЄ і не буде — автоматизувати максимально. Ідеї user: (1) система знає, яке повідомлення з реквізитами і на яку суму сформували для конкретного клієнта — використовувати як сигнал звʼязки; (2) клієнти скидають скріни/PDF квитанцій — розпізнавати їх і звʼязувати контрагента з оплатою, або хоча б сповіщати менеджера; (3) додати обовʼязково ПІБ платника / «за кого сплачують» у призначення платежу в реквізитах (ПІБ з картки клієнта). Вихідні платежі — не потрібні.

**Воронка (кожен рівень — сигнал; бали сумуються, поріг → авто-проведення):**

1. **Код/ПІБ у призначенні платежу (керований нами).** Реквізити формує НАША система (`buildPaymentRequisitesText`) → робимо шаблон призначення обовʼязковим: «Оплата за товар, {ПІБ з картки клієнта}, {номер документа}». Privat несе `OSND`, mono — `comment`/`description`. Точний збіг коду = майже 100%. Обмеження: працює при оплаті за IBAN-реквізитами; при переказі на картку призначення часто губиться/не вводиться.
2. **«Очікування оплати» (PaymentExpectation) — ідея user №1.** Коли менеджер тисне «Скинути реквізити»/«Реквізити передоплати» на суму X для клієнта Y → пишемо запис-очікування (clientId, saleId, сума, рахунок реквізитів, createdAt, TTL ~7–14 дн). Вхідна транзакція на ту саму суму до копійки в вікні → дуже сильний сигнал. **Наша перевага: суми некруглі** (вага×ціна/кг, типу 18 437,50) — колізії рідкісні.
3. **Памʼять платників (самонавчання) — нова модель `ClientPayerRequisite`.** Перший невідомий платіж лінкується вручну одним кліком → система запамʼятовує «IBAN/ЄДРПОУ/ПІБ платника ↔ клієнт» (у клієнта може бути кілька платників — дружина, інша картка). Далі всі платежі цього платника лягають самі. Через 1–2 місяці ручної роботи майже 0.
4. **Fuzzy-матч ПІБ/назви** (`counterName` проти ПІБ картки + назви ТТ; обережно: транслітерація, тезки) — лише допоміжний сигнал, сам по собі не проводить.
5. **Квитанції в чаті (ідея user №2) — через НАЯВНИЙ чат-inbox.** Розмова Viber/Telegram вже привʼязана до клієнта по телефону → клієнт кидає скрін/PDF квитанції боту → AI-розпізнавання (сума, дата/час, банк, останні цифри картки/рахунку) → створюємо «заявлену оплату» від цього клієнта + сповіщення менеджеру. **Квитанція ≠ гроші** (скрін можна підробити/переслати старий) → документ проводиться ЛИШЕ коли банк-фід підтвердив реальну транзакцію; квитанція виступає сигналом звʼязки (сума+час+клієнт з чату) і причиною раннього сповіщення.
6. **Нічого не зійшлось** → дошка «Нерознесені гроші», лінк в 1 клік (який заразом навчає рівень 3).

**Пороги (пропозиція):** ≥2 незалежні сильні сигнали (напр. очікування суми + збіг платника; або код у призначенні) → авто-створити + провести прихід; 1 сильний → чернетка + сповіщення менеджеру «підтверди в 1 клік»; 0 → дошка. Усі авто-проведені — на щоденній дошці звірки (patтерн novapay-check).

**Додаткова ідея — QR-код у повідомленні з реквізитами:** НБУ має стандарт QR для кредитових переказів; Приват24/mono вміють сканувати QR з передзаповненими IBAN+сумою+призначенням → клієнт фізично не може зіпсувати призначення. Перевірити підтримку в банках на Фазі 3; якщо працює — це найдешевший спосіб зробити рівень 1 надійним.

**Технічна примітка для реалізації:** розпізнавання квитанцій — vision-модель через Claude API (JPEG/PDF квитанції Privat24/mono стандартизовані, парсяться добре); зберігати результат розпізнавання + оригінал файла біля «заявленої оплати»; ліміт розміру і антидубль по хешу файла.
