# Аудит блоку «Оплати / Каса» (1С `DataProcessor.Оплата` + `Document.КассовыйОрдер`)

> READ-ONLY аудит старої програми менеджера (`MobileAgent`, 1С:Підприємство) для точного
> відтворення блоку **«Оплати / Каса»** у нашому веб-стеку. Терміни 1С наведені кирилицею
> з коротким англ.-глосом. Джерела цитуються по файлах/рядках.
>
> Ключові файли:
>
> - Обробка-форма: `docs/1c-export-mobile/MobileAgent/DataProcessors/Оплата/Forms/Форма/Ext/{Form.xml,Form/Module.bsl}`
> - Атрибути обробки: `MobileAgent/DataProcessors/Оплата.xml` (ManagerModule.bsl — порожній)
> - Документ каси: `MobileAgent/Documents/КассовыйОрдер.xml` + `…/Ext/ObjectModule.bsl`
> - Форми каси: `MobileAgent/Documents/КассовыйОрдер/Forms/{ФормаДокумента,ФормаСписка}/Ext/{Form.xml,Form/Module.bsl}`
> - Агрегація оплат: `MobileAgent/CommonModules/Общий/Ext/Module.bsl` → `ПолучитьДанныеПоОплате`, `ПолучитьБанковскиеСчетаОплат`, `ОбновитьСумаОплатиНаложкою`
> - Контроль оплати: `MobileAgent/Documents/РеализацияТоваровУслуг/Forms/ФормаДокумента/Ext/Form/Module.bsl` → `КонтрольОплати`, `ПроверкаПолнойОплаты`, `СтворитиОплату`
> - Обмін з центральною 1С: `Central/CommonModules/ОбменАРМ/Ext/Module.bsl` → `СоздатьПКО` (рядок 4629)

## Загальна модель (як це працює у старій програмі)

1. У документі **Реалізація** (`РеализацияТоваровУслуг`) менеджер тисне «Створити оплату»
   (`СтворитиОплату`, Реалізація-форма рядок 1001). Перед цим — `КонтрольОплати` (дубль-гард).
2. Відкривається **обробка `Оплата`** (`Обработка.Оплата.Форма`) з параметрами
   `Реализация / Контрагент / СуммаКОплате / МаршрутныйЛист / ЗаказПокупателя`.
3. Менеджер вводить отримані суми в **3 валютах (UAH / EUR / USD) + UAH безнал** і, якщо
   потрібно, **здачу (решта)** в тих самих валютах. Форма онлайн рахує «Оплата документа»,
   «Решта документа», «Залишок» у EUR (базова валюта).
4. Кнопка **«Сформувати»** (`Сформировать`) створює **проведений `Document.КассовыйОрдер`**
   типу **Приход** на суми оплати і, якщо є здача, **другий `КассовыйОрдер`** типу **Расход**,
   що посилається на прихідний (`СдачаДокумента`).
5. Борг клієнта / переплата рахуються НЕ окремим регістром у мобільному, а **агрегацією
   проведених касових ордерів** (`ПолучитьДанныеПоОплате`) через регістр накопичення
   `ДенежныеСредства`.
6. Касові ордери вивантажуються в **центральну 1С**, де `СоздатьПКО` будує справжні
   `ПриходныйКассовыйОрдер` / `РасходныйКассовыйОрдер` / `ПлатежноеПоручениеВходящее(Исходящее)`
   (для безналу) з рухом по взаєморозрахунках.

### Підказки-рекомендації (LabelField, текст біля поля, колір `FieldSelectionBackColor`)

`ОплатаUAHРек`, `ОплатаEURРек`, `ОплатаUSDРек` — «скільки ще треба внести» у кожній валюті;
`СдачаUAHРек`, `СдачаEURРек`, `СдачаUSDРек` — «скільки решти належить» у кожній валюті.

### Підсумкові написи (read-only під полями)

- `ОплатаДокумента` — **«Сума оплати (EUR)»** (сумарна оплата зведена в EUR).
- `СдачаДокумента` — **«Сума решти (EUR)»**.
- `ОстатокДокумента` — **«Сума залишку (EUR)»** (скільки ще не оплачено / переплата зі знаком).

### Кнопки

- **`Сформировать`** («Сформувати») — створює касові ордери і закриває форму, повертаючи
  масив створених ордерів (`Сформировать` → `СформироватьНаСервере` → `Закрыть(...)`).
- **`ДатьСкидкуНаОстаток`** («Дати знижку на залишок») — списує дрібний залишок як знижку у
  найдорожчий рядок Реалізації; **доступна лише** коли `|залишок| ≤ ПорогЗадолженостиEUR`
  (константа). Це дозволяє «закрити» документ повністю, замість дрібного боргу/переплати.
- **`ОтправитьВайбер`** («Вайбер») — формує текст реквізитів оплати (для безналу) або тексту
  витрат і шерить у Viber (`ПолучитьТекстДляВайбер`).

### Defaults / автозаповнення (`ПриСозданииНаСервере`, рядки 6-58)

- `КурсEUR = Константы.КурсEUR.Получить()`, `КурсUSD = Константы.КурсUSD.Получить()` — **знімок
  курсів** з констант на момент відкриття.
- Якщо `СуммаКОплате = 0` → береться `Контрагент.Борг`.
- `ОплатаUAHРек = Окр(СуммаКОплате * КурсEUR)`; `ОплатаEURРек = СуммаКОплате`;
  `ОплатаUSDРек = СуммаКОплате * КурсEUR / КурсUSD`.
- `ОстатокДокумента = СуммаКОплате`.
- `ВидДвиженияДенежныхСредств = Приход` за замовчуванням; при наявності Реалізації поле
  напряму ховається.

## B. Розрахунок здачі (РЕШТА) — ТОЧНА формула

Уся арифметика — у `СуммаПриИзмененииНаСервере()` (`Form/Module.bsl` 60-77). Викликається з
`OnChange` будь-якого поля оплати/решти/курсу. **Базова валюта зведення — EUR.**

```bsl
// 1) Зведення фактичної оплати в EUR (готівка+безнал UAH/EUR/USD):
Объект.ОплатаДокумента = Объект.ОплатаEUR
    + Окр(Объект.ОплатаUAH        / Объект.КурсEUR, 2)
    + Окр(Объект.ОплатаUAH_Безнал / Объект.КурсEUR, 2)
    + Окр(Объект.ОплатаUSD * Объект.КурсUSD / Объект.КурсEUR, 2);

// 2) Зведення фактичної здачі в EUR (UAH/EUR/USD, без безналу):
Объект.СдачаДокумента = Объект.СдачаEUR
    + Окр(Объект.СдачаUAH / Объект.КурсEUR, 2)
    + Окр(Объект.СдачаUSD * Объект.КурсUSD / Объект.КурсEUR, 2);

// 3) Залишок (скільки ще не оплачено; додатній = борг, відʼємний = переплата/решта):
Объект.ОстатокДокумента = Объект.СуммаКОплате - Объект.ОплатаДокумента + Объект.СдачаДокумента;

// 4) Рекомендації «до сплати» (нуль, якщо вже переплачено):
Объект.ОплатаUAHРек = Окр(?((СуммаКОплате - ОплатаДокумента) < 0, 0, Окр((...) * КурсEUR, 2)));
Объект.ОплатаEURРек = ?((СуммаКОплате - ОплатаДокумента) < 0, 0, (СуммаКОплате - ОплатаДокумента));
Объект.ОплатаUSDРек = ?((СуммаКОплате - ОплатаДокумента) < 0, 0, Окр((...) * КурсEUR / КурсUSD, 2));

// 5) Рекомендована РЕШТА (видається лише коли ОстатокДокумента < 0, тобто переплата):
Объект.СдачаUAHРек = ?(ОстатокДокумента >= 0, 0, ОстатокДокумента * КурсEUR * (-1));
Объект.СдачаEURРек = ?(ОстатокДокумента >= 0, 0, ОстатокДокумента * (-1));
Объект.СдачаUSDРек = ?(ОстатокДокумента >= 0, 0, Окр(ОстатокДокумента * КурсEUR / КурсUSD * (-1), 2));
```

Ключові висновки про **здачу**:

- **Курси — знімок** (`КурсEUR`, `КурсUSD` з констант на момент відкриття; редаговані вручну в
  згорнутій групі). Перерахунок при зміні курсу — `КурсПриИзменении` (212-219) скидає
  рекомендації й `ОстатокДокумента = СуммаКОплате`.
- **UAH→EUR**: ділення на `КурсEUR`. **USD→EUR**: `USD * КурсUSD / КурсEUR` (тобто USD→UAH→EUR).
- Округлення: усі компоненти зведення — `Окр(..., 2)` (2 знаки). Рекомендація UAH — `Окр(... , 0)`
  (ціла гривня) у `ПриСозданииНаСервере`, але `Окр(...,2)` у `СуммаПриИзмененииНаСервере`.
- **Здача НЕ обчислюється автоматично у конкретній валюті** — менеджер вводить її руками
  (`СдачаUAH/EUR/USD`); система лише **підказує** рекомендовані суми решти у 3 валютах
  (`СдачаXxxРек`) і зводить введену решту в EUR (`СдачаДокумента`).
- `ОстатокДокумента` — підсумковий «недоплат/переплата» у EUR з урахуванням вже введеної решти.
- **`СуммаПриИзменении`** (клієнт, 80) щоразу обнуляє `КоректировкаДолга = 0` (щоб ручна знижка
  не накопичувалась).

`ВключитиСумуБоргуПриИзмененииНаСервере` (239-256) перераховує `СуммаКОплате`:
`мБорг(=Контрагент.Борг якщо включено) + Реализация.Сума − вже_оплачено_по_цій_реалізації_у_маршруті`.

`ДатьСкидкуНаОстаток` (162-202): якщо є Реалізація — зменшує `Сума`/`Цена` найдорожчого рядка
на `ОстатокДокумента`, ставить `Знижка=Истина`, перепроводить Реалізацію; якщо Реалізації нема —
коригує `СуммаКОплате` і пише `КоректировкаДолга = ОстатокДокумента`.

## C. Авто-створення другого (Расход) ордера на ЗДАЧУ

**Перший ордер (Приход)** — створюється, якщо сума оплати > 0

```bsl
НовыйКассовыйОрдер.СуммаДокумента = Объект.ОплатаДокумента;   // зведено в EUR
НовыйКассовыйОрдер.СумаUAH        = Объект.ОплатаUAH;
НовыйКассовыйОрдер.СумаUAH_Безнал = Объект.ОплатаUAH_Безнал;
НовыйКассовыйОрдер.БанковскийСчет = Объект.БанковскийСчет;
НовыйКассовыйОрдер.СумаEUR        = Объект.ОплатаEUR;
НовыйКассовыйОрдер.СумаUSD        = Объект.ОплатаUSD;
НовыйКассовыйОрдер.Основание      = Объект.Реализация;       // → Document.РеализацияТоваровУслуг
НовыйКассовыйОрдер.Контрагент     = Объект.Контрагент;
НовыйКассовыйОрдер.КурсEUR = Объект.КурсEUR;  НовыйКассовыйОрдер.КурсUSD = Объект.КурсUSD;
НовыйКассовыйОрдер.МаршрутныйЛист = Объект.МаршрутныйЛист;
НовыйКассовыйОрдер.ЗаказПокупателя = Объект.ЗаказПокупателя;
НовыйКассовыйОрдер.КоректировкаДолга = Объект.КоректировкаДолга;
НовыйКассовыйОрдер.УИДКоректировки  = Новый УникальныйИдентификатор;
// Записать(Проведение); у разі помилки — Записать(Запись)  → у tabular ДокументыОплаты
ПКО = НовыйКассовыйОрдер.Ссылка;   // зберігаємо посилання для звʼязку зі здачею
```

**Другий ордер (Расход / здача)** — створюється, якщо є введена здача
(`СдачаEUR+СдачаUAH+СдачаUSD > 0`), рядки 120-149:

```bsl
НовыйКассовыйОрдер = Документы.КассовыйОрдер.СоздатьДокумент();
НовыйКассовыйОрдер.Дата = ТекущаяДата();
НовыйКассовыйОрдер.ВидДвиженияДенежныхСредств = Перечисления.ВидДвиженияДенежныхСредств.Расход;
НовыйКассовыйОрдер.СуммаДокумента = Объект.СдачаДокумента;  // зведено в EUR
НовыйКассовыйОрдер.СумаUAH = Объект.СдачаUAH;
НовыйКассовыйОрдер.СумаEUR = Объект.СдачаEUR;
НовыйКассовыйОрдер.СумаUSD = Объект.СдачаUSD;
НовыйКассовыйОрдер.Основание = Объект.Реализация;
НовыйКассовыйОрдер.Контрагент = Объект.Контрагент;
НовыйКассовыйОрдер.КурсEUR = …; НовыйКассовыйОрдер.КурсUSD = …;
НовыйКассовыйОрдер.МаршрутныйЛист = …; НовыйКассовыйОрдер.ЗаказПокупателя = …;
НовыйКассовыйОрдер.СдачаДокумента = ПКО;   // ← звʼязок назад на прихідний ордер
// КоректировкаДолга пишеться сюди ЛИШЕ якщо прихідного ордера не було (УстановкаКоректировкаДолга=Ложь)
// Записать(Проведение)/Записать(Запись) → у tabular ДокументыОплаты
```

Ключове:

- **Звʼязок здача→оплата** робиться полем `СдачаДокумента` (тип `DocumentRef.КассовыйОрдер`) у
  Расход-ордері, яке вказує на прихідний `ПКО`. Це аналог нашого `changeForId`.
- Здача — **окремий проведений документ типу Расход** (не рядок у прихідному).
- Безнал у здачу **не йде** (у Расход-ордері немає `СумаUAH_Безнал`).
- Обидва ордери пишуться у tabular `ДокументыОплаты` обробки (для повернення масиву викликачу).

## D. Схема документа `КассовыйОрдер` (Document)

Файл: `Documents/КассовыйОрдер.xml`. Синонім uk = **«Оплати»**, ru = «Кассовый ордер».
Нумерація рядкова, авто, періодичність — рік. **Проведення дозволено**; рухає регістр
накопичення **`AccumulationRegister.ДенежныеСредства`**. `PostInPrivilegedMode = true`.

### Реквізити (Attributes)

## E. Логіка проведення + агрегація боргу

### Проведення документа (`КассовыйОрдер/Ext/ObjectModule.bsl`)

`ПередЗаписью` (14-29): якщо `ВидДвиженияДенежныхСредств` порожній → **Приход**; `Архив`
синхронізується з `ПометкаУдаления`; генеруються `УИДUSD`/`УИДГРН`, якщо порожні.

`ОбработкаПроведения` (31-45) — **єдиний рух**: регістр накопичення `ДенежныеСредства`:

```bsl
Движение.Период = Дата;
Движение.Сумма  = СуммаДокумента;                 // у EUR (база)
Движение.ВидДвиженияДенежныхСредств = Приход (default) АБО фактичний (Расход для здачі);
```

Тобто у мобільному **рух — лише сумарний EUR** (`СуммаДокумента`) з ознакою Приход/Расход. Окремих
регістрів каси/банку/взаєморозрахунків у мобільному НЕМАЄ — це робить центральна 1С при обміні.
`ПриЗаписи` реєструє зміни для обміну (`Общий.ЗарегистрироватьИзменения`), окрім випадку обміну.

### Агрегація: `ПолучитьДанныеПоОплате(Объект)` (`Общий/Ext/Module.bsl` 300-375)

Запит по `Документ.КассовыйОрдер` ЛІВ. СОЕД. `РегистрНакопления.ДенежныеСредства` за умовою:

- якщо `Объект` — Реалізація: `КассовыйОрдер.Основание = &Основание`;
- інакше (ордер як підстава): `КассовыйОрдер.Ссылка = &Основание Или КассовыйОрдер.СдачаДокумента = &Основание`;
- **`И КассовыйОрдер.Проведен`** (лише проведені).

Повертає структуру `Оплаты(Курс, Суммы, Здача, Взаиморасчеты, СуммаУчета)`:

```bsl
Оплаты.Курс    = Структура(EUR, USD)                          // з першого запису
Оплаты.Суммы   = Структура(EUR, USD, UAH, UAH_Безнал)         // сума ПРИХОДУ по валютах
Оплаты.Здача   = Структура(EUR, USD, UAH)                     // сума РОЗХОДУ (здача) по валютах
Оплаты.СуммаУчета = 0                                          // нетто у EUR (Приход − Расход)
Оплаты.Взаиморасчеты = Структура(ВидРеализации, Статус, Сумма)
```

Цикл по вибірці:

- **Приход** → `Суммы.EUR/USD/UAH/UAH_Безнал += відповідна валюта`;
  `СуммаУчета += ДенежныеСредства.Сумма` (тобто += `СуммаДокумента` у EUR).
- **Расход** (здача) → `Здача.EUR/USD/UAH += валюта`; `СуммаУчета -= ДенежныеСредства.Сумма`.

Тобто **`СуммаУчета` = чиста оплата по реалізації у EUR** (приходи мінус здачі).

Розрахунок **взаєморозрахунків** (лише для Реалізації, 360-372):

```bsl
Взаиморасчеты.Сумма = Объект.Сума - Оплаты.СуммаУчета + Объект.Контрагент.Борг;
Якщо < 0 → Сумма = -Сумма; ВидРеализации=" по передплаті"; Статус="Передплата";
Інакше якщо Объект.Наложка → ВидРеализации=" по післяплаті";
// інакше Статус залишається "Борг"
```

У тексті повідомлення (`ПолучитьТекстСообщенияВайбер`, 215-247):

- `Борг по документу = Объект.Сума − Оплаты.СуммаУчета` (якщо > 0), інакше
  `Передоплата = Оплаты.СуммаУчета − Объект.Сума`.
- Якщо `Наложка` І борг>0 → показ **«Наложений платіж»** = `СумаОплатиНаложкою` (грн).
- «Оплачено … €», далі «Фактична оплата:» з розбивкою по валютах
  (`Оплаты.Суммы`: UAH/UAH*Безнал/EUR/USD), для UAH*Безнал додає рядок банк. рахунків
  (`ПолучитьБанковскиеСчетаОплат`, 377-407).
- «Фактична решта:» з розбивкою (`Оплаты.Здача`), якщо здача ≠ 0.

`СумаОплатиНаложкою` (Реалізація) рахується як
`Окр((Объект.Сума − Оплаты.СуммаУчета) * Оплаты.Курс.EUR, 0)` — **залишковий борг у грн за
знімком курсу** (Реалізація-форма 902/943/978/1446; `Общий.ОбновитьСумаОплатиНаложкою` 86-91).

### Рекомендації у формі документа `КассовыйОрдер` (`ФормаДокумента/Form/Module.bsl`)

`СумаUAHПриИзмененииНаСервере` (2-11) — той самий перерахунок зведення в EUR
(`СуммаДокумента = СумаEUR + СумаUAH/КурсEUR + СумаUAH_Безнал/КурсEUR + СумаUSD*КурсUSD/КурсEUR`)
плюс «рекомендовано до сплати»:

```bsl
РекомендованоДоСплатиEUR = СуммаРеализации − (ОплаченаСума + СуммаДокумента);  // СуммаРеализации = Основание.Сума
РекомендованоДоСплатиUAH = РекомендованоДоСплатиEUR * КурсEUR;
РекомендованоДоСплатиUSD = РекомендованоДоСплатиUAH / КурсUSD;
```

де `ОплаченаСума` — сума обороту регістра `ДенежныеСредства` по інших ордерах з тією ж
підставою (`РекомендацииПоОплате`, 94-123, запит з `КассовыйОрдер.Ссылка <> &Ссылка`).

## F. `КонтрольОплати` / валідація (анти-дубль, повна оплата, наложка)

Файл: `РеализацияТоваровУслуг/Forms/ФормаДокумента/Ext/Form/Module.bsl`.

### `КонтрольОплати(ЗаказПокупателя)` (1267-1287) — дубль-гард

```bsl
Якщо ЗаказПокупателя не заповнено → Возврат Ложь;
Запит: чи є хоч один КассовыйОрдер з ЗаказПокупателя = &ЗаказПокупателя;
Возврат Выборка.Следующий();   // True = вже є оплата
```

Використання у `СтворитиОплату` (1007-1010):

```bsl
Если КонтрольОплати(Объект.Сделка) Тогда
    ПоказатьПредупреждение(, "По даній реалізації вже є оплата!");
    Возврат;   // блокуємо повторне створення оплати
```

> Тобто **на одне Замовлення (`Сделка`) дозволено лише один цикл оплати** через цю кнопку
> (перевірка по `ЗаказПокупателя`, не по `Основание`).

### `ПроверкаПолнойОплаты(Отказ)` (1309-1326) — попередження про неповну оплату

```bsl
Если Доставка = НаявністьДоставки.Почта И Наложка = Ложь Тогда
    Оплаты = ПолучитьДанныеПоОплате(Объект.Ссылка);
    Если (Объект.Сума − Оплаты.СуммаУчета) > 0 Тогда   // ще є борг
        Вопрос("Ви дійсно хочете створити реалізацію без повної оплати?");
        Якщо «Нет» → Отказ = Истина;
```

Тобто **при доставці Поштою без наложки** документ не можна «відправити», якщо борг > 0
(питання-підтвердження). Викликається у `ПередЗакрытием` і перед «Відправити Viber».

### Наложка (COD) — взаємодія

- Реалізація має boolean `Наложка` + `СумаОплатиНаложкою` (грн). При увімкненій наложці система
  **не вимагає** повної оплати при пошті (умова `И Наложка = Ложь`).
- `СумаОплатиНаложкою` = залишковий борг у грн (див. §E) — це сума накладеного платежу для НП.
- У тексті Viber наложка показується окремим рядком «Наложений платіж … грн».

### Гард банк. рахунку при приході (`БанковскийСчетПриИзмененииНаСервере`, 512-518)

Якщо обраний рахунок має `НеВідображатиВДодатку = Истина` і вид = **Приход** → рахунок
скидається з повідомленням «… не можна вибирати при приході!».

## H. Контракт обміну з центральною 1С (вивантаження каси)

Файл: `Central/CommonModules/ОбменАРМ/Ext/Module.bsl` → `СоздатьПКО(МассивДокументов, ТаблицаИД, МассивОшибок)` (рядок 4629).

Вхід — масив касових ордерів з мобільного. Ключові моменти:

### Бізнес-ключі (UUID)

```bsl
Реализация_УИД, МаршрутныйЛист_УИД, ЗаказПокупателя_УИД, Контрагент_УИД,
БанковскийСчет_УИД, СтатьяДвиженияДенежныхСредств_УИД,
УИДГРН (UAH), ПКО_УИД (EUR), УИДUSD (USD)        // по одному документу-валюті на ключ
ВидДвижения = "Приход" | "Расход";  Сдача = boolean;  Номер, Дата, КурсEUR, КурсUSD
СумаUAH, СумаUAH_Безнал, СумаEUR, СумаUSD
```

### Логіка `СоздатьПКО`

1. **Пошук Реалізації** в 1С за `Реализация_УИД`, інакше — за `ЗаказПокупателя`+`Контрагент`
   (запит по `РеализацияТоваровУслуг.Сделка`). Якщо не знайдено — помилка
   «Не знайдена реалізація в 1С для оплати …», пропуск.
2. **Безнал** (`СумаUAH_Безнал > 0`, 4731-4845): створюється **`ПлатежноеПоручениеВходящее`**
   (Приход) або **`ПлатежноеПоручениеИсходящее`** (Расход), валюта 980 (UAH), `ВидОперации =
ОплатаПокупателя`, `СтатьяДвиженияДенежныхСредств = ОплатаПокупателя`, заповнюється
   `РасшифровкаПлатежа` (Сделка/Реалізація/ДоговорКонтрагента/СуммаВзаиморасчетов) і проводиться.
3. **Готівка по 3 валютах** (4849-…): цикл по `СтруктураВалют{СумаUAH→980, СумаEUR→978, СумаUSD→840}`:
   - **Приход** → `ПриходныйКассовыйОрдер`, `ВидОперации = ВидыОперацийПКО.ОплатаПокупателя`,
     `.Заполнить(Реализация)`, `Касса = Кассы по валюті`, `Основание = Реалізація`,
     `ВалютаДокумента = валюта`, `СуммаДокумента = СуммаПКО`;
     `РасшифровкаПлатежа.СтатьяДвиженияДенежныхСредств = ОплатаПокупателя`.
   - **Расход** + `Сдача=Истина` → `РасходныйКассовыйОрдер`,
     `ВидОперации = ВидыОперацийРКО.ВозвратДенежныхСредствПокупателю`,
     `РасшифровкаПлатежа.СтатьяДвиженияДенежныхСредств = ВозвратДенежныхСредствПокупателю`.
   - **Расход** без здачі → `РасходДенежныхСредствПрочее` зі статтею з мобільного.
   - Курси: `КурсEUR/КурсUSD/КурсДокумента` беруться з елемента; договір — `ПолучитьДоговорКонтрагента`
     або «ДоговорДляНеплановыхПродаж»; одиниця кратності = 1.
   - `СуммаВзаиморасчетов` рахується через перерахунок валют (для здачі `СуммаПКО*КурсПКО/КурсДоговора`).
4. **Ідемпотентність**: ордери шукаються за UUID (`ПолучитьСсылку`), `УстановитьСсылкуНового` —
   повторний імпорт оновлює, а не дублює.

> Підсумок контракту: **один мобільний `КассовыйОрдер` → до трьох документів у центральній 1С**
> (по одному на валюту з ненульовою сумою) + окремий безнал-документ. Здача = окремий
> РасходныйКассовыйОрдер з `Сдача=Истина`. Ключі звʼязку — `УИДГРН`/`ПКО_УИД`/`УИДUSD`, плюс
> `Реализация_УИД`/`ЗаказПокупателя_УИД` для прив'язки до підстави. Рух — по взаєморозрахунках
> (борг клієнта зменшується через `РасшифровкаПлатежа`).

### Прогалини (gaps) щодо точного відтворення старого додатку

1. **Курси-знімок** (`КурсEUR`/`КурсUSD`) на ордері — відсутні; зараз, ймовірно, беруться
   глобально. Потрібні для відтворення формули зведення та для обміну.
2. **Мультивалютна здача**: 1С тримає `СдачаUAH/EUR/USD` у Расход-ордері; наш `changeCurrency`
   (одна валюта) — спрощення. Для повної відповідності треба зберігати 3 суми здачі (або
   просто використати ті самі `amountUah/Eur/Usd` на Расход-ордері — як і робить 1С).
3. **Довідник банк. рахунків** (`БанковскиеСчета` з `Опис`, `НеВідображатиВДодатку`) — у нас
   `bankAccount` як вільний рядок. Бракує: фільтр рахунків по контрагенту, прапор «не показувати»,
   гард «не можна при приході».
4. **Довідник статей руху ГК** (`СтатьиДвиженияДенежныхСредств`) — у нас `cashFlowArticle` рядок.
5. **Агрегація боргу** (`ПолучитьДанныеПоОплате`): зведення `СуммаУчета` = Приходи − Здачі у EUR,
   статуси Борг/Передплата/«по післяплаті», розбивка фактичної оплати/решти по валютах,
   `СумаОплатиНаложкою`. Це ядро «скільки оплачено / залишок» — треба реалізувати як сервіс
   над `MgrCashOrder` (по `saleId`, лише `posted`).
6. **Рекомендації до сплати** (`ОплатаXxxРек` / `СдачаXxxРек` / `РекомендованоДоСплатиXxx`) у формі.
7. **Кнопка «Дати знижку на залишок»** (поріг `ПорогЗадолженостиEUR`) + `КоректировкаДолга` — нема.
8. **Список з фільтром архівних** + сіре підсвічування архіву + заборона прямого створення/видалення
   (створення лише через payment-форму). Поле `Архив`.
9. **Анти-дубль `КонтрольОплати`** (одна оплата на Замовлення) і **`ПроверкаПолнойОплаты`**
   (попередження при пошті без наложки і борг>0).
10. **Прив'язка до Контрагента** на ордері (зараз — лише через `sale`).
11. **Контракт обміну**: мультивалютні UUID-ключі + ВидДвижения/Сдача-прапор для `СоздатьПКО`.

## J. Підказки до плану відтворення блоку «Оплати» (seed для наступного плану)

**Моделі / довідники:**

- Розширити `MgrCashOrder`: `rateEur`, `rateUsd` (знімок), `changeAmountUah/Eur/Usd` (або
  трактувати Расход-ордер як той самий запис із сумами здачі), `archived`, `customerId`,
  `routeListId?`, `orderId?`, `debtCorrection?`, `correctionUid?`, `uidUah/uidEur/uidUsd`
  (бізнес-ключі обміну).
- Новий довідник **`MgrBankAccount`** (опис, `hideInApp`, привʼязка до клієнта) — замінити вільний
  рядок `bankAccount`.
- Новий довідник **`MgrCashFlowArticle`** (статті руху ГК) — замінити вільний рядок.
- Константа/налаштування **`debtThresholdEur`** (`ПорогЗадолженостиEUR`).

**Сервіси / endpoints:**

- `getPaymentSummary(saleId)` — порт `ПолучитьДанныеПоОплате`: сума по валютах (Приход),
  здача по валютах (Расход), `сумаУчета` (нетто EUR, лише `posted`), статус Борг/Передплата,
  `codAmountUah` (`СумаОплатиНаложкою`).
- `POST /payments` — приймає 3 валюти оплати + безнал + 3 валюти здачі + курси-знімок; **створює
  income-ордер** і (за наявності здачі) **expense-ордер** зі звʼязком `changeForId`; формула
  зведення в EUR (§B); анти-дубль гард (одна оплата на замовлення/реалізацію).
- Перевірка повної оплати (попередження при пошті без наложки).
- Дія «дати знижку на залишок» (поріг + корекція рядка реалізації).
- `enqueueCashOrder*` для майбутнього обміну з 1С (мультивалютний контракт §H: до 3 ПКО/РКО +
  безнал як платіжне доручення; ключі UUID; `Сдача`-прапор; `ВидДвижения`).

**UI:**

- **Payment-форма** (порт обробки `Оплата`): 4 поля оплати (UAH/UAH-безнал/EUR/USD) + банк. рахунок
  (видимий при безналі) + 3 поля здачі + згорнута група курсів; live-підсумки «Оплата/Решта/Залишок
  (EUR)» + рекомендації до сплати/решти по валютах; кнопки «Сформувати», «Знижка на залишок», «Viber».
- **Список «Оплати»** (`/manager/payments`): колонки Дата/Номер/Вид(Приход-Расход)/Контрагент/
  Стаття/Сума(EUR)/Банк.рахунок; фільтр «архівні» (за замовч. сховані) + сіре підсвічування архіву;
  створення лише через payment-форму, видалення заборонене.
- **Панель оплат у Реалізації** (вже є `payments-panel.tsx`) — додати зведення з §E (оплачено/борг/
  переплата/наложка) і розбивку по валютах/рахунках.
