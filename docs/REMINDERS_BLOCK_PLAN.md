# План блоку «Нагадування» (M3.0)

Базується на `docs/REMINDERS_BLOCK_AUDIT.md`. Блок **повністю локальний** (без
обміну з 1С). Рішення user (обсяг):

- **Ручні** нагадування: тип «Звичайне» + «Для товарів».
- **Справжні повторення** (Щодня/Щотижня/Щомісяця/Щороку перестворюються самі).
- **Авто-генеровані**: «з'явилось відео» + «минула бронь» (детектори по наших
  даних). **БЕЗ** «закриття старих замовлень» (окремого екрана нема — відкладено).
- **Спрацювання**: розширений **дзвіночок у шапці + список** (без модалки-попапа).

Один воркер на етап; orchestrator перевіряє → комітить на feature → мердж у main → deploy.

---

## Модель даних (Етап 1, additive міграція)

Розширюємо наявний `MgrReminder`:

| Поле                | Зміна                                             | Призначення                                |
| ------------------- | ------------------------------------------------- | ------------------------------------------ |
| `clientId`          | `String` → **`String?`** (relax NOT NULL)         | звичайне нагадування може бути без клієнта |
| `periodicity`       | новий enum `MgrReminderPeriod` (default `none`)   | none/daily/weekly/monthly/yearly/event     |
| `isProductReminder` | `Boolean @default(false)`                         | тип «Для товарів» (НагадатиПроТовар)       |
| `orderVideo`        | `Boolean @default(false)`                         | прапорець «Заказ відео»                    |
| `actionType`        | новий enum `MgrReminderAction` (default `none`)   | none/continue_bron/viber_video             |
| `source`            | новий enum `MgrReminderSource` (default `manual`) | manual/auto_video/auto_bron                |
| `lotId`             | `String?` (плоский скаляр, без relation)          | лот/мішок для авто-дії                     |
| `productId`         | `String?` (плоский скаляр)                        | товар для відео-дії                        |

Залишок без змін: `body`, `remindAt`, `completedAt`, `snoozedUntilAt`, `ownerUserId`.
`completedAt != null` = виконане (відповідник `Активне=Ложь`). `ДеньНедели` —
обчислюється з `remindAt`, окреме поле не треба.

Нова дочірня таблиця **`MgrReminderItem`** (для «Для товарів»):
`{ id, reminderId (FK Cascade), productId (плоский скаляр), quantity Int, done Boolean @default(false), createdAt }`.

PG-enum-и через snake_case `@@map` (як у проекті) + idempotent міграція.

## Етап 1 — Схема + standalone-екран «Звичайне» (список + створення + фільтри)

- **Міграція**: розширення `MgrReminder` + нові enum-и + `mgr_reminder_items`.
- **Recurrence-хелпер** (pure, tested) `lib/manager/reminder-recurrence.ts`:
  `nextOccurrence(remindAt, periodicity) → Date | null` (daily +1д, weekly +7д,
  monthly +1міс, yearly +1рік, none/event → null). + хелпер тексту-підказки
  («Кожен день о ГГ:хх» тощо).
- **API** `/api/v1/manager/reminders`:
  - `GET` — усі мої нагадування (ownership = ownerUserId), фільтри `completed`
    (за замовч. лише активні), `orderVideo`, пошук по тексту, пагінація.
  - `POST` — створити (clientId опційний; тип/періодичність/orderVideo).
  - `/[id]` `PATCH` (complete/uncomplete/snooze/edit; для recurring complete →
    advance `remindAt = nextOccurrence`, `completedAt` лишається null) + `DELETE`.
  - Наявні client-scoped `/clients/[id]/reminders*` лишаються (картка клієнта
    працює без змін; нові standalone-нагадування з клієнтом теж видно на картці).
- **UI** `/manager/reminders` (заміна заглушки): тулбар (пошук + «Завершені» +
  «Заказ відео» + «Створити») + список (опис, дата, бейдж періодичності, жовтий
  для активних / зелена галочка для виконаних, дії виконати/відкласти/редагувати).
- **Форма «Звичайне»**: опис, дата/час, радіо-періодичність (6) + динамічна
  підказка, чекбокс «Заказ відео», опційний пікер «Контрагент». Велика кнопка
  «Встановити нагадування».
- Sidebar-пункт «Нагадування» уже є (вказує на `/manager/reminders`).

## Етап 2 — Тип «Для товарів»

- Форма «Для товарів»: перемикач «Звичайне»/«Для товарів», пікер **Контрагент**
  (обов'язк.), додавання товарів (реюз product-search) + таблиця Товар/Кількість
  з позначкою «Виконано» по рядку.
- API: items у POST/PATCH + дія complete-per-item; коли всі рядки done → нагадування
  completed. Антидубль: не додавати той самий товар для того самого клієнта в
  активне нагадування (повідомлення user-у).

## Етап 3 — Дзвіночок + список спрацювання

- Розширити `header-notifications-bell` на **всі** прострочені нагадування user-а
  (standalone + по клієнту + авто), а не лише client-bound. Дії виконати/відкласти
  прямо з дзвіночка; діплінк (товарні → екран нагадувань, клієнтські → картка).
- Полірування списку: групування активні/прострочені/виконані.

## Етап 4 — Авто-генеровані (відео + бронь) + дії

- **Cron** `/api/cron/generate-reminders` (Windows Task, як інші): детектори по
  наших даних:
  - **Бронь минула** — лоти з простроченою бронею (`Lot.reserved*`) → авто-нагадування
    «Перенести бронь?» (`source=auto_bron`, `actionType=continue_bron`, dedup по lotId+clientId).
  - **З'явилось відео** — лот/товар отримав `videoUrl`, на який є активний інтерес
    клієнта (замовлення/бронь) → «Відправити у вайбер?» (`source=auto_video`,
    `actionType=viber_video`, dedup по productId/lotId+clientId).
- **Дії** (кнопки на нагадуванні у списку/дзвіночку): «Перенести бронь» → форма
  броні/ре-бронь; «Скинути Viber» → ShareSheet з відео-повідомленням (реюз
  наявних білдерів). Тексти — з аудиту §11.
- Потребує дослідження нашої моделі броні/відео/замовлень — воркеру дати окрему
  research-частину.
- **Відкладено**: «Закриття старих замовлень» (`actionType` третього типу) — після
  появи відповідного екрана.
