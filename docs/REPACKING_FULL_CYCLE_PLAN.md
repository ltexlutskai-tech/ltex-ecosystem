Статус: **план, чекає реалізації.** Гілка: `claude/bag-repackaging-doc-analysis-iavsyu`.
Джерело 1С: `docs/1c-export-2026-06-02/Documents/Перепаковка/` (XML метадані 2034 рядки + `Ext/ObjectModule.bsl` 976 рядків).

## 0. Простими словами (для user)

«Перепаковка» — це коли комірник бере **N старих мішків**, фізично їх розкриває, пересортовує вміст і формує **M нових мішків** (з новими штрихкодами, вагою, ціною, сектором).

- ліва таблиця «Розбір» = які старі мішки розібрали;
- права таблиця «Комплектація» = які нові мішки зібрали.

**Що вже є в нас:** документ уже існує (Документи руху товару → Перепаковка) з рядками «Розбір/Комплектація» і рухом складу для звітів. **Але** зараз це «неповна» перепаковка: нові мішки **не стають реальними лотами** на полиці, старі **не списуються** фізично, собівартість не переноситься, ваги «до/після» не звіряються.

**Рішення user (зафіксовано):**

- **повний цикл як у 1С**: провели документ → старі мішки зникли зі складу (+ знялась бронь), нові мішки з'явились як реальні лоти;
- **перенос собівартості** старих мішків на нові (пропорційно вазі);
- **контроль ваги**: попередження, якщо різниця «розібрали−зібрали» > **2 кг** (значення редаговане в налаштуваннях);
- **автогенерація штрихкодів** для нових мішків;
- старі мішки розбираються **повністю** (цілий мішок);
- доступ: **тільки склад + адмін/власник**.

| --------------------------------------- | ------------------------- | ------------------------------- |
| `Организация`                           | `CatalogRef.Организации`  | ні (один ФОП)                   |

### 1.2. Табличні частини

Дві майже однакові ТЧ. Ключова пара «звідки → куди».

**`Распаковка` (Розбір = джерело, списується):** `Номенклатура`, `КоличествоМест` (к-сть мішків), `ЕдиницаИзмерения`, `Коэффициент`, `Количество`, **`Вес`**, `ЦенаЗакупкиВес`, `Цена`, **`ХарактеристикаНоменклатуры`** (=лот/мішок), `СерияНоменклатуры`, `Качество`, `Сумма`, `ДокументРезерва` (складене посилання — під яке замовлення був резерв), **`Штрихкод`**, `Постачальник`.

**`Упаковка` (Комплектація = ціль, оприбутковується):** ті самі колонки, **без** `СерияНоменклатуры`, **плюс** `ЦенаПродажиВес` (ціна продажу €/кг) та `СекторНаСкладі` (`CatalogRef.Сектори`).

> «Мішок» = `ХарактеристикаНоменклатуры` з власними реквізитами `.Вага` (вага одного мішка) і `.ДатаПоставки`. У нас аналог = `Lot` (barcode/weight/arrivalDate/sector).

### 1.3. Логіка `ОбработкаПроведения` (BSL, рядки 744–945)

1. **Контроль ваги** (749–763): `|Σ Вес(Распаковка) − Σ Вес(Упаковка)| > Константа МаксимальнаяРазницаВесаПриПерепаковке` → **попередження** (не блокує). Контроль рівності сум — закоментований (не блокує).
2. **Списання (за `Распаковка`)**: розхід по `ТоварыНаСкладах`, ваговий регістр `ТовариНаСкладахУВазі` (кількість × `Характеристика.Вага`), зняття резерву `ТоварыВРезервеНаСкладах` (де був `ДокументРезерва`), фіксація `СписанныеТовары`. Партійний облік/`ТоварыОрганизаций` — регламентований, **нам не треба**.
3. **Оприбуткування (за `Упаковка`)**: прихід по `ТоварыНаСкладах`, ваговий `ТовариНаСкладахУВазі`, `ПартииТоваровНаСкладах` (собівартість переноситься). **Ціни** `ЦеныНоменклатуры` — пише тип `ЦенаПродажиВес` (+ за потреби `ЦенаЗакупкиВес`) для нових мішків, усереднення дублів, одиниця = кг. **Сектори** — для кожного нового мішка пише регістр `Сектори` (`ОблікСекторів.ОтриматиСектор`).
4. **`ОбработкаУдаленияПроведения`**: `УдалитьДвиженияРегистратора` (реверс усіх рухів).

### 1.4. `ПередЗаписью` та побічні дії (BSL 724–967)

- `СуммаДокумента = Распаковка.Итог("Сумма")`; автозаповнення якості за замовчуванням; перевірка одиниць.
- **`ПроставитиПостачальникаИДатуПоставки`** (955–967): для кожного нового мішка з `Упаковка` **прямо оновлює `Характеристика.ДатаПоставки = Дата`** → тобто документ модифікує довідник характеристик (у нас = ставить `Lot.arrivalDate = docDate`).
- Валідації рядків: обов'язкові `Номенклатура/Количество/Качество`; заборона послуг/наборів/комплектів (нам не актуально — лише прості товари).
- Генерації штрихкодів у `ObjectModule` **немає** (вводяться у формі/сканером). Обмеження «лише сьогодні» — **немає**.

### 1.5. Пов'язані сутності 1С

Довідники: `Номенклатура`, `ХарактеристикиНоменклатуры` (мішок), `Качество`, `Сектори`, `Контрагенты` (постачальник), `ТипыЦенНоменклатуры` (`ЦенаПродажиВес`/`ЦенаЗакупкиВес`). Регістри: `ТоварыНаСкладах`, `ТовариНаСкладахУВазі`, `ТоварыВРезервеНаСкладах`, `ПартииТоваровНаСкладах`, `СписанныеТовары`, `Сектори`, `ЦеныНоменклатуры`. Константа `МаксимальнаяРазницаВесаПриПерепаковке`.

## 2. Наш поточний стан (аудит)

- **Модель + UI + рух складу вже існують** як `kind="repackings"` в узагальненій системі `stock-documents`:
  - `Repacking` + `RepackingItem` (schema.prisma) — `RepackingItem.role` = `"disassembled"` (розбір) / `"assembled"` (комплектація); поля `productId, charHex, barcode, weight, quantity, priceEur, amountEur, notes`.
  - Шапка `Repacking`: `docNumber (LT-RPK-...), docDate, warehouseId, status(draft/posted/cancelled), inputWeight, outputWeight, lossWeight, notes, postedAt/By`.
  - UI: `manager/(workstation)/stock-documents/[kind]/{page,new,[id]}` + `_components/stock-doc-form.tsx` (рендерить рядки з селектом ролі Розбір/Комплектація).
  - API: `api/v1/manager/stock-documents/[kind]/{route,[id]/route,[id]/post/route}`.
  - Проведення: `lib/manager/stock-documents-repo.ts::postStockDoc` → `lib/manager/stock-movement-hooks.ts::applyStockDocumentMovements` — пише **лише `StockMovement`** (`assembled`→прихід `recordKind=0`, `disassembled`→розхід `recordKind=1`), best-effort, `upsert` за `(recorderCode1C=docId, lineNo, productCode1C)`. Реверс `removeStockDocumentMovements`.
  - Історичний імпорт: `import-1c-historical.ts::importRepack` (Распаковка `_VT6685`→disassembled, Упаковка `_VT6702`→assembled, upsert за `code1C`). 3766 записів на проді.
- **Чого бракує проти 1С (= обсяг цієї роботи):**
  1. нові мішки (`assembled`) **не створюються** як реальні `Lot` (нема нових штрихкодів/лотів на складі);
  2. старі мішки (`disassembled`) **не списуються** фізично (status/quantity/бронь не чіпаються);
  3. **собівартість не переноситься** (`CostMovement` не пишеться);
  4. **контроль ваги** до/після відсутній;
  5. `arrivalDate`/`priceEur`/`sector` нових лотів не виставляються;
  6. автогенерація штрихкодів для нових мішків відсутня.
- Наявне: `WarehouseSector` (довідник), генератор ШК у блоці Поступлення (`L-{articleCode}-{seq:05}`), `CostMovement`/`StockMovement` та звіти Залишки/Маржа, що читають ці регістри.

## 3. Пропонована реалізація (розширення наявного `repackings`)

> Не робимо новий документ з нуля — **розширюємо** наявний `kind="repackings"`. Це мінімізує ризик і зберігає історичний імпорт.

### 3.1. Зміни схеми — 1 міграція `20260713_repack_full_cycle`

Розширити `RepackingItem` (усі поля additive/nullable):

```prisma
// РОЗБІР (disassembled):
sourceLotId      String?  @map("source_lot_id")      // який лот розібрали
sourcePrevStatus String?  @map("source_prev_status") // для реверсу (було 'free')
// КОМПЛЕКТАЦІЯ (assembled):
createdLotId     String?  @map("created_lot_id")      // який лот створили при проведенні
salePriceEur     Decimal? @map("sale_price_eur") @db.Decimal(12,2)  // ЦінаПродажуВес €/кг
qualityId        String?  @map("quality_id")          // якість (сорт)
sector           String?                               // сектор нового мішка (текст)
sectorId         String?  @map("sector_id")            // FK → WarehouseSector
costPerKgEur     Decimal? @map("cost_per_kg_eur") @db.Decimal(12,2) // рознесена собівартість
```

Довідник допуску ваги: додати key-value-налаштування (нова легка модель `MgrSetting { key @id, value, updatedAt }` **або** перевикористати наявний key-value патерн `MgrSyncState`). Ключ `repack.weight_tolerance_kg` (дефолт `2`). Значення редаговане на сторінці Налаштувань.

Опційно `WarehouseSector.code String? @unique` (find-or-create за назвою) — спільно з планом «Зміна стану мішка».

### 3.2. Логіка проведення — розширити хук проведення `repackings`

Через окремий хук `lib/manager/repack-full-cycle.ts::applyRepackFullCycle(repackId)` (викликається у `postStockDoc` для `kind==="repackings"` поряд з наявним `applyStockDocumentMovements`), у `$transaction`, ідемпотентно:

1. **Контроль ваги**: `inW = Σ disassembled.weight`, `outW = Σ assembled.weight`. Якщо `|inW − outW| > tolerance(2 кг)` → повернути прапорець попередження (UI показує портальний діалог «Все одно провести / Скасувати», як контроль відхилення ціни в реалізації). НЕ блокує.
2. **Списання джерела** (для кожного `disassembled`): знайти `Lot` за `barcode`/`sourceLotId` → зберегти `sourcePrevStatus` → `status = "repacked_out"`, `quantity = 0`, зняти бронь (`reserved* = null`). (StockMovement розхід — уже пишеться наявним хуком.)
3. **Пул собівартості**: `totalCost = Σ (source.purchasePriceEur × source.weight)` [€], `totalOutW = Σ assembled.weight`. `costPerKg = totalOutW > 0 ? totalCost / totalOutW : 0`.
4. **Створення нових мішків** (для кожного `assembled`): `Lot.create`:
   - `barcode`: з рядка (сканер) **або** автоген `L-{articleCode}-{seq:05}` (перевикористати генератор Поступлення); гарант унікальності.
   - `productId`, `weight`, `quantity = 1`, `status = "free"`, `priceEur = salePriceEur` (ЦінаПродажуВес), `purchasePriceEur = costPerKg`, `arrivalDate = docDate`, `sector`/`sectorId` (find-or-create), якість.
   - записати `createdLotId` + `costPerKgEur` у рядок.
   - **CostMovement**: писати рух собівартості (`recorderCode1C = repack.code1C ?? repack.id`, `lineNo`, `productCode1C/productId`, `qty`, `costEur = costPerKg × weight`, `occurredAt = docDate`). (StockMovement прихід — уже пишеться.)
5. `arrivalDate` нових лотів = `docDate` (дзеркало `ПроставитиПостачальникаИДатуПоставки`).
6. `outputWeight`/`inputWeight`/`lossWeight = inW − outW` у шапку.

**Реверс `removeRepackFullCycle`** (при `cancel`/розпроведенні): видалити створені лоти (`createdLotId`), відновити джерельні (`status = sourcePrevStatus ?? "free"`, `quantity = 1`), прибрати `CostMovement` за реєстратором. (StockMovement прибирає наявний `removeStockDocumentMovements`.)

> Собівартість/залишки/маржа з'являться у звітах **автоматично**, бо звіти вже читають `StockMovement`/`CostMovement`.

### 3.3. UI (розширити `stock-doc-form.tsx` для repackings)

- Рядок `disassembled`: сканер ШК → підтягує лот (артикул/вага/собівартість) як джерело.
- Рядок `assembled`: артикул/вага/якість/**ЦінаПродажуВес**/**сектор** (селект `WarehouseSector`) + ШК (сканер **або** «Згенерувати ШК» на рядок).
- Панель зведення: `Вага розбору` / `Вага комплектації` / `Різниця` (підсвітка червоним якщо > 2 кг) / `Собівартість джерела` / `Собівартість €/кг` на нові мішки.
- «Зберегти та провести» → якщо різниця ваги > 2 кг → портальний діалог «Все одно провести».
- Друк етикеток нових мішків (перевикористати наявний друк ШК).

### 3.4. Права

Створення/проведення `repackings`: обмежити до `warehouse`, `admin`, `owner` (звузити наявний доступ саме для цього kind, або лишити спільний список і додати guard у `[id]/post`). Інші — перегляд.

### 3.5. Тести

- Чисте ядро `buildRepackApply`: пул собівартості, `costPerKg`, розподіл на нові лоти, `inW/outW/loss`, прапорець допуску ваги (2 кг, з налаштування).
- Проведення: створення лотів (з автоген-ШК і зі сканером), списання джерела + зняття броні, `CostMovement`.
- Реверс: видалення нових лотів + відновлення джерела + прибирання `CostMovement`.
- Регрес: наявний `applyStockDocumentMovements` для repackings не зламано; історичний імпорт `importRepack` сумісний (нові поля nullable).

## 5. Свідомо НЕ робимо (як і в реалізації менеджерки)

- Бухгалтерський/податковий облік, `ТоварыОрганизаций`, повний партійний облік — немає живого 1С, лише **управлінський** облік (залишки/собівартість/вага).
- `ДокументРезерва` (поліморфне посилання на замовлення-підставу резерву) — знімаємо бронь лота напряму (`reserved* = null`), без окремого регістру резервів.
- `ЦеныНоменклатуры` на рівні товару — ціну тримаємо на лоті (`Lot.priceEur`); окремий регістр цін не пишемо.
