# Резервні копії та відновлення (ТЗ 8.0 Блок D)

**Навіщо це важливо саме зараз.** Після повної відмови від 1С наша PostgreSQL стає **єдиним джерелом правди**. Раніше 1С фактично була «другою копією» усіх даних; тепер її не буде. Тому надійність резервного копіювання — критична.

> ⚠️ Поки компанія ще працює в 1С — це «страхувальна сітка». Після фінального переносу даних і вимкнення 1С цих рекомендацій треба дотриматися обовʼязково.

## Поточний стан

- Щоденний дамп: `pg_dump -Fc` о **03:00** → `E:\ltex-backups\`.
- Зберігання (retention): **14 днів**.
- Крон-задачі стартують о 03:30 (після дампу) — див. `docs/CRON_SETUP.md`.
- Supabase (Frankfurt) — «холодне» дзеркало БД (не активне).

## Рекомендації (пріоритет після відмови від 1С)

### 1. Збільшити глибину зберігання

14 днів мало: якщо помилку в даних помітять пізніше (напр. через місяць), відновити не буде з чого.

- Щоденні копії: тримати **60 днів**.
- Щотижнева копія (неділя): тримати **6 місяців**.
- Щомісячна копія (1-е число): тримати **2 роки**.

Приклад ротації (PowerShell, ідея — окремі підпапки daily/weekly/monthly + видалення старих за віком файлу):

```
# daily → E:\ltex-backups\daily\  (видаляти > 60 днів)
# weekly → E:\ltex-backups\weekly\ (неділя; видаляти > 180 днів)
# monthly → E:\ltex-backups\monthly\ (1-е число; видаляти > 730 днів)
```

### 2. Копія поза сервером (offsite) — найважливіше

Зараз і база, і бекапи — на диску `E:`. Втрата диска/сервера знищить і те, й інше.

- Копіювати щоденний дамп на **інший фізичний носій** (окремий диск/NAS) та/або у **хмару** (напр. зашифрований bucket).
- Мінімум — інший фізичний диск у тому ж сервері; краще — інша будівля/хмара.
- Правило «3-2-1»: 3 копії, 2 різні носії, 1 поза локацією.

### 3. Регулярний тест відновлення

Бекап, який жодного разу не відновлювали, — це не бекап.

- Раз на місяць: відновити останній дамп у **тестову** базу (`pg_restore` у окрему БД) і переконатися, що застосунок піднімається й дані на місці.
- Задокументувати процедуру відновлення (нижче) і час відновлення (RTO).

### Процедура відновлення (шпаргалка)

```
# Відновлення повного дампу у НОВУ базу (не перезаписуючи робочу):
& "E:\PostgreSQL\16\bin\createdb.exe" -U ltex ltex_restore_test
& "E:\PostgreSQL\16\bin\pg_restore.exe" -U ltex -d ltex_restore_test --no-owner "E:\ltex-backups\daily\<файл>.dump"
```

## Захист даних усередині застосунку (уже зроблено / плановане)

- **Позначка на вилучення (ТЗ 8.0 Блок B)** — користувач не може стерти дані напряму; лише адміністратор після перевірки звʼязків. Клієнт/документ з боргом чи історією фізично не видаляється — лише архів. Це головний захист від випадкової втрати.
- **Регістр боргу самовідновлюваний** — `MgrClient.debt` — кеш; джерело правди — рухи `MgrDebtMovement`. За потреби борг перераховується скриптом `scripts/recompute-client-debt.ts`.
- **Журнал аудиту** (`audit_logs`) — append-only (лише додавання в коді). Тепер фіксує і позначки/видалення через чергу.

### (D2, опційно, низький пріоритет) Незмінність аудиту на рівні БД

Зараз `audit_logs` та таймлайн append-only лише на рівні коду. Для юридичної незаперечності можна додати тригер PostgreSQL, що забороняє `UPDATE`/`DELETE` у `audit_logs`:

```sql
CREATE OR REPLACE FUNCTION forbid_audit_mutation() RETURNS trigger AS $$
BEGIN RAISE EXCEPTION 'audit_logs is append-only'; END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER audit_no_update BEFORE UPDATE OR DELETE ON audit_logs
  FOR EACH ROW EXECUTE FUNCTION forbid_audit_mutation();
```

Впроваджувати лише за потреби (ускладнює обслуговування).

## Чек-лист для user (коли вимикатимемо 1С)

- [ ] Retention щоденних копій збільшено до 60 днів.
- [ ] Додано щотижневу + щомісячну копії з довшим зберіганням.
- [ ] Налаштовано offsite-копію (інший диск / хмара).
- [ ] Проведено перший успішний тест відновлення у тестову базу.
- [ ] Процедуру відновлення задокументовано й перевірено час відновлення.
