## Фаза A — виправити 3 видимі баги (старт 5.3)

### A1. Дати документів = today

**Симптом:** усі замовлення/реалізації/маршрути показують `10.06.2026`
(сьогодні), замість 2021-2026.

**Корінь:** 1С зберігає дати з `_YearOffset` (offset до базового року, типово
2000). `mssql`-драйвер віддає дату як є (без offset). У скрипті
`asDate()` фільтрує `year < 1980` → null → Order.createdAt лишається
дефолтним `now()`.

**Фікс:**

1. Прочитати один раз значення offset з 1С: `SELECT * FROM _YearOffset`
   (таблиця у `docs/1c-mssql-schema/config_tables.tsv`).
2. У `asDate()` — додати offset роки до `Date` яке повернув mssql.
3. Зняти фільтр `< 1980` (стане не потрібен).
4. Реімпорт: `--entity orders,sales,cashorders,routesheets,prices`
   (довідники без дат — не чіпаємо). UPDATE-only через upsert по
   `code1C=hex`.

### A2. Замовлення/реалізації показують 0₴ у списку

**Симптом:** у `/manager/sales` і `/manager/orders` колонка «Сума» = 0₴
(або просто не відображається). У формі — позиції з цінами є.

**Корінь:** скрипт пише items, але **не обчислює** `Order.totalEur`,
`Order.totalUah`, `Sale.totalEur`, `Sale.totalUah`. Список читає поле,
форма складає з items на льоту.

**Фікс:** у `importOrders` / `importSales` після створення items:

```ts
const totalEur = items.reduce((s, i) => s + Number(i.priceEur), 0);
const totalUah = totalEur * (rateEur ?? defaultRate);
// далі prisma.order.update({ where:{id}, data:{ totalEur, totalUah } })
```

Або порахувати ДО create в одному upsert.

### A3. Маршрутні листи: порожні всі вкладки

**Симптом:** шапка є (1001 рядок), але Заказы/Товари/Загрузка/
Реалізації/Продажі/Оплати/Завдання — `(0)` скрізь.

**Корінь:** свідомо застаблено в 5.2. `HISTORY_MIGRATION_MAP §10.2` НЕ
декодував поіменно VT-таблиці маршрутного.

**Фікс план:**

1. Декодувати кожну VT (їх ~10) — через `dbnames.txt` знайти UUID кожної
   VT з `Documents/МаршрутныйЛист.xml`, потім поля кожної VT через
   `_Document6630_VT<N>` колонки.
2. Дописати у `HISTORY_MIGRATION_MAP §10.2` повну розкладку.
3. У скрипті розширити `importRouteSheets` — для кожного маршрутного
   читати VT і писати дочірні моделі (`RouteSheetOrder`, `RouteSheetSale`,
   `RouteSheetPayment`, etc — див. `packages/db/prisma/schema.prisma`).

## Фаза B — повний аудит паритетності (після A)

Для **кожного з 6 блоків** запустити research-агента (паралельно), який
звірить:

1. **Замовлення** (`Document.ЗаказПокупателя`)
2. **Реалізація** (`Document.РеализацияТоваровУслуг`)
3. **Каса** (`Document.ПриходныйКассовыйОрдер` + `РасходныйКассовыйОрдер`)
4. **Маршрутний лист** (`Document.МаршрутныйЛист`)
5. **Картка клієнта** (`Catalog.Контрагенты`)
6. **Прайс** (`Catalog.Номенклатура` + `ХарактеристикиНоменклатуры` + `_InfoRg5225`)

Кожен агент читає:

- 1С форму: `docs/1c-export-2026-06-02/<Catalogs|Documents>/<X>/Forms/*`
- 1С бізнес-логіку: `docs/1c-export-2026-06-02/CommonModules/ОбменАРМ/Ext/Module.bsl`
  (7635 LOC) + Module.bsl самого об'єкта
- Наша поточна реалізація: `apps/store/app/manager/<block>/`
  - `apps/store/lib/manager/<block>*.ts`

Результат — `docs/PARITY_AUDIT_<block>.md` зі списком:

- ✓ є в обох
- ⚠️ є в 1С, нема в нас (gap — пріоритизувати)
- ⚠️ є в нас, нема в 1С (поліпшення — лишити)

Після всіх 6 — спільний `docs/PARITY_SUMMARY.md` з пріоритетною чергою,
читаєте і затверджуєте порядок реалізації.
