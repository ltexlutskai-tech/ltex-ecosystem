## 0. Передумови та мета

Бізнес **планує повністю відмовитися від 1С**. Жодних _живих обмінів_ (SOAP / pull / push / черги вивантаження) більше не буде — їх уже знесено. Водночас треба **максимально зберегти функціонал**: історичні дані, реєстри руху (борг, продажі, склад), звʼязки клієнт↔замовлення↔продаж.

> 🛑 **СТАТУС ПЕРЕХОДУ (уточнено user-ом 2026-07-09).** Компанія **ЩЕ ПРАЦЮЄ на 100% у 1С**. Перехід на нашу систему = ще один **повний перенос усіх даних з 1С**, який попереду. Тому **вся інфраструктура офлайн-імпорту та дампи 1С мають лишитися недоторканими** до завершення фінального переносу: `apps/store/scripts/import-1c-historical.ts` (+похідні), `scripts/import-1c.ps1`, `scripts/dump-1c-*.ps1`, `docs/1c-export*`, `docs/1c-mssql-schema/`, `docs/HISTORY_MIGRATION_MAP.md`, `docs/1C_MSSQL_CODES.md`, `docs/1C_REGISTRY_INVENTORY.md`. Прибирати їх — **лише після** успішного фінального cutover, окремим рішенням. Чистимо зараз ТІЛЬКИ **мертву живу-синхронізацію** (черги обміну, орфан-код, оманливі UI-написи), яка не стосується імпорту.

Дві великі цілі цього ТЗ:

1. **Вичистити все, що завʼязане на обміни з 1С** (мертвий код, таблиці черг, UI-індикатори «синхронізовано», BSL-файли), **не зламавши** реєстри та історію.
2. **Впровадити «позначку на вилучення» у стилі 1С**: звичайний користувач може лише **позначити** обʼєкт на вилучення (він зникає з його списків), а **фізично видаляє тільки адміністратор**, який отримує ці позначки як **чергу завдань**. Перед позначенням користувач мусить вказати **причину**, щоб адміністратор розумів, навіщо видаляти.

> ⚠️ **Критичний інваріант.** Поля з назвами `code1C`, `uid1C`, `number1C`, `parentCode1C`, `*Code1C` (у movement-таблицях) **НЕ видаляти й не перейменовувати**. Попри назву «1C», вони більше не про обмін — це **внутрішні стабільні ключі**, на яких тримаються реєстри руху, розрахунок боргу та звʼязок клієнт↔документ. Їх видалення зламає борг і історію. Перейменування — поза обсягом цього ТЗ (окремий великий рефактор у майбутньому, за бажанням).

## БЛОК B — «Позначка на вилучення» (soft-delete у стилі 1С) 🎯 ядро ТЗ

**Навіщо / як у 1С:** у 1С звичайний користувач ставить об'єкту **позначку на вилучення** — об'єкт лишається в базі, але перекреслений і зникає зі звичайних списків. Фізично стирає лише той, хто має право, окремою операцією «Видалення позначених об'єктів». Ми відтворюємо цю модель 1:1.

### B1. Загальна логіка (застосовується до клієнтів, документів, довідників)

**Ролі:**

- **Звичайний користувач** (manager / senior_manager / warehouse / analyst / …): може **лише позначити** на вилучення обʼєкти, до яких має доступ (свої). НЕ може стирати фізично.
- **Адміністратор** (`admin` / `owner`): бачить **чергу завдань на вилучення**, і по кожному приймає рішення: **Видалити** (фізично, з усіма реверсами) або **Відхилити** (зняти позначку, обʼєкт повертається користувачу).

**Сценарій користувача (обовʼязковий):**

1. Користувач тисне «Видалити» на клієнті / документі / записі довідника.
2. Зʼявляється діалог: **«Справді позначити на вилучення? Вкажіть причину:»** + обовʼязкове текстове поле (мін. 3 символи).
3. Після підтвердження: обʼєкту ставиться позначка → він **зникає зі списків користувача** → створюється **завдання адміністратору** з причиною.
4. Користувач бачить тост: «Позначено на вилучення. Остаточне рішення прийме адміністратор».

**Сценарій адміністратора:**

1. Черга `/manager/admin/deletions` показує всі позначки: **що** (тип + назва/номер), **хто** позначив, **коли**, **причина**.
2. По кожному запису — дії:
   - **Видалити остаточно** → викликає наявну фізичну логіку видалення (з реверсом боргу для документів; з перевірками захисту нижче).
   - **Відхилити** (з опційним коментарем) → знімає позначку, обʼєкт повертається у списки користувача, автор бачить це у своєму таймлайні/сповіщенні.
3. Адмін також бачить позначені обʼєкти у звичайних списках із бейджем «⚠ на вилученні» (щоб міг зайти й перевірити).

### B2. Зміни схеми БД

**Нова універсальна модель черги** (полиморфна — одна на всі типи обʼєктів):

```prisma
model DeletionRequest {
  id                String    @id @default(cuid())
  entityType        String    // "client" | "order" | "sale" | "cash_order" | "route_sheet" | "dictionary" | "category" | "product"
  entityId          String
  entityLabel       String    // знімок людської назви/номера для черги (напр. "Замовлення L0000002477" або "ТОВ Ромашка")
  dictType          String?   // лише для entityType="dictionary": який довідник (client-statuses / routes / …)
  reason            String    @db.Text            // причина від користувача (обовʼязкова)
  status            String    @default("pending") // pending | approved | rejected
  requestedByUserId String
  requestedByName   String?                       // знімок імені автора
  requestedAt       DateTime  @default(now())
  resolvedByUserId  String?
  resolvedAt        DateTime?
  resolutionNote    String?   @db.Text            // коментар адміна при відхиленні/видаленні

  @@index([status, requestedAt])
  @@index([entityType, entityId])
  @@map("deletion_requests")
}
```

**Прапорець-позначка на самих обʼєктах** (щоб фільтрувати списки без джойну на чергу). Додати **однакове** поле кожному верхньорівневому обʼєкту:

```prisma
markedForDeletion Boolean @default(false) @map("marked_for_deletion")
```

Куди додати:

- `MgrClient` (зараз позначки немає взагалі — див. B5).
- `Order`, `Sale`, `MgrCashOrder`, `RouteSheet` (уже мають `archived`/`isActual`; це **інша** ознака — саме «на вилученні»).
- Прості довідники: `MgrClientStatus`, `MgrSearchChannel`, `MgrCategoryTT`, `MgrDeliveryMethod`, `MgrRoute`, `MgrProducer`.
  > Для довідників Фази 8 (`Unit`/`Region`/`Cityy`/…), які вже мають `archived` — позначку теж додати для однорідності, але можна fallback-нути на `archived`, якщо не хочемо роздувати схему (рішення виконавця; головне — однаковий фільтр у списках).

Міграція: `20260710_soft_delete_marks` (усі `ADD COLUMN … marked_for_deletion boolean NOT NULL DEFAULT false` + `CREATE TABLE deletion_requests`).

**Опційно для клієнта** — `archived Boolean @default(false)` (див. B5, для результату «залишити, але сховати назавжди»).

### B3. Уніфікувати перевірку «хто адміністратор»

**Навіщо:** зараз перевірки розкидані (`ADMIN_ROLES`, `requireRole(["admin","owner"])`, `role==="admin"`). Черга видалень вимагає єдиного визначення.

- Додати у `apps/store/lib/auth/manager-auth.ts`:
  - `isAdminRole(role): boolean` → `role === "admin" || role === "owner"` (перевикористати `ADMIN_ROLES` з `jwt.ts`).
  - `requireAdmin(req)` → обгортка над `requireRole(["admin","owner"])`.
- Замінити розкидані перевірки в admin-роутах на `requireAdmin` (поступово, без зміни поведінки).

### B4. Спільний бекенд позначення / вирішення

- Новий сервер-модуль `apps/store/lib/manager/deletion-queue.ts`:
  - `markForDeletion({ entityType, entityId, reason, user })` — транзакційно: (1) валідація доступу користувача до обʼєкта (перевикористати `canEditClient`/`canViewOrder`/`canViewSale`/`canDeleteManagerDoc`); (2) `<model>.update({ markedForDeletion: true })`; (3) `deletionRequest.create({... status:"pending"})`; (4) `logAuditEvent(action:"update", summary:"позначено на вилучення")`; (5) для клієнта — запис у таймлайн (`kind:"comment"`). Ідемпотентність: якщо вже є `pending`-запит на цей обʼєкт — не дублювати.
  - `rejectDeletion({ requestId, note, admin })` — зняти `markedForDeletion=false`, `status="rejected"`, сповістити автора (нагадування/таймлайн).
  - `approveDeletion({ requestId, admin })` — виконати **фактичне** видалення (див. B5–B7), `status="approved"`, `logAuditEvent(action:"delete")`.
- Нові API:
  - `POST /api/v1/manager/deletions` — створити позначку (будь-який автентифікований менеджер, з перевіркою доступу до обʼєкта).
  - `GET /api/v1/manager/deletions?status=pending` — черга (лише `requireAdmin`).
  - `GET /api/v1/manager/deletions/[id]/references` — попередній перегляд звʼязків обʼєкта (`requireAdmin`); показує адміну, чи можна стерти, чи лише архів, ДО натискання «Видалити».
  - `POST /api/v1/manager/deletions/[id]/approve` — виконати рішення (`requireAdmin`): фіз. видалення (якщо 0 посилань і не-1С) або авто-архів (див. B4a).
  - `POST /api/v1/manager/deletions/[id]/reject` — відхилення, зняти позначку (`requireAdmin`).
  - `POST /api/v1/manager/archive/[entityType]/[id]/restore` — відновити з архіву (`requireAdmin`): `archived=false`.

### B4a. Контроль посилань перед остаточним видаленням (1С «Видалення позначених обʼєктів») 🔒

**Навіщо (уточнення user 2026-07-09):** потрібен механізм **остаточного** видалення справді непотрібної інформації — але лише **після ретельної перевірки всіх звʼязків**. Якщо на обʼєкт бодай щось посилається — фізично видалити НЕ можна, лише перенести в архів. Точно як у 1С: команда «Видалення позначених обʼєктів» спершу шукає всі посилання, і захищає ті, на які є посилання.

**Спільна функція** `apps/store/lib/manager/reference-check.ts::findReferences(entityType, entityId)`:

- Повертає масив блокуючих посилань: `[{ table, count, sampleLabel }]` — де і скільки разів згадується обʼєкт.
- Перевіряє **усі** можливі звʼязки для типу обʼєкта (нижче — мапа перевірок).
- Плюс правило: обʼєкт з `code1C != null` (історичний, з 1С) вважається таким, що **має історичну цінність** → навіть без явних FK-посилань фіз. видалення заборонене (лише архів), щоб не осиротити реєстри руху (`*Code1C`).

**Мапа перевірок звʼязків по типах:**

| Тип обʼєкта                        | Що перевіряти (якщо є хоч одне — тільки архів)                                                                                                                                                                                                            |
| ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Клієнт** (`MgrClient`)           | `MgrDebtMovement` (рухи боргу); `Order`/`Sale`/`MgrCashOrder` за спільним `code1C` або `phonePrimary`; рухи реєстрів з `clientCode1C` (`MgrSalesMovement`, `MgrCashFlowMovement`); непорожні дочірні (таймлайн з подіями, презентації); `code1C != null`. |
| **Замовлення** (`Order`)           | `Sale.orderId` (є реалізації-нащадки); `MgrReminder.orderId`; `OrderRemainderMovement` за `orderCode1C`; `code1C != null` (проведене в 1С).                                                                                                               |
| **Реалізація** (`Sale`)            | `MgrCashOrder.saleId` (є оплати); `MgrDebtMovement`/`MgrSalesMovement`/`MgrCostMovement` за `saleCode1C`/`recorderCode1C`; `code1C != null`.                                                                                                              |
| **Оплата/каса** (`MgrCashOrder`)   | парний ордер-здача (`changeForId`); `MgrDebtMovement`/`MgrCashFlowMovement` за `sourceId`/`recorderCode1C`; `code1C != null`.                                                                                                                             |
| **Маршрутний лист** (`RouteSheet`) | привʼязані `Sale`/`MgrCashOrder`/`Order` (плоскі `routeSheetId`); `code1C != null`.                                                                                                                                                                       |
| **Запис довідника**                | будь-який FK з бізнес-моделей (напр. `MgrClient.statusGeneralId` → на статус); `code1C != null`.                                                                                                                                                          |
| **Товар/категорія**                | `Lot`/`OrderItem`/`SaleItem`/`ReceivingItem`/`CartItem`; для категорії — дочірні категорії + товари; `code1C != null`.                                                                                                                                    |

**Рішення `approveDeletion` (адмін у черзі):**

1. Викликати `findReferences(...)`.
2. **Є посилання АБО `code1C != null`** → фізичне видалення **ЗАБОРОНЕНО**. Дія автоматично стає **«Перенести в архів»** (`archived=true`, `markedForDeletion=false`). Адміну показати список блокуючих посилань: «Не можна видалити: використовується у — Реалізації (3), Рухи боргу (12). Буде перенесено в архів (дані збережуться)».
3. **Посилань немає І `code1C == null`** → показати фінальне попередження «Це остаточне видалення. Скасувати неможливо» → після підтвердження виконати наявну фізичну DELETE-транзакцію (з реверсом боргу де треба) + `logAuditEvent(action:"delete")`.

> Тобто «остаточно стерти» доступне **лише** для обʼєкта, який (а) не має жодних посилань і (б) не є історичним записом 1С. Усе інше — архів. Це і є поведінка 1С.

**Архів vs позначка на вилучення — різниця (важливо):**

- `markedForDeletion` — тимчасова позначка «кандидат на видалення», чекає рішення адміна; можна зняти (Відхилити).
- `archived` — остаточне рішення «лишити в базі, але приховати назавжди», коли стерти не можна через звʼязки. Обʼєкт зникає з усіх робочих списків, але доступний у окремому фільтрі «Архів» (адмін/owner) і повністю зберігає дані та історію. З архіву можна **відновити** (`archived=false`) — на випадок помилки.

### B5. Клієнти (MgrClient) — заміна хака `hideTrash`

**Поточний стан:** DELETE клієнта немає; «прибирання» — брудний хак через numeric-префікс імені (`TRASH_NAME_PREFIXES` у `load-clients.ts` та `clients/route.ts`).

**Зробити:**

1. Кнопка «Видалити клієнта» у картці (`ClientHeader` / `ClientActionButtons`) → діалог з причиною → `POST /deletions {entityType:"client"}`. Доступ: `canEditClient` (свій клієнт або admin).
2. Списки клієнтів (`load-clients.ts` + `clients/route.ts`) за замовчуванням **виключають** `markedForDeletion=true` для не-адмінів. **Видалити `TRASH_NAME_PREFIXES`** і всю логіку `hideTrash` (обидва місця) — її замінює позначка. Дадуть перехідну міграцію даних: клієнтів з іменами-префіксами `1111…9999` разово позначити `markedForDeletion=true` (окремий одноразовий скрипт `scripts/migrate-trash-clients.ts`, dry-run за замовчуванням).
3. **Фактичне видалення клієнта адміном — з ЗАХИСТОМ ЦІЛІСНОСТІ** (критично, бо `MgrDebtMovement.clientId` та всі дочірні — `onDelete: Cascade`, тобто фіз. видалення зітре борг-регістр і таймлайн):
   - Якщо у клієнта є **будь-які рухи боргу** (`MgrDebtMovement`) АБО повʼязані документи (`Order`/`Sale`/`MgrCashOrder` за спільним `code1C`/телефоном) → **фізичне видалення ЗАБОРОНЕНО**. Замість нього адмін-дія = **архівування**: `archived=true` + `markedForDeletion=false`, обʼєкт зникає з усіх списків назавжди, але дані й реєстри збережені. У черзі показати підказку: «Клієнт має історію — буде заархівовано (дані збережуться), не стерто».
   - Фізично стерти можна лише «порожнього» клієнта (0 рухів, 0 документів) — тоді `mgrClient.delete()` (каскад безпечний).

> Це і є 1С-логіка: у 1С об'єкт з рухами теж не стирається, поки не почищено рухи.

### B6. Документи (Order / Sale / MgrCashOrder / RouteSheet)

**Поточний стан:** hard-DELETE вже є (`doc-delete-permission.ts` + транзакції з реверсом боргу + `recomputeDebtForClients`). Хук UI — `use-doc-delete.tsx`.

**Зробити:**

1. Перевести кнопку «Видалити» у таблицях документів (`sales-table`, `orders-table`, `payments-table`, `route-sheets-table`) з прямого DELETE на **позначку** (`POST /deletions`) для не-адмінів. Змінити `use-doc-delete.tsx`: діалог тепер вимагає **причину**; за замовчуванням шле `POST /deletions`. Для адміна лишити опцію «видалити одразу» (обходить чергу) — на розсуд, але простіше: **усі йдуть через позначку**, адмін підтверджує у черзі (однорідність).
2. Позначені документи зникають зі списків не-адмінів (додати `markedForDeletion:false` у `buildOrdersWhere` та аналоги для sales/payments/routes).
3. `approveDeletion` для документів викликає **наявні** DELETE-транзакції (реверс боргу вже написаний — `mgrDebtMovement.deleteMany` + `recomputeDebtForClients`). **Додати `logAuditEvent(action:"delete")`** у ці транзакції (зараз аудит видалень не пишеться!).
4. **Захист:** документ з `code1C != null` (історичний, проведений у 1С) — при `approve` показати сильне попередження; за замовчуванням для таких пропонувати **архів** (`archived=true`), а не фіз. стирання, щоб не осиротити рухи реєстрів (продажі/склад/кошти тримаються на `recorderCode1C`).

### B7. Довідники

**Поточний стан:** прості довідники (`simple-dict-actions.ts::deleteDictEntry`) — hard-delete, дозволено `{owner, admin}`, захисту «1С не видаляти» немає (лише FK `try/catch`). Довідники Фази 8 — soft через `archived` у PATCH.

**Зробити:**

1. Для **не-адміна** (якщо колись отримає право на довідники) — маршрут через позначку. Наразі запис у довідники — лише owner/admin, тож практично: позначку тут можна пропустити, але **додати захист «1С не видаляти»**: якщо `code1C != null` — фіз. видалення заборонено, лише `markedForDeletion`/`archived` (аналог логіки `delete-non-1c-products.ts::deleteEmptyNon1CCategories`, де 1С-дерево захищене).
2. `deleteDictEntry`: перед `delete()` перевіряти `code1C` і використання (FK). Якщо використовується або має `code1C` → м'яко позначити (`markedForDeletion`/`archived`), а не стирати. Це прибирає незрозумілу помилку «можливо, значення вже використовується».
3. Позначені/архівні записи довідника **не показувати у випадаючих списках** вибору (дропдауни статусів/каналів/маршрутів у картці клієнта), але лишати у вже збережених документах (не ламати історію).

### B8. Черга завдань адміністратора (UI)

- Нова сторінка `/manager/admin/deletions` (гейт `requireAdmin`):
  - Таблиця: Тип • Обʼєкт (клікабельний лінк на картку) • Хто позначив • Коли • Причина • [Видалити остаточно] [Відхилити].
  - Фільтри: тип обʼєкта, статус (Очікують / Оброблені), автор.
  - Лічильник-бейдж «Завдання на вилучення (N)» у лівому меню менеджерки (полінг як у дзвіночка нагадувань).
- Опційно — дублювати сповіщення адміну через наявний механізм `MgrReminder` (створювати нагадування `ownerUserId=<кожен admin>` при новій позначці). Якщо робити — додати значення `MgrReminderSource.deletion_request`. **Рекомендація:** для MVP достатньо бейджа + сторінки черги, без нагадувань.

### B9. Що НЕ входить у soft-delete (лишити як є)

- Дрібні дочірні записи, що редагуються всередині картки: телефони, месенджери, маршрути-призначення, рядки таймлайну клієнта (`clients/[id]/phones|messengers|routes|timeline/[..]` DELETE) — це редагування картки її власником, а не «видалення обʼєкта». Лишити прямий DELETE (власник картки).
- Технічні DELETE: `me/sessions/[id]`, `me/view-prefs/[..]`, `message-templates/[id]` — не бізнес-обʼєкти.

### B10. Тести Блоку B (vitest)

- `reference-check.test.ts`: `findReferences` знаходить рухи боргу/документи/реєстри; обʼєкт з `code1C != null` завжди «має посилання» (архів); порожній не-1С обʼєкт → 0 посилань (можна стерти).
- `deletion-queue.test.ts`: mark створює request+прапорець+аудит; повторний mark не дублює; reject знімає прапорець; approve обʼєкта БЕЗ посилань і без `code1C` → фіз. delete; approve обʼєкта З посиланнями → archive (не delete) + показ списку блокерів; approve документа → реверс боргу + recompute + audit; відновлення з архіву (`archived=false`).
- API-тести: `/deletions` (403 для чужого обʼєкта; 200 для свого), `/approve` та `/reject` (403 для не-адміна).
- Список клієнтів/документів: `markedForDeletion` виключено для менеджера, видно адміну.
- Регрес: `hideTrash` видалено, старі тести на префікси оновлено/прибрано.

### B11. Критерії приймання Блоку B

- Менеджер НЕ може фізично видалити клієнта/документ (лише позначка); спроба прямого DELETE не-адміном → 403.
- Діалог позначення вимагає причину; без причини — не відправляє.
- Позначений обʼєкт зникає зі списку менеджера, зʼявляється у черзі адміна з причиною.
- Адмін «Відхилити» → обʼєкт повертається. Адмін «Видалити» → перед стиранням **перевіряються всі звʼязки**; фіз. видалення тільки якщо посилань 0 і `code1C == null`, інакше автоматично архів (з показом списку блокерів), борг перерахований, запис у AuditLog.
- Обʼєкт з посиланнями або з `code1C` **неможливо** фізично стерти (лише архів). З архіву можна відновити.

## БЛОК C — Надійність боргу

**Навіщо:** борг — фінансові дані; зараз є два слабких місця (з попереднього аудиту).

### C1. Борг-рух — в одній транзакції з документом

- Зараз `applyDebtMovementSafe` (`lib/manager/debt-register.ts`) — fire-and-forget **після** коміту документа. При збої рух може не записатися.
- Зробити: у `sale-create.ts` та `cash-order.ts` записувати `MgrDebtMovement` **усередині** тієї ж `prisma.$transaction`, що й документ (upsert за ідемпотентним ключем `kind+sourceType+sourceId`), а `recomputeDebtForClients` — одразу після коміту (кеш можна оновлювати поза транзакцією, він похідний).
- Лишити нічний `recompute-client-debt.ts` як страхувальник.

### C2. Точний прострочений борг

- Зараз `overdueDebt` спрощений. Відтворити логіку 1С `ОтриматиБорг`: прострочка = сума рухів/документів, старших за граничну дату (термін оплати). Реалізувати у `debt-register.ts` розрахунок `overdueDebt` за датами `occurredAt` рухів (поріг — узгодити з user, напр. 14/30 днів) + оновлювати у `recomputeDebtForClients`.
- Тести на межові дати.

### C3. Критерії приймання

- Видалення/створення реалізації-оплати завжди лишає борг консистентним навіть при штучному збої поза транзакцією (тест).
- `overdueDebt` збігається з ручним підрахунком на 2-3 клієнтах.

## БЛОК D — Резервні копії та відновлення

**Навіщо:** після відмови від 1С наша PostgreSQL — **єдине** джерело правди. Раніше 1С була «другою копією» даних; тепер її немає.

### D1. Розширити політику бекапів (інструкція user-у, не код)

- Зараз: щоденний `pg_dump -Fc` о 03:00 → `E:\ltex-backups\`, retention **14 днів**.
- Рекомендації:
  - Збільшити retention (напр. 60 днів; + щотижнева копія на 6 міс).
  - Додати копію на **інший фізичний носій / хмару** (щоб втрата диска E: не знищила і базу, і бекапи).
  - Раз на місяць — **тест відновлення** з бекапу у тестову базу (перевірка, що бекап робочий).
- Задокументувати у `DEPLOYMENT.md` / `docs/CRON_SETUP.md`.

### D2. (опц.) Незмінність аудиту на рівні БД

- Зараз `AuditLog`/таймлайн — append-only лише в коді. За бажанням — тригер/right на рівні PostgreSQL, що забороняє UPDATE/DELETE у `audit_logs`. Низький пріоритет.
