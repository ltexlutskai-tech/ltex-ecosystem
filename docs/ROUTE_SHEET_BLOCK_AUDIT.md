> READ-ONLY аудит старої програми менеджера (`MobileAgent`, 1С:Підприємство 8.3) + центральної
> бази (`Central`) для точного відтворення блоку **«Маршрутний лист»** у нашому веб-стеку.
> Терміни 1С — кирилицею з коротким глосом. Цитати — по файлах/рядках.

## 0. Загальна модель (як це працює)

**Маршрутний лист (МЛ)** — документ на день виїзду агента власним транспортом. Зв'язна ланка
офіс↔виїзд:

1. **Менеджер/диспетчер** збирає у МЛ **Замовлення** клієнтів (`Заказы`), товари замовлень
   агрегуються (`ТоварыЗаказов`).
2. **Склад** фізично **сканує конкретні рандомні лоти** (мішки) під позиції замовлень — вкладка
   **Загрузка** (`ЗагрузкаМашины`); скан резолвить ШК → лот і авто-прив'язує до замовлення.
3. **Експедитор** усередині МЛ **створює Реалізації** (`Реализации`/`Продажи`) та **Оплати**
   (`Оплаты`, через процесор «Оплата») по кожному клієнту.
4. Те, чого не вистачило на складі — **Бракує** (`НеХватает`, обчислюване).
5. МЛ передається на мобільний через **SOAP-обмін** (бінарний `ХранилищеЗначения`), повертається
   заповненим; статуси **Составляется → Отправлен → Завершен**.

**⚠️ Документ існує і в мобільному, і в центральній базі**, але у нас в експорті є тільки
**мобільне** визначення об'єкта (`MobileAgent/Documents/МаршрутныйЛист/*`). Центральна структура
відома **лише з BSL-запитів** обміну (`ОбменАРМ`), `.xml` центрального документа в експорті немає.

## A. Дані документа (реквізити + 8 табличних частин)

### Реквізити шапки (`МаршрутныйЛист.xml`)

| Реквізит                     | Тип                               | Призначення                     |
| ---------------------------- | --------------------------------- | ------------------------------- |
| `Номер`                      | String(11), серія Рік, авто       | № документа                     |
| `Дата`                       | dateTime (обов'язк.)              | Дата складання                  |
| `СтатусДокумента`            | `EnumRef.СтатусыМаршрутногоЛиста` | Составляется/Отправлен/Завершен |
| `ДатаПриезда`                | dateTime                          | Планована дата візиту/доставки  |
| `СуммаДокумента`             | Decimal(15,2), ≥0                 | Сума по маршруту                |
| `Комментарий`                | String                            | Нотатка (= «назва» у списку)    |
| `Архив`                      | bool                              | Архів (lifecycle/обмін, mobile) |
| `В_1с`                       | bool                              | «Вивантажено в 1С» (mobile)     |
| `Проведен`/`ПометкаУдаления` | bool                              | Стандартні                      |

**Чого НЕМАЄ на документі (важливо для плану):**

- **Немає `Маршрут`** (`CatalogRef.Маршруты`) — попри існування довідника. Маршрут на документі не
  зберігається (на `Контрагенты` є таб. частина `Маршруты`, але не на МЛ).
- **Немає водія/експедитора** — Контрагент лише в рядках (по клієнту), особи експедитора як поля нема.
- **Немає кілометражу** — це окремий регістр (див. §I). CLAUDE.md це підтверджує («у замовленні є заглушка»).
- **Немає окремого UUID-ключа обміну** — ідентичність = стандартний `Ref` (GUID) + `В_1с`.

## B. Статуси + довідник Маршрути

- **`СтатусыМаршрутногоЛиста`** — 3 значення: `Составляется` / `Отправлен` / `Завершен`. **Без
  «Скасовано».** Дефолт при створенні — `Составляется` (`Module.bsl:1019`). **Логіки переходів
  немає** — статус редагується вручну (поле у шапці). Рядкове ↔ enum мапиться лише в обміні.
- **`Маршруты`** (довідник) — **порожній**: тільки `Код`(9) + `Наименование`(25). Без регіону/міста/
  агента. Форма елемента — read-only stub (`Отказ=Истина`). Тобто маршрут = просто назва-довідник.

---

## C. Проведення (`ObjectModule.bsl`) — НЕ рухає регістрів

- **`<RegisterRecords/>` порожній** — документ **не рухає жодних регістрів накопичення/відомостей**.
  `ОбработкаПроведения` **відсутня**. Проведення (`Проведен`) — лише прапор-гейт для зворотних посилань.
- **`ПриЗаписи`** (рядки 3-55, лише коли `Проведен`): для кожного рядка `Оплаты`/`Заказы`/`Реализации`
  — якщо у дочірнього документа поле `.МаршрутныйЛист` порожнє → ставить `= Ссылка` і `.Записать()`
  (у `Попытка…Исключение`). **Зворотний FK**: `Заказ`/`РеализацияТоваровУслуг`/`КассовыйОрдер` →
  назад на МЛ. Плюс `Общий.ЗарегистрироватьИзменения` для черги обміну (якщо не `мЭтоОбмен`).
- **`ПередЗаписью`** (57-96): синхронізує `Продажи.Цена/Сумма` з рядків Реалізації (ключ
  Ном+Хар+`ЦенаПродажиВес`), `Реализации.Сумма = Реализация.Сума`, і **знімає** `МаршрутныйЛист` із
  замовлень, прибраних з МЛ.
- **Немає** синхронізації Статус↔Архів↔ПометкаУдаления і **немає** генерації UUID у коді.

## D. Форма документа — поведінка вкладок (`Form/Module.bsl`)

**Життєвий цикл:** `ПриОткрытии`→розгорнути дерево+`ОбновитьДеревоТоваров`+`СтатисткаДокумента`
(лічильники). **Майже кожна дія викликає `ЗаписатьПриИзменении` — документ автозберігається** між
операціями (`Module.bsl:1153`).

### Заказы

- **«Добавить»** — порожній рядок, обрати наявне замовлення; **«Створити»** (`СоздатьЗаказ`,
  `Module.bsl:10`) — відкриває **нове** `Документ.Заказ.ФормаОбъекта` з `МаршрутныйЛист=Ссылка`
  (можна створити замовлення прямо з МЛ).
- **Пікер замовлення** (`ЗаказыЗаказПокупателяНачалоВыбора`, `:641`): фільтр = **виключити
  замовлення, що вже в будь-якому МЛ** (одне замовлення — в одному МЛ). **Без фільтра за маршрутом/
  містом/клієнтом.** Мульти-вибір підтримується.
- Зміна замовлення → резолв `Контрагент`+`Город`, копіювання рядків `Заказ.Товары` у `ТоварыЗаказов`
  (`ТоварыЗаказа`, `:159`). Видалення рядка → каскад видалення по `ТоварыЗаказов/Реализации/Продажи/Оплаты`.

### Товари

- Дві таблиці: прихована редагована `ТоварыЗаказов` + видиме **read-only дерево** `ДеревоТоваров`
  (групування **по замовленню**, підрядки по Номенклатура, підсумки Кількість/Сума).
- **«Заповнити»** (`ЗаполнитьТовар`, `:458`) — повна перебудова `ТоварыЗаказов` з усіх замовлень.
  **«Розгорнути»/«Згорнути»** — дерево.

### Загрузка (ядро) — скан

- **«Сканировать ШК»** (камера, mobile `СредстваМультимедиа.ПоказатьСканированиеШтрихКодов`) /
  **«Ввести ШК»** (ручне `ВвестиСтроку`). Деталі скану — §E.

### Реалізації

- Read-only список створених `РеализацияТоваровУслуг` (1 рядок/документ + Сума).
- **«Реалізація»** (`СоздатьРеализацию`, `:254`): анти-дубль `КонтрольРеализации` (вже є реалізація по
  цьому замовленні+МЛ? → блок), відкриває `РеализацияТоваровУслуг.ФормаОбъекта` з `Сделка`=замовлення,
  `Контрагент`, `МаршрутныйЛист`, `Наложка`=COD замовлення. **«Непланова реалізація»** — без замовлення.
- На створення → `ПроданныеТовары` (`:522`): рознесення `Реализация.Товары` у **`Продажи`** + **видалення
  відповідних рядків із `ЗагрузкаМашины`** (лот «спожитий» продажем). Viber-надсилання клієнту/у групу.

### Продажи

- Read-only **порядкова деталізація** проданого (на відміну від `Реализации` — підсумок по документу).

### Оплати

- **«Создать оплату»** (`СоздатьОплату`, `:541`): анти-дубль `КонтрольОплати`, рахує `СуммаКОплате`
  (сума реалізації − вже оплачено), **відкриває `Обработка.Оплата.Форма`** (той самий процесор каси з
  3 валютами+здачею). Повертає масив `КассовыйОрдер` → рядки `Оплаты`. Є й **«незапланована оплата»**.

### Бракує (`НеХватает`) — §G.

### Завдання

- Проста редагована таблиця `Контрагент`+`Коментар`. **Жодного коду** — вільні нотатки експедитора.

## E. Скан-логіка (баркод → лот) — `ОбработкаСканирования` (`:1080`) + `ДобавитьТовар` (`:1115`)

1. `ТипЦен` — **хардкод** `ТипыЦенНоменклатуры` код `"000000001"` (`:1162`).
2. `Общий.ПолучитьДанныеТовараПоШК(ШК, ТипЦен)` → `{Номенклатура, Характеристика, ЕдиницаИзмерения}`;
   якщо порожньо → «Не найден товар».
3. **Гард броні** `АктивнаБроньМішка` (`:1104`): якщо лот заброньований іншим (`ПериодБрони>now AND NOT
МояБронь`) → «Активна бронь мішка до …» і **відмова сканувати чужий лот**.
4. `ДобавитьТовар`:
   - **Дедуплікація**: якщо рядок з таким Ном+Хар уже є → «Товар уже добавлен!» і **відмова** (лот
     сканується **рівно раз**, без інкременту — мішок один фізичний).
   - **Вага** = `Число(Строка(Характеристика))` (презентація лота = вага у кг; fallback 20).
   - **Ціна**: `ЦенаПродажиВес = Общий.ПолучитьЦенуНоменклатуры(...)`, `Цена=ЦенаПродажиВес*Вес`,
     `Сумма=Кільк*Цена`.
   - **Авто-прив'язка до замовлення**: пошук у `ТоварыЗаказов` за Ном+Хар → копіює `ЗаказПокупателя`+
     `Контрагент` з першого співпадіння. Якщо нема — рядок без замовлення (неплановий).
   - **Без перевірки кількості при скані** (over/under — лише у Бракує/лічильнику).

## F. Створення Реалізацій + Оплат зсередини МЛ

- **Реалізація**: реюз `РеализацияТоваровУслуг` (наш блок **Реалізація**) з preset
  `Сделка`/`Контрагент`/`МаршрутныйЛист`/`Наложка`. Анти-дубль по (замовлення+МЛ). На створення —
  рознесення у `Продажи` + споживання рядків `ЗагрузкаМашины`.
- **Оплата**: реюз процесора **«Оплата»** (наш блок **Оплати/Каса**) з preset
  `СуммаКОплате`/`ЗаказПокупателя`/`Реализация`/`Контрагент`/`МаршрутныйЛист`. Анти-дубль по
  (замовлення+підстава). Повертає `КассовыйОрдер` → рядки `Оплаты`.
- **Видалення** реалізації/оплати — **фізичне** (`УдалитьНепосредственно`), з гардом: **відмова якщо
  `Архив`/`В_1с`** («синхронізований з 1С!»).

## G. Бракує (нестача) + лічильники

- **`ОбновитьИнформациюОНезаполненныхТоварахНаСервере`** (`:1406`): нестача =
  **замовлено − наявний складський залишок** (`ХарактеристикиНоменклатуры.Залишок`, **чужі брані лоти =
  0**), розподілена по рядках замовлень. Тобто це **НЕ замовлено−завантажено**, а замовлено−доступно-на-складі.
  Кнопка «Обновить…» просто `Записать()` (а `ПередЗаписью` перераховує).
- **Лічильник** (`СтатисткаДокумента`, `:1342`): `Заказов:` = к-ть рядків `Заказы`; `заказано:` =
  Σ `ТоварыЗаказов.Количество`; `загружено:` = Σ **`ТоварыЗаказов.КоличествоЗагружено`**; `не хватает:`
  = `НеХватает.Итог`.
- **⚠️ `КоличествоЗагружено` НЕ заповнюється кодом форми** — підтримується **сервером (обмін/центральна)**.
  У нашому порту «загружено» доведеться рахувати самим із рядків `ЗагрузкаМашины` (Σ по замовленні+номенклатурі).

## H. Обмін з 1С (центральний контракт) — `ОбменАРМ/Ext/Module.bsl`

> **Корекція до звіту user-а:** функції `СформироватьДанныеДляОтправки()` **не існує**. Реальний
> вивантаж — `СформироватьПакетДанных(ИдКлиента, Пароль)` (`:4`, WS-операція `MobileExchange.xml:111`)
> → `ДобавитьДокументы` (`:2138`) → `ДобавитьМаршрутныйЛист` (`:2632`). Серіалізація — **бінарний
> `ХранилищеЗначения` зі стисненням 9** (`Сериализовать`, `:5756`), не XML/JSON.

### Вивантаж (центральна → мобільний) — `ДобавитьМаршрутныйЛист` (`:2632`)

- Відбір: **лише МЛ, зареєстровані в `РегистрСведений.РегистрацияОбмена` для цього агента**
  (`Продавец=&ТорговийАгент`). **Без фільтра `Архив`.**
- **Не один UNION**, а 2 запити + обчислення: Запит #1 `Заказы×ТоварыЗаказов` (`:2636`), Запит #2
  `ЗагрузкаМашины` (`:2715`), + обчислена `НеХватает` (`ПолучитьТаблицуНеХватаетМаршрутныйЛист`, `:2806`,
  проти залишків `РегистрНакопления.ТоварыНаСклади`).
- Шапка у пакеті: `ИД`, `Ссылка`, `Дата`, `СтатусДокумента`(рядок), `Комментарий`, `ПометкаУдаления`.
  **`Номер`/`ДатаПриезда`/`СуммаДокумента` — НЕ кладуться у вивантаж** (хоч і вибираються).
- Масиви: `Заказы[]` (ЗаказПокупателя, Контрагент — **без `Город`**), `Товары[]` (=ТоварыЗаказов, з
  `КоличествоЗагружено`), `Загрузка[]` (=ЗагрузкаМашины, з `Штрихкод`/`Вес`/`ЦенаПродажиВес`),
  `НеХватает` (обчислена).

### Завантаж (мобільний → центральна) — `ОбработатьПакетДанных` (`:3199`)

- **Двофазно** (МЛ обробляється двічі): **Фаза A** `СоздатьМаршрутныеЛистыТовары` (`:5283`) — шапка +
  `Заказы`/`ТоварыЗаказов`/`Завдання`/`ЗагрузкаМашини`; **Фаза B** `СоздатьМаршрутныеЛистыДокументы`
  (`:5466`, **після** створення Реалізацій/ПКО) — `Продажи` + `Оплата`.
- Рядки — плоскі з дискримінатором **`ТабЧасть`**: оброблювані значення — `Заказы`, `ТоварыЗаказов`,
  `Завдання`, `ЗагрузкаМашини`, `Продажи`. **Немає `ТабЧасть="Реализации"`/`"Оплаты"`**: Реалізації
  приходять окремим верхнім ключем `Реализации`; Оплати **відновлюються на сервері** з пов'язаних
  ПКО/РКО (`:5582`). `ДатаПриезда` **приймається** на вході (`:5311`).
- **Статус рядок→enum** через тернар (`:5312`, `:5489`).
- **«Замок після завершення»**: `ВозвратПоЗакритомуМаршрутнику` (`:3993`) — якщо МЛ має статус
  `Завершен`, вхідний об'єкт **пропускається** (ID підтверджується, щоб мобільний перестав надсилати).
  Це реальний lock, **а не `Архив`** (рядків `Архив`/`мЭтоОбмен` у центральній **немає**; прапор
  обміну там — `мОбменАндроид`, і для МЛ він навіть не ставиться).
- `РеализацияТоваровУслуг.МаршрутныйЛист` (DocRef) — пишеться на вході (`:3718`), читається у вивантажі.
- **Відповідь**: `{ТаблицаИД, МассивОшибок, ИнфоМаршруты}` — `ИнфоМаршруты` = перерахована сервером
  `НеХватает` по кожному МЛ, повертається мобільному.

## I. Суміжне: Кілометраж, GPS, прапорець «замовлення в маршруті»

- **`КонтрагентыСАктуальнымиЗаказами`** (mobile InformationRegister): вимір `Контрагент`, ресурси
  `ЕстьПростроченыеАктуальныеЗаказы` + **`ЗаказВключенВМаршрутныйЛист`** (bool). Центральна **обчислює**
  його (LEFT JOIN `МаршрутныйЛист.Заказы`, `:6006`) для списку клієнтів. ✅ (звіт user-а вірний).
- **Кілометраж** (mobile `РегистрСведений.Кілометраж`, вимір Період+СтатусДня, ресурс Кілометраж):
  агент вводить на початок/кінець дня (`ФормаВводаКілометражу`). **Незакритий попередній день →
  попередження «Немає кінцевого кілометражу…» + блок закриття форми** (`Кілометраж/Module.bsl:6`,
  `ПередЗакрытием Отказ=Истина`). ✅ (звіт user-а вірний).
- **GPS** (mobile `РегистрацияКоординат`: Дата/Широта/Долгота) — `Геопозиціювання`/`ОпросКоординат`.
- Обидва **їдуть у тому ж мобільному пакеті** (`Обмен_АнализДанних`: ключі `Кілометраж` за 7 днів,
  `Координаты` за сьогодні). **⚠️ Центральний імпорт Кілометраж/GPS — «сирота»:** writers
  `ЗаписатьКілометражТорговогоАгента` (`:4103`) / `…Координаты` (`:4131`) **не мають викликача** в
  експорті, і `ОбработатьПакетДанных` не диспетчить ці ключі; назви регістрів навіть різняться
  (`КілометражТорговогоАгента`/`РегистрацияКоординатТорговых`). Тобто за наявними файлами не можна
  підтвердити, що центральна реально зберігає кілометраж/GPS з обміну.

### Рішення, що треба узгодити з user (питання)

1. **Маршрут/Експедитор/Кілометраж на документі** — 1С їх НЕ зберігає на МЛ (маршрут — порожній
   довідник, кілометраж — окремий регістр з блоком незакритого дня, GPS). Чи додаємо їх у наш МЛ і чи
   робимо механізм кілометражу/GPS зараз, чи відкладаємо?
2. **Визначення «Бракує»** — 1С рахує замовлено−складський*залишок (без чужих броней). У нас немає
   повного складського регістру 1С — рахувати замовлено−доступні*вільні_лоти, чи простіше
   замовлено−завантажено?
3. **Хто що робить у нас (ролі)** — у 1С усе в одній формі. У нас менеджер/склад/експедитор —
   різні люди. Розділяти права по вкладках (склад бачить лише Загрузку-скан; експедитор —
   Реалізації/Оплати), чи поки одна форма для всіх?
4. **Скан на складі** — камера/ручне поле ШК (як скрізь), лот сканується раз, авто-прив'язка до
   замовлення за Ном+лот; вага з лота. Підтверджуєте такий самий підхід?
5. **Обсяг/етапність** — блок великий (8 вкладок). Реалізації/Оплати **реюзають** готові блоки. Реальний
   1С-обмін (двофазний) — лише spec+mock, як домовлялись?

### Корекції до попереднього звіту user-а (звірка з кодом)

- Табличних частин **8**, не 6 (додатково `ЗагрузкаМашины` + `НеХватает`).
- `СформироватьДанныеДляОтправки()` не існує → `СформироватьПакетДанных`→`ДобавитьМаршрутныйЛист`.
- Вивантаж — 2 запити + обчислена нестача (не один UNION); `Номер/ДатаПриезда/Сумма` у вивантаж не йдуть.
- `мЭтоОбмен`/`Архив` — **мобільні** (у центральній їх нема; там `мОбменАндроид`, для МЛ не ставиться).
- «Замок» після роботи — статус `Завершен` (`ВозвратПоЗакритомуМаршрутнику`), не `Архив`.
- `Город` у `Заказы` — мобільна деривація з `Контрагент.Город`, не центральний вхід.
- Кілометраж/GPS у центральній — імпорт-«сирота» (writers без викликача) — потребує уточнення.
