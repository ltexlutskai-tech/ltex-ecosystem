# Сесія 5.7 — Доробка довідників/категорій/статусів (за фідбеком user 2026-06-18)

> Джерело: 3 аналітичні агенти. Кожен пункт = задача з планом. Позначки:
> 🟢 однозначна (роблю одразу) · 🟡 потребує рішення user · 🔴 деструктивне (підтвердження).

## 1. 🟢 Клієнт → менеджер (торговий агент)

**Знахідка:** поле `MgrClient.agentUserId` Є, але імпортер його не заповнює. У 1С агент клієнта = `Catalog.Контрагенты._Fld6889RRef` (тип `ТорговыеАгенты`, той самий простір, що `User.code1C`).
**План:** у `import-1c-historical.ts` додати `_Fld6889RRef` у `CUSTOMER_COLS`; lazy-guard `agentUserIdByHex`; у create+update MgrClient ставити `agentUserId = agentUserIdByHex.get(hex(_Fld6889RRef))`. UI вже є (admin-edit). Backfill: реімпорт `--entity customers`. Міграцій нема.

## 2. 🟢 Торгові агенти ↔ Користувачі

**Знахідка:** звʼязок уже існує (`User.code1C`=1С-агент; документи тримають `assignedAgentUserId`; `MgrTradeAgent.userId` авто-лінкується по code1C в імпорті). Бракує: авто-лінк при РУЧНОМУ створенні User + показ звʼязку в обох картках.
**План:** при POST/PATCH User з `code1C` → `mgrTradeAgent.updateMany({where:{code1C,userId:null},data:{userId}})`; у списку Користувачів показати «Торговий агент (1С)»; у `/manager/trade-agents` — фільтр «без User» + бейдж code1C. Канон: User = агент; MgrTradeAgent = дзеркало 1С. Документи НЕ переводити на MgrTradeAgent.

## 3. 🟢 Типи цін (MgrPriceType)

**Знахідка:** на ціноутворення НЕ впливає (документи рахують через `Price.priceType` опт/акція). Лишається лише як довідкова характеристика клієнта (читається у 3 завантажувачах картки). Видаляти не можна.
**План (Варіант A):** проста read-only сторінка `/manager/price-types` (code/label) + у каталозі `partial→ready` + опис «довідкове поле клієнта, не впливає на ціни документів».

## 4. 🟡 Категорії товарів з 1С (дерево груп)

**Знахідка:** імпортер СВІДОМО пропускає групи-папки 1С (`_Folder`) → усі товари в запасній «Імпортовано з 1С». `Category` має дерево (`parentId`), але НЕ має `code1C`. 1С Номенклатура — ієрархія `_Reference76` (`_ParentIDRRef`, `_Folder`).
**План:** +`Category.code1C` (additive міграція); новий 2-фазний `importCategories` (папки→Category по hex, тоді parentId); `importProducts` резолвить `categoryId` за `_ParentIDRRef` (+прапор `--recategorize` для backfill наявних); картка/фільтр за групою (піддерево).
**❓ Рішення user:** (Q1) окреме 1С-дерево груп паралельно веб-каталогу [рек.] чи злити з курованим; scope `--recategorize` (лише товари з `imported-1c` чи всі).

## 5. 🟡 Доступи за групами товарів (дизайн)

**План:** поле `Category.hiddenForRoles String[]` (deny-list, спадковість на піддерево, server-side enforcement у `buildPricesWhere` + картка + вітрина; admin/owner bypass). Самі правила user пропише пізніше — зараз лише місце під них.
**❓ Рішення user:** (Q непряме) deny-прапор [рек.] чи whitelist-таблиця. За замовчуванням роблю deny-прапор.

## 6. 🟡 Статуси клієнтів (механізм як у 1С)

**Знахідка:** у 1С — АВТО через ScheduledJob `ИзменениеСтатусовКонтрагентов` (BSL `МодульРегламентныхЗаданий:276-500`): 7 статусів (Активний/Неактивний/Новий/Малоактивний/Закрився/ТимчасовоНеПрацює/Потенційний); **«Статус»** = за ПОПЕРЕДНІЙ місяць (≥2 продажі→Активний, =1→Малоактивний, =0→Неактивний), **«Оперативний статус»** = за ПОТОЧНИЙ місяць; нові (<30 днів)→Потенційний; кожна зміна → історія. У нас структура є, авто-механізму нема.
**План:** seed 7 статусів (коди/назви відомі з `Predefined.xml` — БЕЗ MSSQL); helper `recompute-client-statuses.ts` (порт BSL-правил, рахує з наших `Sale`/`SalesMovement`); cron-роут + Windows Task; «Оперативний статус» у UI → read-only (бейдж «авто»), «Статус» лишити редагованим. Поточний статус наповниться сам із перерахунку (MSSQL-декод не потрібен!).
**❓ Рішення user:** (Q4) увімкнути авто-перерахунок як у 1С (щомісяця) чи лишити лише ручне.

## 7. 🔴 Чистка не-1С товарів/лотів/штрихкодів

**Знахідка:** дискримінатор Product: `code1C IS NULL` = не-1С. ⚠️ АЛЕ Excel-каталог S71 теж `code1C NULL` (легітимний!). Lot взагалі без `code1C` (надійний дискримінатор не-1С: `receivingId IS NOT NULL`). Історія продажів захищена FK-Restrict (OrderItem/SaleItem на product/lot блокують видалення).
**План:** аудит-скрипт (порахувати) → мʼяке архівування (зворотно) → фіз. видалення лише без посилань (за окремим підтвердженням). Рекомендація: спершу `--entity lots` з новим `Lot.code1C` для надійного дискримінатора.
**❓ Рішення user:** (Q2) підхід: лише аудит / архівувати / видаляти; (Q3) Excel-каталог S71 — лишити як легітимний чи теж чистити.

## Порядок

🟢 1,2,3 — роблю одразу (без рішень). 🟡 4,5,6 — після відповідей. 🔴 7 — після відповідей + окреме підтвердження на кожен DELETE-батч.
**Серверні блокери для частини:** декод `_Fld` статусу контрагента (для п.6 backfill — НЕ критично, перерахунок дає сам); `Lot.code1C` реімпорт (для п.7 дискримінатора).
