## Прапорці CLI

| Прапорець            | Дія                                                                                                                |
| -------------------- | ------------------------------------------------------------------------------------------------------------------ |
| `--dry-run`          | читає 1С + резолвить зв'язки, НЕ пише; друкує звірку кількостей                                                    |

## Ключові технічні моменти

### 1. `_IDRRef` binary(16) → наш id

1С зберігає посилання як 16 байт у власному порядку (НЕ як .NET GUID).
Підхід: під час імпорту кожного довідника будуємо **словник у пам'яті** `hex(_IDRRef) → наш cuid/code1C`.
Документи резолвлять FK через ці словники. Порядок імпорту (нижче) гарантує, що довідник уже в словнику.
`hex` беремо як lower-case hex 16 байт БЕЗ перестановки (звіряємо консистентно з обох боків — головне однаковий ключ).

### 2. Поліморфні посилання (`_Fld<N>_TYPE`/`_RTRef`/`_RRRef`)

Напр. Контрагент/ДокументОснование у ПКО/РКО, Владелец у штрихкодах.
`_RTRef` (4 байти) = код типу таблиці; `_RRRef` (16) = сам ref.
Читаємо `_RRRef` лише коли `_RTRef` = потрібний тип (мапу типів будуємо з dbnames).

### 3. Enum / статуси

`Доставка` = `EnumRef.НаявністьДоставки`; статуси замовлень = каталоги `_Reference6930/6931`.
Перший прохід: ВИЧИТАТИ унікальні значення цих полів з 1С → побудувати мапу на наші `draft|sent|posted|cancelled`. Зробити окремий маленький допоміжний прохід `--entity _enums_probe` що друкує знайдені значення.

### 4. Стримінг (не вантажити 785К у пам'ять)

Читаємо MSSQL пагінацією: `ORDER BY _IDRRef OFFSET @off ROWS FETCH NEXT @batch ROWS ONLY` (MSSQL 2012+, є).
Пишемо Prisma батчами: довідники — `createMany({skipDuplicates})` АБО `upsert` по `code1C`; документи з рядками — транзакція на батч.

### 5. Idempotency

- Довідники: `upsert` по унікальному `code1C` (= 1С `_Code`).
- Документи: `upsert` по `code1C` (= 1С `_Number` або `_IDRRef`-hex якщо номер не унікальний).
- Рядки документів: видалити+вставити в межах батч-транзакції власника (без дублів).

### 6. Звірка (reconciliation)

Наприкінці друкуємо таблицю: сутність | у 1С | записано | пропущено | помилок. У dry-run — «would write».

## Порядок імпорту (FK-залежності)

1. Довідники-листочки: `Города`, `Качество`, `ВидыЦен`, `ЕдиницыИзмерения`, `Склады` → наші dict-моделі / поля.
2. `Номенклатура` → `Product` (словник hex→id).
3. `Контрагенты` → `Customer` (+ дзеркало у `MgrClient` по `code1C`).
4. `Штрихкоды` (`_InfoRg5249`) → план barcode→характеристика.
5. `ХарактеристикиНоменклатуры` → `Lot` (barcode з кроку 4; без ШК — синтетичний `L-<hex>`).
6. `ЦеныНоменклатуры` → `Price`.
7. `ЗаказПокупателя` (+VT) → `Order`+`OrderItem`.
8. `МаршрутныйЛист` → `RouteSheet`.
9. `РеализацияТоваровУслуг` (+VT) → `Sale`+`SaleItem`.
10. ПКО/РКО → `MgrCashOrder`.
