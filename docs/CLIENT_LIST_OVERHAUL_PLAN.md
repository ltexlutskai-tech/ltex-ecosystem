# Блок «Список клієнтів / Картка / Ліди» — план вдосконалення (2026-07-16)

Мета: зробити роботу з клієнтською базою максимально швидкою й гнучкою — щоб менеджер за секунди знаходив потрібних клієнтів, бачив кого треба «дотиснути» сьогодні, і фільтрував базу по будь-якому зрізу (історія, теги, асортимент, колір-пріоритет).

Комунікація з user — проста мова. Гілка: `claude/client-list-filtering-search-wdnvre`.

---

## A. Що вже є (факти з коду)

### Список клієнтів

- Пошук (тулбар) матчить: ім'я, телефон, місто, телефони. **API-роут** ще матчить `keywords`, але **сторінковий лоадер `load-clients.ts` — НІ** (розбіжність: пошук по тегах у реальному списку не працює).
- Фільтри у шторці: статус (загальний/оперативний), канал, доставка, категорія ТТ, тип цін, асортимент-код, маршрут, агент, борг/переплата, діапазон днів з останньої покупки, діапазон дати створення, регіон/місто (мульти), «тільки мої», «прибрати приховані».
- API-роут має ЩЕ фільтри, які UI не використовує: діапазони боргу/простроч.боргу/місячного обсягу, `licenseExpiresBefore`, `dialogStatus`, `hasNewMessage`, `isViberLinked`.
- Кастомізація вигляду (`MgrUserViewPrefs`): видимі колонки + видимі фільтри, порядок. Ключа `keywords`-колонки/фільтра **немає**.
- Сортування по: ім'я, телефон, code1C, ТТ, регіон, місто, борг, простроч.борг, обсяг, днів з покупки, синхронізовано, створено, агент.

### Картка (11 вкладок)

Історія · Нагадування · Соцмережі · Реквізити · **Ключові слова** · Історія продаж · **Асортимент** · Замовлення · Презентації · Іст.презентацій · Рухи боргу.

- **Ключові слова** = одне текстове поле `MgrClient.keywords` (теги через кому), чипи, автозбереження. Чипи сірі (без кольорів).
- **Асортимент** (вкладка) рахується наживо з продажів (`SaleItem`), а не з моделі `MgrClientAssortmentItem` (модель вантажиться, але UI її не показує). Колонки: артикул, назва, разів, мішків, сер.ціна €/кг, остання покупка. Пошуку/фільтра всередині немає.
- **Історія** (timeline): види подій `order/sale/payment/bron/reminder/comment/close_orders/debt_correction` (+легасі viber/sync/note_1c). Ручний лише `comment`. **Пошуку/фільтра всередині історії немає**; на сторінці — останні 50 записів; API історії приймає лише `page/pageSize`.
- Колір: лише через `MgrClientStatus.colorHex` (бейдж статусу). Окремого клієнт-кольору/пріоритету немає.

### Ліди

- Один фільтр `filter`: активні / конвертовані / відхилені / усі. **Без текстового пошуку**, без пагінації (ліміт 200). Колонки: ім'я/телефон/місто/статус/дата. Поле `region` є в моделі, але в UI не показане/не фільтрується.

## B. Що беремо у 1С (знайдено в конфізі)

### 1. Фільтр/розмальовка по кольорах (головна знахідка)

У 1С **колір не зберігається** — він обчислюється з давності останнього контакту + наявності активних замовлень (`РегистрСведений.РаботаСКлиентом` + `КонтрагентыСАктуальнымиЗаказами`). Поле `СтатусКлиента` (рядок), розмальовка через «Условное оформление», і по ньому ж можна **фільтрувати**:

| Колір (наш аналог)   | Правило                         | Значення 1С |
| -------------------- | ------------------------------- | ----------- |
| 🟢 Зелений           | є активні замовлення            | PaleGreen   |
| 🟡 Жовтий            | контакт сьогодні                | Yellow      |
| ⚪ Білий             | контакт за останні ≤7 днів      | білий       |
| 🌸 Рожевий           | контакт 7–14 днів тому          | MistyRose   |
| 🔴 Червоний          | контакт >14 днів тому           | Tomato      |
| 🩷 Особливий рожевий | взагалі немає роботи з клієнтом | #FF8DAD     |

Це фактично «світлофор пріоритету»: червоні — терміново дзвонити, зелені — в роботі. У сучасних CRM це зветься "rotting"/last-activity color.

### 2. Пошук по історії роботи

Поле `ФільтрИстория` — вводиш текст → SQL `Запись ПОДОБНО %текст%` по регістру `РаботаСКлиентом` (нотатки взаємодій, до 1024 символів) → список звужується до клієнтів, у чиїй історії є цей текст. Тобто **пошук по історії як фільтр списку**.

### 3. Ключові слова (теги)

У 1С теги = поле `Асортимент.Артикул` (синонім буквально «Артикул / Ключові слова»). Фільтр `ФільтрКлючовіСлова` = **список слів** з тумблером **«або» (OR)** vs режим AND (клієнт має містити ВСІ введені слова). Матч — `Содержит` (Contains).

## C. Аналіз CRM-систем → що беремо

| Система                  | Сильна сторона                                                                               | Що беремо для нас                                                                                                |
| ------------------------ | -------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **SalesDrive / KeyCRM**  | Швидкі мультиканальні продажі, збережені фільтри-вкладки, масові дії, кольорові теги/статуси | Збережені пресети фільтрів («Мої вибірки»); масові дії у списку; кольорові теги                                  |
| **Pipedrive**            | «Rotting» — угода червоніє коли давно без руху; активності-нагадування                       | Прямий аналог 1С-кольору (light-signal по давності контакту) — робимо як головну «фішку» списку                  |
| **HubSpot**              | Сегментація списків, збережені views, колонка+фільтр «last activity», кольорові labels       | Колонка «останній контакт» + фільтр; збережені сегменти; кольорові мітки замість сірих чипів                     |
| **Sitniks**              | Месенджер-центричний inbox                                                                   | У нас вже є чат-inbox — додати «останнє повідомлення/непрочитане» у список як фільтр (`hasNewMessage` вже в API) |
| **NetHunt**              | B2B, картка як CRM-запис із таймлайном                                                       | Пошук/фільтр всередині таймлайну картки                                                                          |
| **KeepinCRM / Zoho**     | Прості кастомні поля, AI-підказки                                                            | Кастом-теги з кольором; (AI — пізніше)                                                                           |
| **Creatio / Salesforce** | Корпоративні воронки, дашборди                                                               | Не беремо — надлишок для опту; беремо лише ідею «сегмент = збережений фільтр»                                    |

**Наскрізні висновки (беремо):**

1. Кольоровий світлофор пріоритету у списку + фільтр по ньому (1С + Pipedrive).
2. Пошук по історії — і всередині картки, і як фільтр списку (1С + NetHunt/HubSpot).
3. Теги: кольорові, з мультивибором AND/OR, пошук+фільтр у списку (1С + HubSpot).
4. Фільтр/пошук по асортименту клієнта (що він реально бере) — сегментація «кому пропонувати цей товар».
5. Збережені пресети фільтрів («Мої вибірки») + швидкі чипи згори.
6. Ліди: текстовий пошук + пагінація + фільтр по місту/регіону/джерелу + видимий трек конвертації.

## D. План робіт (пропозиція, по етапах)

### Етап 1 — Фундамент пошуку/фільтрів списку

1.1. **Фікс пошуку по тегах**: додати `keywords` у пошук `load-clients.ts` (зараз матчить лише API). +додати ключ колонки `keywords` і фільтр-ключ у `view-defaults.ts`.
1.2. **Пошук по історії як фільтр списку**: новий параметр `historySearch` → підзапит по `MgrClientTimelineEntry.body` (contains) → звузити список до клієнтів з таким записом. Врахувати ownership. UI — поле у шторці фільтрів + швидкий інпут.
1.3. **Фільтр по ключових словах (теги)**: параметр `keywords` як список слів + тумблер AND/OR (як 1С `ФільтрКлючовіСлова100`). Матч `contains` по полю `keywords`.
1.4. **Фільтр по асортименту**: параметр `assortment` (артикул/назва товару) → клієнти, що купували цей товар (через `SaleItem`→`code1C`, або через `MgrClientAssortmentItem`). Рішення про джерело — див. питання №3.

### Етап 2 — Кольоровий світлофор пріоритету

2.1. Обчислити «статус-колір» клієнта на льоту з `daysSinceLastPurchase` (є) + активних замовлень (є в 1С-логіці; у нас — pending/актуальні замовлення). Порогів дотримуємось 1С: сьогодні/≤7д/7–14д/>14д/немає роботи/є активні.
2.2. Колонка-індикатор (кольорова крапка/смужка рядка) у списку + легенда.
2.3. **Фільтр по кольору** — швидкі чипи «🔴 Давно не працювали / 🟡 Сьогодні / 🟢 В роботі / 🩷 Без історії».
2.4. (Опційно) ручний колір-пріоритет клієнта поверх авто (нове поле) — див. питання №4.

### Етап 3 — Картка клієнта

3.1. **Пошук+фільтр у вкладці Історія**: поле пошуку по тексту + фільтр по виду події (замовлення/оплата/коментар/…) + діапазон дат. Розширити API історії (`search`, `kind`, `from`, `to`) + прибрати «стелю 50» на пагінацію.
3.2. **Кольорові теги** замість сірих чипів: палітра на тег (проста — набір кольорів на вибір). Клік по тегу → перехід у список з цим тегом.
3.3. **Асортимент**: додати пошук/фільтр всередині вкладки + кнопку «показати всіх клієнтів, що беруть цей товар». Визначитись із джерелом (продажі vs довідник) — питання №3.
3.4. Показати колір-світлофор і в шапці картки.

### Етап 4 — Збережені вибірки + масові дії

4.1. **Пресети фільтрів** («Мої вибірки»): зберегти поточний набір фільтрів під назвою, швидко застосувати. Нова таблиця або розширення `MgrUserViewPrefs`.
4.2. Швидкі чипи-пресети згори списку (напр. «Боржники», «Давно не дзвонили», «Нові ліди»).
4.3. Масові дії у списку (перепризначити агента вже є для admin — розширити: масово додати тег, змінити статус).

### Етап 5 — Ліди

5.1. Текстовий пошук (ім'я/телефон/місто) + пагінація.
5.2. Фільтри: місто, регіон, джерело, діапазон дат.
5.3. Колонка «джерело» + трек конвертації (коли/ким конвертований, лінк на картку).

## E0. РІШЕННЯ USER (2026-07-16) — зафіксовано

1. **Що = «контакт» для світлофора:** будь-яка подія в історії (timeline) — замовлення/оплата/коментар/бронь/нагадування. Рахуємо давність від останньої події timeline (fallback — `lastPurchaseAt`/`daysSinceLastPurchase`).
2. **Асортимент-фільтр:** з поля «Асортимент» у картці, яке заповнюється авто з продажів → тобто з реальних продажів (`SaleItem` по `code1C`). Ручний довідник не використовуємо.
3. **Ручний колір:** НЕ потрібен. Лише авто-світлофор.
4. **Обсяг:** робимо етапи 1–3 одним великим блоком.

## E. Питання до user (вирішено — див. E0)

1. **Пороги кольору** — лишаємо 1С-логіку (сьогодні / ≤7д / 7–14д / >14д / немає роботи / є активні замовлення), чи хочете інші дні?
2. **Що вважати «контактом»** для світлофора: остання покупка (є поле), АБО будь-який запис в історії (замовлення/оплата/коментар/дзвінок)? У 1С — будь-яка взаємодія в регістрі. Рекомендую: остання подія timeline.
3. **Асортимент-фільтр джерело**: рахувати з реальних продажів (`SaleItem` — що клієнт справді брав) чи з ручного довідника «Асортимент» (`MgrClientAssortmentItem`)? Зараз вкладка показує продажі. Рекомендую: продажі (правдиві дані), а довідник — окремо як «цільовий асортимент».
4. **Ручний колір/пріоритет клієнта** поверх авто-світлофора — потрібен, чи достатньо авто?
5. **Теги — кольори**: даємо фіксовану палітру (напр. 8 кольорів на вибір) чи лишаємо теги єдиного стилю, а колір — тільки світлофор?
6. **Пріоритет етапів**: пропоную 1 → 2 → 3, бо пошук/фільтри + світлофор дають найбільший приріст швидкості. Ліди (5) і пресети (4) — після.

## F. Технічні нотатки для воркерів

- Дві гілки фільтрації (`load-clients.ts` для сторінки vs `route.ts` API) розходяться — при додаванні нових фільтрів синхронізувати обидві або звести до спільного білдера `where`.
- Ownership (`client-visibility.ts`) застосовувати до всіх нових підзапитів (історія/асортимент), інакше менеджер побачить чужих.
- Регіон/місто-опції — це `distinct` з наявних клієнтів, не довідник.
- Перед мержем: typecheck + `vitest run` (store) + prettier. Мерж у `main` + деплой (без SSH — диктувати user).
