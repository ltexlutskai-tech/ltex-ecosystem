## 1. Логіка 1С

### 1.1. Документ `ОбщаяПотребність` — це наш орієнтир для блоку «Потреби»

`Document.ОбщаяПотребність` — **легкий кастомний** документ L-TEX (НЕ типовий УПП). Його
екранна форма (`Forms/ФормаДокумента/Ext/Form.bin`, binary managed-form) має **3 вкладки**,
що відповідають його табличним частинам (підтверджено `Documents/ОбщаяПотребність.xml`):

| Вкладка (екран)        | Таблична частина    | Колонки                                                                                                                                            |
| ---------------------- | ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Заказы покупателей** | `ЗаказыПокупателей` | `ЗаказПокупателя`, `Контрагент`, `Город`                                                                                                           |
| **Товары**             | `Товары`            | `Номенклатура`, `ХарактеристикаНоменклатуры`, `ЕдиницаИзмерения`, **`Заказано`**, **`Остаток`**, **`Количество`**, `Контрагент`, `Зарезервировано` |
| **Торговые агенты**    | `ТоварыАгентов`     | `ТорговийАгент`, `Номенклатура`, `ЕдиницаИзмерения`, **`Заказано`**                                                                                |

(Документ також має приховані ТЧ `Поставщики` / `ЗаказыПоставщикам` — для генерації замовлень
постачальникам через `СформироватьЗаказыПоставщикам()` у ObjectModule.bsl:6. Для L-TEX
менеджерки **не релевантно** — постачальники секонд-хенду не замовляються через систему.)

> ⚠️ `Document.ФормированиеПотребностей` — це **інший**, важкий типовий-УПП документ
> (пише в регістр `Календарные потребности в номенклатуре`, працює з планами продажу/закупівлі,
> сценаріями, проєктами — `Ext/Help/ru.html`). Скриншоти L-TEX відповідають **ОбщаяПотребність**,
> не ФормированиеПотребностей. У нашій реалізації ми **відтворюємо ОбщаяПотребність** і
> ФормированиеПотребностей ігноруємо.

### 1.2. Формули вкладки «Товары» (точний порт з Form.bin)

Активний (не закоментований) запит заповнення (`Заполнить`) працює так:

1. **`ВТТовар`** — `Заказано`: сума замовленого по товару серед перелічених замовлень:

   ```
   ВЫБРАТЬ Номенклатура, СУММА(Количество) КАК КоличествоЗаказано, ЕдиницаИзмерения
   ИЗ Документ.ЗаказПокупателя.Товары
   ГДЕ Ссылка В (&Заказы)
   СГРУППИРОВАТЬ ПО Номенклатура, ЕдиницаИзмерения
   ```

2. **`ВТСвободныеОстатки`** — `Остаток` (вільний складський залишок) = баланс складу мінус резерв:

   ```
   КоличествоСвободныйОстаток =
       РегистрНакопления.ТоварыНаСкладах.Остатки.КоличествоОстаток
     − РегистрНакопления.ТоварыВРезервеНаСкладах.Остатки.КоличествоОстаток
   ```

   (групується до рівня номенклатури; склади звужуються до складів зі списку замовлень).

3. **Фінальний select + цикл заповнення рядків `Товары`:**

   ```
   НоваяСтрока.Заказано  = СУММА(КоличествоЗаказано)            // по товару
   НоваяСтрока.Остаток   = вільний залишок (з ВТСвободныеОстатки)
   // capping по попередніх проведених документах ОбщаяПотребність за добу:
   НоваяСтрока.Остаток   = МИН(Остаток, попередній_документ.Остаток)
   НоваяСтрока.Количество = НоваяСтрока.Заказано − НоваяСтрока.Остаток
   ```

   **➜ КЛЮЧОВА ФОРМУЛА: `Количество = Заказано − Остаток`** — це чиста потреба (скільки треба
   знайти/докупити). Додатне = бракує, ≤ 0 = вистачає на складі.
   Источник «Остаток» — регістр **`ТоварыНаСкладах.Остатки − ТоварыВРезервеНаСкладах.Остатки`**
   (вільний залишок). Capping `МИН(...)` по попередніх документах за добу — щоб два документи
   «Потреба» за один день не рахували один і той самий залишок двічі (для нас неактуально —
   ми рахуємо on-the-fly без збереження документа).

### 1.3. Вкладка «Торговые агенти»

Та сама потреба, розбита по (торговий агент, товар). Заказано береться з кожного замовлення
**по його агенту** (`ЗаказПокупателяТовары.Ссылка.ТорговийАгент`):

```
ВЫБРАТЬ Ссылка.ТорговийАгент, Номенклатура, СУММА(Количество) КАК Заказано, ЕдиницаИзмерения
ИЗ Документ.ЗаказПокупателя.Товары
ГДЕ Ссылка В (&Заказы)
СГРУППИРОВАТЬ ПО Ссылка.ТорговийАгент, Номенклатура, ЕдиницаИзмерения
```

Тобто кожне замовлення «належить» агенту (`Заказ.ТорговийАгент` / `Заказ.Контрагент.ТорговыйАгент`),
і його позиції додаються у кошик цього агента. **`Остаток` тут НЕ враховується** — лише `Заказано`.

### 1.4. Які замовлення потрапляють у документ

`Заказы` = перелік у ТЧ `ЗаказыПокупателей`. Заповнюється кнопкою «Заполнить»/«Подбор»
(вибірка з форми підбору). Бізнес-логіка L-TEX: у документ кладуть **актуальні** замовлення
(`СтатусЗамовлення = Новий`). Підтверджено непрямо: і екран Закриття старих замовлень, і
мобільний обмін (`ОбменАРМ.ПолучитьДанныеНезакрытыхЗаказов_v1`) фільтрують замовлення по статусу.

## 2. Актуальне / Неактуальне замовлення (1С → наш стан)

### 2.1. Як це у 1С (з research-агента, цитати з `docs/1c-export-2026-06-02/`)

- **Поле актуальності** = `ЗаказПокупателя.СтатусЗамовлення` (`CatalogRef.СтатусиЗамовлень`),
  значення `Новий` (актуальне) / `Неактуальний` (закрите). НЕ окремий boolean.
  (`Documents/ЗаказПокупателя.xml:1509`; `Catalogs/СтатусиЗамовлень/Ext/Predefined.xml`).
- **Жорсткого правила «одне актуальне замовлення на клієнта» у БД 1С НЕМАЄ.** Технічно
  кілька `Новий` можуть співіснувати. Нове замовлення стартує `Новий`
  (`ЗаказПокупателя/Ext/ObjectModule.bsl:2950`), старе **не** перемикається автоматично при
  створенні нового.
- Замовлення стає `Неактуальний` через **явні** механізми:
  1. **`Документ.ЗакрытиеЗаказовПокупателей`** — при проведенні `ПриЗаписи` ставить
     `мЗаказ.СтатусЗамовлення = Неактуальний` для кожного замовлення у документі
     (`ЗакрытиеЗаказовПокупателей/Ext/ObjectModule.bsl:445-452`).
  2. **Проведення `МаршрутныйЛист`** → `ИзменитьАктуальностьЗаказа` перемикає `Новий → Неактуальний`
     для замовлень у маршруті (`МаршрутныйЛист/Ext/ObjectModule.bsl:2196-2227`).
  3. **Мобільний `ОбменАРМ.ЗакрытьСтарыеЗаказы_v1`** (`Module.bsl:5174-5279`): знаходить найстаріше
     відкрите замовлення з залишком, опційно створює нове, і **авто-створює `ЗакрытиеЗаказовПокупателей`**
     для старих → проведення → `ПриЗаписи` ставить `Неактуальний`.
  4. `ОбменАРМ.ПолучитьДанныеНезакрытыхЗаказов_v1` (`Module.bsl:382-398` → `:6590+`) — віддає
     мобільному агенту тільки `Новий`-замовлення (фільтр `СтатусЗамовлення = &СтатусЗамовлення`).

**Висновок:** «одне актуальне на клієнта» — це **операційна угода менеджерів** (нове велике
замовлення на сезон → старі закривають через Закриття старих замовлень), а не DB-constraint.

### 2.2. Наш поточний стан

- `Order.isActual: Boolean @default(true)` (= `СтатусЗамовлення == Новий`) — уже існує
  (schema.prisma:241). Окремо `archived` (= проведено в 1С), `closedAt`/`closeReasonId`/`closedByUserId`/`closeNotes`
  (блок Закриття, міграція `20260609`). `Sale` має дзеркальні `isActual`/`archived` (schema:344-345).
- `orders-list.ts` уже віддає `isActual` (рядки 184/204) і має колонку «Актуальний» у таблиці.
- **Створення замовлення (`lib/manager/order-create.ts::createOrderWithItems`) НЕ перевіряє
  наявність актуального замовлення** — нове завжди `isActual=true` (дефолт), старі не чіпаються.
- Блок Закриття (`app/api/v1/manager/closures/[clientId]/route.ts`) — локальний READ-ONLY:
  показує незакриті замовлення клієнта з прогресом «продано/замовлено» (через `Sale.orderId`→`SaleItem`);
  саме закриття робиться на сторінці замовлення `/manager/orders/[id]` (кнопка «Закрити»).

### 2.3. План: Актуальне/Неактуальне + правило «одне актуальне на клієнта»

**(a) Відображення/встановлення** — нічого не міняємо у візуалі: колонка «Актуальний» уже є.
`isActual` ставиться `true` при створенні; знімається коли:

- замовлення закрито (`closedAt != null` → також `isActual=false`); АБО
- проведено/архівовано (`archived=true` → `isActual=false`); АБО
- менеджер вручну зняв (через нову дію — див. нижче).

> Рекомендація: тримати інваріант **`closedAt != null ⇒ isActual=false`** і **`archived ⇒ isActual=false`**.
> Закриття замовлення (`POST /orders/[id]/close`) має додатково виставляти `isActual=false`
> (зараз воно ставить `closedAt`/reason, але треба перевірити що `isActual` теж знімається — **TODO для воркера: дописати у close-endpoint**).

**(b) Правило «одне актуальне замовлення на клієнта» при створенні** — рекомендований UX
(дзеркалить операційну угоду, але без 1С-автозакриття, бо у нас закриття = свідома дія менеджера):

При `POST /api/v1/manager/orders` (і `createOrderWithItems`):

1. Перед створенням — запит: чи є у цього `customerId` замовлення з `isActual=true AND archived=false AND closedAt IS NULL`?
2. **Якщо є → НЕ створювати мовчки.** Повернути `409 { code: "active_order_exists", existingOrderId, existingOrderNumber }`.
3. UI на 409 показує діалог: _«У клієнта вже є актуальне замовлення №X. Що зробити?»_ з опціями:
   - **«Відкрити існуюче»** → редирект на `/manager/orders/{existingOrderId}` (дозаповнити його);
   - **«Закрити старі та створити нове»** → відкрити блок Закриття
     (`/manager/closures?clientId=...` або модалку close) → після закриття старого `POST /orders` проходить;
   - **«Все одно створити»** (опційно, тільки admin/senior) → повторний POST з прапором `?force=true`,
     що при створенні нового **знімає `isActual` зі старих** цього клієнта у тій самій транзакції
     (м'який аналог 1С-автозакриття, БЕЗ постингу документа закриття).

**Прив'язка до блоку Закриття:** опція «закрити старі» — це наявний flow Закриття. Якщо є
зв'язка з `Sale` (продано) — закривати з причиною; якщо ні — пропонувати причину з `OrderCloseReason`.

**(c) Partial unique index чи app-level?**

| Підхід                                                                                                 | Оцінка                                                                                                                                                                                                                                                |
| ------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Partial unique index** `@@unique([customerId]) WHERE isActual AND NOT archived AND closedAt IS NULL` | Prisma НЕ підтримує partial-unique декларативно; довелося б raw-SQL у міграції. **Жорстко** — зламає історичний імпорт (де у клієнта реально кілька `Новий`-замовлень з 1С, бо 1С це дозволяв), а також ускладнить «force create». **НЕ рекомендую.** |
| **App-level enforcement** (перевірка у endpoint + UX-діалог)                                           | Гнучко, дружньо до історичних даних, дозволяє admin-override. **Рекомендую.**                                                                                                                                                                         |

**Висновок: app-level (b), без DB-constraint.** Опційно — non-unique індекс
`@@index([customerId, isActual])` для швидкого lookup (мінорна оптимізація, можна без міграції на старті).

## 3. Блок «Потреби» — `/manager/needs`

Нова сторінка-аналітика (НЕ документ — у нас потреба рахується **on-the-fly**, без збереження
ОбщаяПотребність-документа; так само як Закриття/Бракує рахуються наживо).

### 3.1. Джерело даних

- **Замовлення-джерела**: усі **актуальні, не-архівні, не-закриті** замовлення
  (`isActual=true AND archived=false AND closedAt IS NULL`) — опційно з фільтром по клієнту/агенту/місту/даті.
  Ownership: manager бачить тільки свої (через `getMyClientCodes1C` → `customer.code1C`), admin — усі
  (той самий патерн, що `orders-list.ts` / закриття).
- **Заказано** = `Σ OrderItem.weight` (або `quantity`) по товару серед цих замовлень.
  > ⚠️ Одиниця: L-TEX торгує **вагою (кг)** — `OrderItem.weight`. У 1С формула в одиницях
  > `ЕдиницаИзмерения` (кг для більшості, шт/пар для взуття). Рекомендація: рахувати **по weight (кг)**
  > для kg-товарів і по `quantity` (шт/пар) для `Product.priceUnit != 'kg'`. На старті — **тільки weight**,
  > одиниця показується з `Product.priceUnit` (kg/piece/pair).
- **Остаток** (вільний складський залишок) = доступні **вільні лоти** товару:
  `Lot WHERE status='free' AND NOT isActiveReservation` — **переюзати визначення з
  `lib/manager/route-sheet-loading.ts`** (там «Бракує» рахує саме так).
  > ⚠️ Нюанс одиниці: route-sheet рахує вільні лоти **по count** (1 лот = 1). Для потреби у кг
  > треба **`Σ Lot.weight`** вільних лотів, не count. Рекомендація: новий хелпер агрегує
  > `Σ weight` (kg-товари) / `Σ quantity` (шт/пар-товари) вільних лотів — **це розширення, не
  > прямий reuse `computeShortage`**; але предикат вільності (`status='free'` + `isActiveReservation`)
  > переюзаємо 1-в-1.
- **Потрібно (`Количество`)** = `max(0, Заказано − Остаток)` (показуємо також від'ємні/нульові
  опційно — за фільтром «тільки дефіцит»).

### 3.2. Хелпери (lib)

`lib/manager/needs.ts`:

- `computeNeeds(filters, viewer)` → `{ products: NeedRow[], byAgent: AgentNeedRow[], orders: NeedOrderRow[] }`.
  - `NeedRow`: `{ productId, articleCode, name, unit, ordered, available, needed }` (Товары tab).
  - `AgentNeedRow`: `{ agentUserId|agentName, productId, name, unit, ordered }` (Торгові агенти tab).
  - `NeedOrderRow`: `{ orderId, orderNumber, customerName, city, agentName, status }` (Заказы tab).
- Чисті частини (без I/O) тестовані окремо: `aggregateOrdered(items)`, `aggregateAvailable(lots)`,
  `computeNeeded(ordered, available)` — як `allocateShortage`/`buildOrderTotals`.
- Предикат вільного лота — імпорт `isActiveReservation` з `lib/manager/lot-booking.ts`.

### 3.3. API

`GET /api/v1/manager/needs` (auth + ownership):

- query: `clientId?`, `agentUserId?`, `city?`, `dateFrom?`, `dateTo?`, `deficitOnly?` (default true).
- повертає `{ products, byAgent, orders }` (структура `computeNeeds`).
- manager — лише свої замовлення; admin — усі.

### 3.4. UI

`app/manager/(workstation)/needs/page.tsx` (server) + `_components/needs-client.tsx`:

- 3 вкладки (як 1С): **Замовлення** / **Товари** / **Торгові агенти** (Tabs з `@ltex/ui`).
- Тулбар: пошук + фільтри (клієнт-пікер, агент-select, місто, період) + чекбокс «Тільки дефіцит».
- **Товари** tab — таблиця Артикул / Номенклатура / Од. / Заказано / Остаток / **Потрібно**
  (підсвітка червоним коли `needed > 0`, зеленим коли ≤ 0). Можливість сортування по «Потрібно».
- **Замовлення** tab — Номер / Контрагент / Місто / Статус / Агент (рядки = джерела потреби).
- **Торгові агенти** tab — Агент / Артикул / Номенклатура / Од. / Заказано (group by агент).
- Кнопка експорту CSV (переюзати `lib/reports/csv-export.ts`, UTF-8 BOM + `;`) — корисно для закупівлі.
- Sidebar: додати пункт у `_components/sidebar-links.ts` (SECONDARY_LINKS), напр.
  `{ href: "/manager/needs", label: "Потреби", icon: ClipboardCheck }`.

## 4. Схема БД

- **Міграція НЕ потрібна** для ядра блоку: `Order.isActual` уже є; потреба рахується on-the-fly
  (без таблиці документа). Поля closure/archived уже існують.
- **Опційно (не блокує):** non-unique `@@index([customerId, isActual])` на `Order` — пришвидшує
  перевірку «active_order_exists». Можна відкласти.
- **Partial unique index — НЕ робимо** (див. §2.3c): зламає історичні дані + force-create.

## 5. Фазовий план (worker-розмір)

**Фаза N1 — Актуальність guard (backend + UX).**

- `order-create.ts`: перед create — перевірка active-order; повертати маркер для 409.
- `POST /orders`: 409 `{ code:"active_order_exists", existingOrderId, existingOrderNumber }`; підтримка `?force=true`
  (admin/senior) → у транзакції зняти `isActual` зі старих актуальних цього клієнта.
- close-endpoint (`/orders/[id]/close`): переконатися що ставить `isActual=false` (дописати якщо ні).
- Інваріант `archived ⇒ isActual=false` (перевірити у update/archive шляхах).
- Тести: guard (є актуальне → 409; force → знімає старі; немає → створює).

**Фаза N2 — UX-діалог на створенні.**
