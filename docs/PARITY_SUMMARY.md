# PARITY SUMMARY — зведення аудиту паритетності з 1С (сесія 5.3, Фаза B)

Зведення 6 блочних аудитів (`docs/PARITY_AUDIT_*.md`). Мета — пріоритетна
черга «що добудувати», щоб максимально відтворити функціонал 1С (Шлях B).
Кожен блочний аудит має повну таблицю ✓ / ⚠️ gap / ➕ extra; тут — синтез
наскрізних тем + пріоритетна черга для сесій 5.4+.

Джерела: `PARITY_AUDIT_orders.md`, `_sales.md`, `_cash.md`, `_routes.md`,
`_client.md`, `_prices.md`.

## 1. Наскрізні теми (повторюються у кількох блоках)

### T1. Друк документів (PDF/A4) — НАЙБІЛЬШИЙ наскрізний gap

Відсутній у ВСІХ блоках:

- Замовлення: `Рахунок-замовлення`, `Замовлення покупця` (orders #11)
- Реалізація: `Видаткова Накладна` з підписами «відпустив/отримав» (sales #2,9,10)
- Каса: офіційні `КО-1` / `КО-2` МФУ з ЄДРПОУ (cash #9)
- Маршрутний: `Маршрутний лист` + `…з Оплатою` (per-client борг/Витрати/км) (routes #8)

→ Спільна друк-підсистема (HTML→PDF) обслужила б усі 4 блоки. Великий, але
один раз.

### T2. Зв'язка Реалізація↔Замовлення + «Продано»

- `Sale.orderId` (sales #1) → дозволяє «Продано vs Замовлено» на замовленні
  (orders #8), форму «створити реалізацію із замовлення» (orders #7), і блок
  «Закриття старих замовлень» (давно відкладений).

### T3. Борг / взаєморозрахунки (running register)

- `MgrClient.debt` — статичний знімок. Проведення реалізації/оплати має
  оновлювати борг (sales #6, cash #5,11). Корекція боргу не створює документ
  (cash #1).
- ⚠️ Повний імпорт боргів з `_AccumRg…` — окрема сесія (план 5.3 свідомо
  відклав). Але «перерахунок боргу при проведенні» — окремий менший gap.

### T4. Курс — історичний знімок, не «живий»

- Маршрутний і Каса беруть live `ExchangeRate` замість збереженого на документі
  курсу (routes #12, cash #8). Документи минулого перераховуються невірно.

### T5. Inbound sync 1С→App (поза 5.x — це POST_1C)

- Документи створені прямо в 1С (телефонні замовлення, реалізації бухгалтера)
  не з'являються в нас (orders #12, sales #7). Свідомо поза цією фазою.

## 2. Пріоритетна черга по блоках (топ-гепи)

### Замовлення

1. `ДатаОтгрузки`/`ДатаОплаты` + `НомерТТН` (orders #1,2)
2. Per-line знижка % + `ЦінаПродажіВес` persist (orders #3,4)
3. «Продано» колонка + форма «реалізація із замовлення» (orders #7,8) — зал. T2
4. Друк (orders #11) — T1; Копіювати замовлення (orders #5)

### Реалізація

1. `Sale.orderId` (sales #1) — T2, фундамент
2. Повернення товарів `ВозвратТоваровОтПокупателя` (sales #4) — новий документ
3. Видаткова Накладна друк + підписи + per-row знижка (sales #2,9,10) — T1
4. Backdating `saleDate` (sales #3); гард закритого МЛ (sales #5)

### Каса

1. Структура мульти-валютних ПКО (3 окремі ПКО + безнал=інший документ) — cash #2,3 (критично для майбутнього sync)
2. Корекція боргу → документ + перерахунок (cash #1,5) — T3
3. Курс з маршрутного (cash #8) — T4
4. Друк КО-1/КО-2 (cash #9) — T1; Viber Расход-шаблон (cash #6)

### Маршрутний лист

1. Дії при «Відправлено»: авто-зняття броні + замовлення→неактуальні (routes #2,3) — поведінковий, важливий
2. Витрати (VT7334) + поля `Автомобіль`/`Склад`/`ЦінаЗаКМ` + транспорт-витрати (routes #1,5,6,7)
3. Курс-знімок (routes #12) — T4; per-client баланс (routes #10)
4. Друк маршрутного ± Оплати (routes #8) — T1; контроль завантаж/реалізацій (routes #9)

### Картка клієнта

1. **Історія продаж tab** (stub) — щоденний workflow, простий фікс (client #1)
2. Відсутні поля (email/ЮрФіз/ИНН/ЕДРПОУ/Комментар/ієрархія/мульти-менеджер) — client #2,3,4,5,7 → §0
3. Показ `ДатаПоследнейПокупки`/днів на картці — quick (client #9)
4. Іст. презентацій (stub M1.6) + Viber inline (client #6,8)

### Прайс

1. Per-lot ціна у sync + поля `Ефір` (prices #1,2) → §0
2. UI «таймлайн змін ціни» (prices #3) — дані вже є
3. Per-seller default тип цін + «Рекомендовані» + `ЦеныПоставщиков` toggle (prices #4,5,6)
4. Журнал змін броні (prices #7)

## 3. Рекомендований порядок реалізації (5.4+)

**5.4.0 (коротка):** §0 — additive міграції + розширення імпортера. Розблоковує
коректний дозбір даних і debt-сесію.

**5.4.1 — Картка клієнта (швидкі перемоги):** Історія продаж tab + показ
полів + нові поля у формі. Малий ризик, висока щоденна цінність.

**5.4.2 — T2 Реалізація↔Замовлення:** `Sale.orderId` + «Продано» + форма
«реалізація із замовлення». Розблоковує блок «Закриття старих замовлень».

**5.4.3 — Маршрутний лист поведінка:** дії при відправленні (бронь/актуальність)

- Витрати + курс-знімок.

**5.4.4 — T1 Друк-підсистема:** спільний PDF-движок → Накладна / Рахунок /
КО-1/2 / Маршрутний. Один раз для всіх блоків.

**5.4.5 — T3 Борг + Каса-структура:** перерахунок боргу при проведенні +
мульти-валютна каса + корекція боргу. (Повний debt-імпорт з регістрів — окремо.)

**Решта (P3):** дрібні удосконалення прайсу/каси/маршрутного за §2.

> Порядок — пропозиція orchestrator-а. User затверджує/переставляє перед 5.4.
