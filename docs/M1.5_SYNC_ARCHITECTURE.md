# M1.5 — Sync Backbone Architecture

Backbone двостороннього обміну з 1С: outbound write-back з програми менеджерів у 1С через HTTP-proxy + persistent queue з retry/backoff.

**Status (M1.5):** код shipped, **SOAP-канал ще не налаштований** з 1С-стороны. Усе працює у mock-режимі (`SYNC_MOCK_MODE=true` дефолт). Real SOAP wiring — у M1.5b після того як 1С-розробник реалізує модулі за `docs/1C_SYNC_MODULES_SPEC.md`.

---

## Goals

1. **Decoupling** — Next.js не блокується на повільних SOAP-сесіях
2. **Reliability** — будь-який збій 1С (downtime / network / DB) не втрачає write — retry з exponential backoff до 5 разів
3. **Idempotency** — повторні спроби з тим самим payload не створюють дублікатів у 1С
4. **Observability** — у будь-який момент видно скільки jobs у черзі, останній sentAt, які failed
5. **Mock-first development** — worker / CI можуть тестувати без живого 1С

## Запитання якi ми вирiшили

### Why HTTP-proxy (services/manager-sync), not direct SOAP from Next.js?

- Next.js серверні route-handlers — short-running, не призначенi для довгих SOAP-sessions з connection pooling
- Окремий PM2-процес можна restart-нути без впливу на основний site (`pm2 restart ltex-manager-sync`)
- Proxy maintainit-ить in-memory state (idempotency cache, потенційно SOAP client connection)
- M1.6 inbound polling з 1С теж буде у тому ж сервісі — common SOAP layer

### Why in-memory idempotency cache?

- Single-instance proxy → Map TTL 5min достатньо для retry-dedup
- Multi-instance буде потребувати redis/DB — але це для M1.5b+ якщо горизонтально масштабуємо
- 1С зберігає власний idempotency registry (`СинкЛог`) для дублів що дiйшли — це primary guard

### Why exponential backoff 1m/5m/30m/2h/6h?

Збігається з `lib/email.ts` (S70 email DLQ) — однаковий UX для admin, що дивиться на failed jobs. Усього 5 attempts ≈ 8.5 годин total напіввідкритий retry-window, після чого job → `failed` і потребує manual intervention.

### Why `force-dynamic` cron endpoint?

Будь-який cron в Next.js, що пише в DB, має бути dynamic — щоб не cache-уватись. `runtime = "nodejs"` явно, бо prisma не працює в edge runtime.

## SyncJob lifecycle

```
            ┌────────────────────────┐
PATCH ────► │ pending (just created) │
            │ attempts=0             │
            │ nextAttemptAt=now      │
            └───────────┬────────────┘
                        │
                cron triggers
                        ▼
              ┌──────────────────┐
              │ sendToProxy()    │
              │ + idempotencyKey │
              └────────┬─────────┘
                       │
              ┌────────┴─────────┐
            success            failure
              │                  │
              ▼                  ▼
       ┌──────────┐    ┌───────────────────┐
       │ sent     │    │ attempts < max:   │
       │ sentAt   │    │   status=retrying │
       └──────────┘    │   nextAttemptAt=  │
                       │     now + backoff │
                       │ attempts ≥ max:   │
                       │   status=failed   │
                       │   lastError       │
                       └───────────────────┘
                                 │
                       retry round (next cron)
                                 │
                                 ▼
                  back to sendToProxy() loop
```

## Mock mode

`SYNC_MOCK_MODE=true` — manager-sync приймає request, симулює delay 100-500ms (random), повертає `{ok: true, code1C, mockMode: true}`. Це шлях за замовчуванням до моменту коли 1С-розробник реалізує SOAP-модулі.

`SYNC_MOCK_MODE=false` — manager-sync робить real SOAP-call до `ONEC_SOAP_URL`. У M1.5 цей шлях coded але не тестується (нема живого 1С). У M1.5b / окремiй follow-up — перший real handshake.
