# M1.5.3 BSL Audit — підтвердження Option C з реальним кодом

## TL;DR

**Option C confirmed — з нюансом.** Реальний BSL код у `ОбменАРМ.ОбработатьПакетДанных` уже виконує **точно ту JSON-Структуру роботу** яку Option C проектувала: він приймає Структуру з полями `Контрагенты`/`Заказы`/`ПКО`/`Презентации` та делегує її у `ОбновитьКонтрагентов`/`СоздатьВнутренниеЗаказы`/`СоздатьПКО`. Тобто новий мінімальний `ManagerSync.1cws` — це фактично **wrapper-thin façade** який тільки переформатовує наш JSON у 1С `Структура`, прокидає у `ОбработатьПакетДанных(ПарольВхода, Данные)` і повертає `Сериализовать(Ответ)`. Idempotency built-in через **client-supplied UUID Ссылка** (`Документы.ПолучитьСсылку(UUID).ПолучитьОбъект()` upsert pattern), а не через `НомерВходящегоДокументаЭлектронногоОбмена` як ми припускали у SESSION_M1.5_RESPEC.

**R1 verdict — ПЕРЕОЦІНЮЄТЬСЯ:** `ЗаданиеОбменССайтом` НЕ шле у наш REST `/api/sync/*`. Це **стандартний 1С CommerceML 2.0 / 1C-Bitrix exchange** (`?type=catalog&mode=checkauth`) — а наш Next.js BSL такого endpoint-у не має. URL налаштовується у 1С UI через `Справочник.НастройкиОбменаССайтом.HTTPОбменАдресСайта`. Тобто **outbound `ОбменССайтом` не активний у production** (або налаштований на якийсь інший таргет, ймовірно Bitrix/legacy). Це **відкритий критичний gap** — S66 endpoints не отримують inbound traffic від 1С.

Pivot to Option D **не потрібен** — а навіть шкідливий: розширення `ПроцедурыОбменаССайтом` зламає offline CommerceML protocol contract.

**Класичний CommerceML 2.0** — це стандартний 1С обмін у XML форматі. Перевірка connection (`Module.bsl:324`):

## 3. Q5 — Order idempotency

**Критична reinterpretation:** Я раніше припускав що idempotency через `НомерВходящегоДокументаЭлектронногоОбмена`. **Невірно.**

Реальний механізм — `ОбменАРМ.СоздатьВнутренниеЗаказы` (line 3402, 3414-3415):

```bsl
НовыйОбъект = Документы[ИмяДок].ПолучитьСсылку(СтрокаДанныеЭлемента.Ссылка).ПолучитьОбъект();
...
Если НовыйОбъект = Неопределено Тогда
    Если СтрокаДанныеЭлемента.ПометкаУдаления  Тогда
        Продолжить;
    КонецЕсли;
    НовыйОбъект = Документы[ИмяДок].СоздатьДокумент();
    НовыйОбъект.УстановитьСсылкуНового(Документы[ИмяДок].ПолучитьСсылку(СтрокаДанныеЭлемента.Ссылка));
```

**Idempotency через client-supplied UUID** у полі `Ссылка` структури:

- Якщо клієнт постачає GUID існуючого документа → `ПолучитьСсылку().ПолучитьОбъект()` повертає **existing object** → upsert (overwrite Товары).
- Якщо GUID невідомий 1С → `СоздатьДокумент() + УстановитьСсылкуНового(GUID)` створює **new with this exact GUID** (1С GUIDs можуть бути client-set).

`НомерВходящегоДокументаЭлектронногоОбмена` — це лише timestamp (`= ТекущаяДата()` line 3465), audit-only, не key.

**Implication для ManagerSync write-back:** наш `MgrSyncJob.idempotencyKey` (UUID v4 з `lib/sync/enqueue.ts`) — це і є GUID для 1С. Просто треба передавати його як `Ссылка` field у Структура.

### 6.1 Контрагенти OnWrite chain

Кожен запис Контрагента викликає `ЗарегистрироватьОбъектОбменАндроид` (line 40 `ОбменАндроид`) → реєстрація у `РегистрСведений.РегистрацияОбмена` для повернення цього зміненого об'єкта назад mobile-agent клієнтам.

### 6.2 Recursion-prevention mechanism

`мОбменАндроид = Истина` flag — встановлюється у `ОбновитьКонтрагентов` (lines 4217, 4247). При наступному `ПередЗаписью` (line 92):

```bsl
Если НЕ мОбменАндроид И ЗначениеЗаполнено(НомерТелефона) Тогда
    УникальностьНомераТелефона(НомерТелефона, Отказ);
КонецЕсли;
```

Pre-write phone uniqueness check **пропускається** для inbound exchange — інакше collision flag заглушив би оновлення коли клієнт уже з тим телефоном існує.

**Проте ця flag НЕ пропускає `ОбменАндроид.ЗарегистрироватьОбъектОбменАндроид`** — реєстрація для outbound завжди відбувається. Це означає що **наш inbound update тригерить новий outbound exchange** до інших mobile-agent users з тими ж клієнтами. Це **не recursion** (нескінченний цикл), але це **echo-back amplification** — манагер міняє клієнта через сайт, 1С приймає, реєструє для всіх інших manager-agents у mobile-agent план обміну.

Для ManagerSync це **OK** — це бажана поведінка (other managers should see updates). Але треба бути обережно щоб **не запустити echo-back назад до ManagerSync** через outbound. Захист: ManagerSync inbound endpoint НЕ повинен бути зареєстрований як `ТорговыйАгент` у плані обміну (`Справочник.ТорговыеАгенты`) — або повинен бути виключеним вузлом.

`Контрагенти ObjectModule.bsl:336-378` — **критичний механізм** який раніше пропустив M1.3d audit:

```bsl
Если НЕ ЭтоНовый() И Не ЭтоГруппа И Ссылка.ДатаОновлення > ДатаОновлення Тогда
    СтрСохр = Новый Структура;
    СтрСохр.Вставить("КоличествоДнейОтПоследнейПокупки", КоличествоДнейОтПоследнейПокупки);
    ЗаполнитьЗначенияСвойств(ЭтотОбъект, Ссылка, , "Владелец");
    ЗаполнитьЗначенияСвойств(ЭтотОбъект, СтрСохр);
    ...
```

Якщо incoming `ДатаОновлення` **старіша** за поточну `Ссылка.ДатаОновлення` → **incoming changes ВІДКОЧУЮТЬСЯ**: всі поля копіюються назад з existing object (крім `Владелец`), `Наименование = Ссылка.Наименование`, table parts (`Маршруты`, `Асортимент`, `БанковскиеСчетаДляОплаты`, `АсортиментПрезентацій`) теж відкочуються.

**Implication для M1.3d conflict policy:** наша spec каже "last-write-wins". Це **частково правда** — 1С `ДатаОновлення` server timestamp визначає переможця. Наш Next.js PATCH `/clients/[id]` НЕ передає `dateUpdated` (бо ми не знаємо 1С timestamp) — у write-back через ManagerSync треба:

1. Або **завжди примусово set** `ДатаОновлення = ТекущаяДата()` (заглушити перезапис) — це повертає policy до "client-wins, регардлес".
2. Або зчитати поточний `ДатаОновлення` через окремий read-call перед write — round-trip.
3. Або просто прийняти що при race conditions 1С wins — це консервативний default.

**Recommendation:** option (1) — додати у ManagerSync wrapper примусове `Структура.Вставить("ДатаОновлення", ТекущаяДата())` перед делегуванням у `ОбновитьКонтрагентов`. Це симетрично з тим як `СоздатьВнутренниеЗаказы` робить (line 3465: `НовыйОбъект.ДатаВходящегоДокументаЭлектронногоОбмена = ТекущаяДата()`).

## 7. Strategy verdict

### 7.1 Option C feasibility — confirmed

**Реалізація:**

1. Створити новий `WebServices/ManagerSync/Module.bsl` — wrapper з 2-3 операціями:
   - `ОбновитиКлиента(пароль_або_токен, JSONданих)` — `ПрочитатьJSON()` → Структура → `ОбменАРМ.ОбновитьКонтрагентов(Структура.Контрагенты, Склад, ТаблицаИД, Продавец, МассивОшибок)` → `ЗаписатьJSON(Ответ)`
   - `СтворитиЗамовлення(пароль, JSON)` — те саме але для `Структура.Заказы` → `СоздатьВнутренниеЗаказы(...)`
   - `СтворитиОплату(пароль, JSON)` — `Структура.ПКО` → `СоздатьПКО(...)`
   - Або **одна** операція `ОбробитиПакет(пароль, JSON)` — повністю прозорий wrapper над `ОбробитиПакетДаних(пароль, ХранилищеЗначения)`. Це buy-one-get-all.
2. Auth — `ОбменАндроид.ПолучитьПродавцаПоКодуПартнера(КодПартнера)` (reuse).
3. Anti-recursion — leverage existing `мОбменАндроид` flag (auto-set у ОбновитьКонтрагентов).
4. Idempotency — client-supplied UUID у `Структура.Ссылка`.
5. Conflict policy — overwrite `ДатаОновлення = ТекущаяДата()` у wrapper.

**Blockers — немає.** Все business logic уже існує.

### 7.2 Alternative Option D — extend existing ПроцедурыОбменаССайтом

**Pros:**

- Один scheduled job. Один URL для адміна.
- Reuse `НастройкиОбменаССайтом` config (auth, endpoint, retry).

**Cons (значно сильніші):**

- `ПроцедурыОбменаССайтом` — це **outbound-only** code (push CommerceML 2.0 XML). Розширити її як inbound HTTP endpoint = два протоколи у одному модулі = архітектурний bad smell.
- CommerceML XML mode (`?type=catalog&mode=*`) breaks contract: якщо ми додамо inbound JSON path — будь-який legacy Bitrix consumer (старі тести, інтеграційний партнер) зламається.
- `Обработки.ОбменССайтом` — typed DataProcessor — щоб додати handlers ми мусимо modify exported configuration object (потребує re-export + manual merge у 1С Designer). High risk.
- Контрагенти **не реєструються** у цьому плані обміну (див. §1.5). Розширювати план обміну == migration on `mgr_clients` data == big footprint.

**Висновок: D — pivot не recommended.**

### 7.3 Recommendation

**Option C, single-operation flavor.** Спрощена spec:

```
WebServices/ManagerSync.1cws (новий)
└── Operation: ОбробитиПакет(токен: String, packageJSON: String) → resultJSON: String
    Internal:
    1. Продавець = ОбменАндроид.ПолучитьПродавцаПоКодуПартнера(токен)
       Якщо Неопределено → return JSON {error: "auth"}
    2. Десеріалізуй JSON → Структура
    3. Для кожного field ("Контрагенты"/"Заказы"/"ПКО"/"Презентации") →
       викликати відповідну ОбменАРМ.ОбновитьXxx(...) у tx-bracket
    4. Серіалізуй ТаблицуИД + МассивОшибок → return JSON
```

**ManagerSync proxy у Next.js (services/manager-sync) залишається без змін** — тільки SOAP wrapper уже існує (`src/soap/envelope.ts`). 1C developer створює ManagerSync.1cws → ми wire-up `ONEC_SOAP_URL=https://1c.../ManagerSync.1cws` → працює.

## 9. New findings / surprises

### 9.1 Контрагенти НЕ у `ОбменССайтом` плані обміну

EventSubscription source — тільки 3 catalog types (ХранилищеДополнительнойИнформации/ЕдиницыИзмерения/ХарактеристикиНоменклатуры). Тобто **навіть якщо 1С CommerceML push активний — Контрагенти не йдуть на сайт**. Customer.code1C/телефони/інші реквізити на сайті заповнюються через окремий канал (мабуть один-в-один манулен через 1С-side script, або через інші ad-hoc API).

**Implication для S64 + S70+ etc.:** Customer table на сайті може мати **stale data** відносно 1С Контрагенти. Зокрема, наша M1.3a `MgrClient` model live-snapshots з 1С Контрагенты — це **manual seed** на цей момент, не auto-sync.

### 9.2 1С GUID можуть бути client-set

`УстановитьСсылкуНового(GUID)` (used 9+ разів у ОбменАРМ) дозволяє створити документ з конкретним заданим UUID. Це **architectural assumption** mobile-agent: клієнт генерує UUID v4 локально, шле у 1С, 1С upsert-ить за ним. Це **те саме що Postgres `INSERT ... ON CONFLICT (id) DO UPDATE`** з client-provided UUID.

Наш `MgrSyncJob.idempotencyKey` (UUID v4) — це може бути **прямо** значенням `Структура.Ссылка`. Single mapping, тривіально.

### 9.3 `мОбменАндроид` flag — protects only phone-uniqueness check

`ContragentsModule.ПередЗаписью` line 92 — це **єдина** перевірка яка пропускається коли `мОбменАндроид=Истина`. Усі інші guards (`КонтрольВерсийПерезапису` line 122, `головной контрагент` checks line 137+) **виконуються завжди**. Тобто `мОбменАндроид` — це **дуже вузький** anti-recursion gate, не повний bypass.

**Implication:** наш ManagerSync wrapper повинен **встановлювати**:

- `Структура.Контрагенты[i].мОбменАндроид = Истина` — щоб phone uniqueness не блокував update існуючого клієнта з тим же номером
- `Структура.Контрагенты[i].ДатаОновлення = ТекущаяДата()` — щоб КонтрольВерсийПерезапису не відкатив наші зміни як "stale"

### 9.4 ОбменАРМ — central, not branch

Файл `docs/1c-export-mobile/Central/CommonModules/ОбменАРМ/Ext/Module.bsl` — це **central / головна база** module, не branch. Назва `Central/` у path підтверджує: ОбменАРМ existing module **уже в production central 1С**. Тобто **ми не змінюємо існуючий код** — лише додаємо новий WebService який його викликає. Pure additive change. **Найнижчий risk деплоя для 1С-розробника.**

### 9.5 SOAP serialization — `Сериализовать(ОбъектСериализации) → ХранилищеЗначения`

`MobileExchange.Module.bsl:109-113`:

```bsl
Функция Сериализовать(ОбъектСериализации)
    Возврат Новый ХранилищеЗначения(ОбъектСериализации, Новый СжатиеДанных(9));
КонецФункции
```

Це **бінарний формат 1С `ХранилищеЗначения`** (a 1C-specific compressed serialization), НЕ JSON, НЕ XML. SOAP envelope містить base64-encoded binary blob який можна десеріалізувати тільки усередині 1С runtime.

**Implication для нашого manager-sync proxy:** наш існуючий `src/soap/envelope.ts` (custom SOAP builder/parser) НЕ зможе декодувати response payload з MobileExchange, бо це бінарний 1С-format. Він буде працювати тільки для **plain string** returns.

**Action для Option C ManagerSync:** новий `ManagerSync.1cws` повинен повертати **plain JSON string** (через `ЗаписатьJSON()`), не `ХранилищеЗначения`. Це різниця у contract design — на користь нам, бо парсимо JSON стандартно.
