# Внутрішній корпоративний месенджер L-TEX — план розробки

**Гілка:** `claude/internal-messenger-design-3db8tp`
**Створено:** 2026-07-11
**Замовник:** Тарас (owner). Комунікація — простою мовою (без технічних термінів).

## Мета

Окремий внутрішній чат для співробітників L-TEX (менеджери, склад, експедитори,
керівник) — щось на кшталт Telegram усередині нашої системи. **Не плутати** з
наявним чатом із клієнтами (`/manager/chat` — вхідні Telegram/Viber ззовні). Це
**нове** — спілкування «своїх зі своїми».

## Зафіксовані рішення (2026-07-11)

| №   | Питання                  | Рішення                                                                                                                       |
| --- | ------------------------ | ----------------------------------------------------------------------------------------------------------------------------- |
| 1   | Доступ                   | **Усі з усіма + групи.** Будь-хто пише будь-кому. Owner бачить усе.                                                           |
| 2   | Пристрої                 | **Браузер (ПК + телефон) + push через наш Telegram-бот**, коли написали. Окремого мобільного застосунку поки НЕ робимо.       |
| 3   | Медіа                    | **Фото + документи (PDF/Excel)** одразу. Відео — пізніше за потреби.                                                          |
| 4   | Видалення                | **Зникає у співрозмовників, але зберігається в архіві для owner.** (soft-delete)                                              |
| 5   | Групи                    | Створювати може **будь-хто**.                                                                                                 |
| 6   | Оголошення               | **НЕ робимо** (не користуються).                                                                                              |
| 7   | Посилання на документи   | **Усі**: клієнт, замовлення, реалізація, маршрутний лист, товар/лот + **друковані форми** (накладна, рахунок-фактура, звіти). |
| 8   | «Обговорити» з документа | **Так** — кнопка з картки документа відкриває чат саме про цей документ.                                                      |
| 9   | Користувачів             | **15–20** активних.                                                                                                           |
| 10  | Миттєвість               | **Затримка 2–3 с** (polling) на старті. «Блискавичність» (WebSocket/SSE) — окремий етап.                                      |

## Аналіз: за зразок — Telegram

Беремо: особисті чати + групи, редагування/видалення, відповідь-цитата, реакції,
фото/файли, непрочитані, пошук, закріплені. **Не** беремо: дзвінки, треди-в-тредах,
боти/інтеграції, шифровані секретні чати, платежі/стікер-магазин.

Своє (чого нема в Telegram): пересилання посилань на документи системи + кнопка
«Обговорити» з картки документа + owner бачить усі чати й архів видалених.

## Технічна архітектура (стисло)

- **БД:** нові моделі Prisma — `Conversation`, `ConversationMember`, `Message`,
  `MessageAttachment`, `MessageReaction`, `MessageReceipt` (або поле `readAt` у member).
  Іменування — snake_case `@map`, cuid id, `createdAt/updatedAt`, індекси (як
  `ChatConversation`/`RouteSheet`).
- **Тип чату:** `direct` (1-на-1) або `group`. Для direct — унікальний ключ пари
  користувачів, щоб не плодити дублі.
- **Real-time:** старт — polling (список 10–15 с, відкритий тред 2–3 с, як у
  наявному клієнтському чаті). Пізніше — SSE/WebSocket.
- **Файли:** self-hosted `lib/media/storage.ts` (`MEDIA_ROOT`), роздача через
  `/media/[...path]`. Фото — стиснення sharp; PDF/Excel — as-is з whitelist типів
  і лімітом розміру.
- **Push:** наявний Telegram-бот (`User.telegramChatId`) — DM, коли користувачу
  написали, а він офлайн/не в чаті.
- **UI:** новий розділ `/manager/messenger` (двоколонковий inbox, як клієнтський
  чат) + пункт у меню з лічильником непрочитаних + дзвіночок.
- **Права:** усі бачать усіх (рішення №1). Owner — доступ до всіх розмов + архіву
  видалених. Серверна перевірка членства в розмові на кожен запит.

## Етапи розробки

### Етап 1 — Фундамент + особисті чати (текст) ⬅️ ПОТОЧНИЙ

- Моделі БД + міграція (`Conversation`, `ConversationMember`, `Message`).
- Пошук/старт особистого чату з колегою (список користувачів).
- Список моїх чатів + відкритий тред, надсилання тексту.
- Позначка «прочитано», лічильник непрочитаних.
- Пункт меню `/manager/messenger` + бейдж.

### Етап 2 — Групові чати

- Створення групи (назва, учасники), додавання/видалення учасників.
- Адмін групи (творець), перейменування, вихід із групи.
- Системні повідомлення («X додав Y», «X вийшов»).

### Етап 3 — Редагування / видалення / відповідь

- Редагувати своє повідомлення (позначка «змінено»).
- Видалити (soft-delete: у всіх зникає, owner бачить в архіві).
- Відповідь-цитата на конкретне повідомлення.

### Етап 4 — Медіа й файли

- Завантаження фото (sharp-стиснення) + PDF/Excel.
- Прев'ю зображень, іконка+назва+розмір для файлів, скачування.
- Ліміт розміру, whitelist типів.

### Етап 5 — Реакції

- Emoji-реакції на повідомлення (👍 ✅ 🔥 …), лічильник, хто поставив.

### Етап 6 — Пересилання + посилання на документи системи

- Пересилання повідомлення в інший чат.
- Вставка «картки» документа (клієнт/замовлення/реалізація/маршрут/товар/лот +
  друковані форми) — клікабельне посилання з підписом, відкриває документ.
- Кнопка «Обговорити» на картках документів → створює/відкриває чат по документу.

### Етап 7 — Сповіщення

- Дзвіночок у шапці, лічильники непрочитаних скрізь.
- Push у Telegram-бота, коли написали (з урахуванням «не спамити, якщо в чаті»).

### Етап 8 — Миттєвість, пошук, закріплення

- Real-time доставка (SSE/WebSocket) замість polling.
- Пошук по повідомленнях.
- Закріплені повідомлення в чаті.

## Деплой

Кожен етап — additive міграція + звичайний деплой:
`git pull` → `pnpm install --frozen-lockfile` →
`pnpm --filter @ltex/db exec prisma migrate deploy` → `prisma generate` →
`.\scripts\deploy.ps1 -SkipInstall`.
