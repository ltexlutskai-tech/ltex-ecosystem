# L-TEX OUTBOUND SYNC — BSL артефакти (Етап 2, rework під Molenari OU)

**Що це:** готовий пакет BSL-коду + 1С метаданих для додавання у Central-базу
1С Підприємства. Реалізує JSON-обгортки навколо існуючих ОбменАРМ-процедур,
щоб наш Node.js-проксі (`services/manager-sync`) міг писати у central без
розбирання `ХранилищеЗначения`.

Це **Етап 2** master-плану `docs/1C_INTEGRATION_PLAN.md`.

---

## 1. Чому коротший пакет ніж було заплановано

Постачальник конфігурації LTEX_TEST — **`Molenari OU`** — блокує додавання
НОВИХ кореневих об'єктів конфігурації:

- Кнопка «Зняти з підтримки» сіра (повна заборона).
- Режим сумісності `8.2.13` — не дозволяє Розширення (Extension) як обхід.
- Корінь конфігурації у статусі «Редагується зі збереженням підтримки» —
  це дозволяє лише **редагувати** existing об'єкти постачальника
  (наприклад модуль вже існуючого `WebService.MobileExchange`), але НЕ
  створювати нові `Constant`, `Catalog`, `CommonModule`, `ScheduledJob`.

Тому початковий план Етапу 2 (5 артефактів — `Constant.СинкСистемнийПароль`,
`Catalog.СинкЛог`, `CommonModule.СинкВхідний`, `ScheduledJob.ЧисткаСинкЛогу`,
плюс зміни у `WebService.MobileExchange`) — **скорочено до 1 файлу + XML
diff**: весь код інлайн у модулі веб-сервісу, idempotency перенесена на Node.

## 3. Послідовність розгортання у 1С Конфігураторі

⚠ **Перед розгортанням:** зробити резервну копію центральної бази (DT-файл).

### Крок 1. Згенерувати пароль обміну

PowerShell (одноразово):

```powershell
[Convert]::ToBase64String([byte[]] (1..32 | ForEach-Object { Get-Random -Maximum 256 }))
```

Запам'ятати значення — буде потрібне у Кроці 2 (BSL) і потім у `services/manager-sync/.env`.

### Крок 2. Вставити BSL у `WebService.MobileExchange.Module`

1. У Конфігураторі LTEX_TEST → Web-сервіси → **MobileExchange** → правий клік →
   «Відкрити модуль».
2. Перейти у самий кінець файлу (після `КонецФункции` функції
   `ОтриматиДаніХарактеристики`, орієнтовно line ~163).
3. Скопіювати весь вміст [`Module.bsl.append`](Module.bsl.append) і вставити.
4. **⚠ ОБОВ'ЯЗКОВО:** знайти у вставленому тексті рядок
   ```
   ОчікуванийПароль = "LTEX_SYNC_PASSWORD_PLACEHOLDER";
   ```
   (всередині функції `_LTEX_ПеревіритиПароль`, біля початку вставленого блоку)
   і ЗАМІНИТИ `LTEX_SYNC_PASSWORD_PLACEHOLDER` на ваш пароль з Кроку 1.
5. Зберегти модуль (Ctrl+S).

### Крок 3. Додати 8 нових `<Operation>` декларацій

У Конфігураторі: Web-сервіси → **MobileExchange** → Операції → правий клік → Додати,
для кожної з 8 операцій:

| Назва операції                      | Параметри                                          | Тип значення, що повертається |
| ----------------------------------- | -------------------------------------------------- | ----------------------------- |
| `ОбновитиКлієнтаJSON`               | `ПарольВхода` (xs:string) + `JSONДані` (xs:string) | `xs:string`                   |
| `СтворитиЗамовленняJSON`            | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |
| `СтворитиОплатуJSON`                | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |
| `СтворитиКасовийОрдерJSON`          | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |
| `СтворитиРеалізаціюJSON`            | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |
| `СтворитиМаршрутнийЛистJSON`        | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |
| `ОтриматиДаніЗакриттяЗамовленьJSON` | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |
| `ЗакритиСтаріЗамовленняJSON`        | `ПарольВхода` + `JSONДані`                         | `xs:string`                   |

Властивості для кожної: `Ім'я процедури` = той самий ім'я, `Транзакційно` = Ні.
Параметри: `TransferDirection = In`.

Альтернатива: вставити XML-блок з [`MobileExchange.xml.diff`](MobileExchange.xml.diff)
вручну (8 GUID-ів треба згенерувати — інструкції там же).

### Крок 4. Оновити конфігурацію БД (F7) + перепублікувати веб-сервіс

```
Адміністрування → Публікація на веб-сервері → Перепублікувати
```

Або через `webinst -publish` (Apache 2.4) — у LTEX_TEST вже опублікований
веб-сервіс через Apache.

### Крок 5. Узгодити пароль на Node-стороні

У `services/manager-sync/.env`:

```
ONEC_SOAP_URL=http://localhost/ltex_test/ws/MobileExchange.1cws
ONEC_SOAP_PASSWORD=<той самий пароль що у BSL>
SYNC_MOCK_MODE=false
```

```
pm2 restart ltex-manager-sync --update-env
pm2 save
```

## 4. Перевірка (smoke-тести)

Після перепублікації перевірити, що 8 нових операцій з'явились у WSDL:

```bash
curl -s "https://<host>/<base>/ws/MobileExchange.1cws?wsdl" \
  | grep -oE 'name="(ОбновитиКлієнтаJSON|СтворитиЗамовленняJSON|СтворитиОплатуJSON|СтворитиКасовийОрдерJSON|СтворитиРеалізаціюJSON|СтворитиМаршрутнийЛистJSON|ОтриматиДаніЗакриттяЗамовленьJSON|ЗакритиСтаріЗамовленняJSON)"' \
  | sort -u
```

Очікувано: 8 рядків.

### Smoke-тест 1: ОбновитиКлієнтаJSON (новий клієнт)

```bash
PASSWORD='<пароль з кроку 1>'
URL='https://<host>/<base>/ws/MobileExchange.1cws'
PAYLOAD='{"idempotencyKey":"smoke-001","password":"'$PASSWORD'","data":{"name":"L-TEX SMOKE 001","tradePointName":"ТТ-Smoke","region":"Волинська","city":"Луцьк"}}'

cat > /tmp/req.xml <<EOF
<?xml version="1.0" encoding="utf-8"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <ms:ОбновитиКлієнтаJSON xmlns:ms="http://arm_mobile">
      <ms:ПарольВхода></ms:ПарольВхода>
      <ms:JSONДані>$(echo -n "$PAYLOAD" | sed 's/&/\&amp;/g; s/</\&lt;/g; s/>/\&gt;/g; s/"/\&quot;/g')</ms:JSONДані>
    </ms:ОбновитиКлієнтаJSON>
  </soap:Body>
</soap:Envelope>
EOF

curl -s -X POST "$URL" \
  -H "Content-Type: text/xml; charset=utf-8" \
  -H "SOAPAction: \"http://arm_mobile#MobileExchange:ОбновитиКлієнтаJSON\"" \
  --data @/tmp/req.xml | grep -oE '<[a-z]+:return[^>]*>[^<]*</[a-z]+:return>'
```

Очікувана відповідь (текст всередині `<return>`):

```json
{ "ok": true, "code1C": "000005800", "alreadyProcessed": false, "error": null }
```

⚠ Зверни увагу: `<ms:ПарольВхода>` — порожній (auth перевіряється по `password` ВСЕРЕДИНІ JSONДані). Перший параметр лишений для backward compatibility з рештою операцій MobileExchange.

### Smoke-тест 2: Невірний пароль

Замінити значення `password` у JSON на навмисно неправильне. Очікувано:

```json
{
  "ok": false,
  "code1C": null,
  "alreadyProcessed": false,
  "error": { "code": "auth_failed", "message": "Невірний пароль" }
}
```

### Smoke-тест 3: Невалідний JSON

Передати `JSONДані = "не json"`. Очікувано:

```json
{
  "ok": false,
  "code1C": null,
  "alreadyProcessed": false,
  "error": { "code": "bad_json", "message": "Невірний JSON у JSONДані" }
}
```

### Smoke-тест 4: Idempotency (повторний виклик з тим самим ключем)

⚠ **Зміна по rework:** у попередній версії idempotency була на 1С-стороні
(`Catalog.СинкЛог`) → відповідь могла містити `alreadyProcessed: true`.
**Тепер** ідентичний повторний виклик СТВОРЮЄ ДРУГИЙ документ у 1С
(`alreadyProcessed` завжди = `false`).

Це OK для нормальної роботи: Node-сторона у
`apps/store/lib/sync/queue-processor.ts` фільтрує
`status IN ('pending','retrying')` — `sent` і `failed` НЕ ретраяться.
Подвійний запис можливий лише при дуже специфічному race condition
(наприклад одночасний запит від двох екземплярів cron). Тимчасові
дубль-чернетки менеджер у 1С може видалити вручну.

### Smoke-тест 5: ОтриматиДаніЗакриттяЗамовленьJSON

```bash
PAYLOAD='{"idempotencyKey":"smoke-closures-get-001","password":"'$PASSWORD'","data":{"clientCode1C":"000001"}}'
# ...надсилаємо SOAP-запит з body $PAYLOAD у параметрі ms:JSONДані
```

Очікувано: `{"ok":true, "code1C":null, "alreadyProcessed":false,
"data":{"items":[{"orderUid":"...", "orderNumber":"L-2026-...", "orderDate":"...",
"productUid":"...", "productName":"...", "quantity":<n>, "sum":<n>, "sold":<n>,
"status":"Новий"}]}, "error":null}`.

Якщо у контрагента немає незакритих замовлень — `items: []`.

### Smoke-тест 6: ЗакритиСтаріЗамовленняJSON

**УВАГА:** реально створює документи в 1С — спершу на тестовій базі.

```bash
PAYLOAD='{"idempotencyKey":"smoke-closures-write-001","password":"'$PASSWORD'","data":{"clientCode1C":"000001","items":[{"orderUid":"abcd1234-...","productUid":"efgh5678-...","quantity":75,"price":50,"addToNewOrder":true}]}}'
```

Очікувано: `{"ok":true, "code1C":"L-2026-00567", "alreadyProcessed":false,
"data":{"newOrderUid":"...", "newOrderNumber":"L-2026-00567",
"closedCount":1}, "error":null}`.

## 5. Відомі обмеження поточної реалізації

### 5.1 ОбновитиКлієнтаJSON

- **Телефони (`phones[]`)** не передаються нашою стороною.
- **`region`/`city`** — рядкові імена; TODO резолвити через `НайтиПоНайменуванню`.
- **`primaryAssortmentCode`** — немає очевидного 1С-довідника-аналогу.
- Реквізити `КонтактViber` / `СтатусДіалогу` / `Геолокация` / `СайтURL` /
  `ВідділенняНП` / `МісячнийОбсяг` / `ЛіцензіяДо` / `НаименованиеТТ` — обернено
  у `Попытка...Исключение` (відсутній реквізит → просто не записується).

### 5.2 Документи

- Реквізит `ВнешнийИД` (для прив'язки до нашого cuid) — обернено у `Попытка`;
  якщо відсутній — ігнор.
- `СтворитиКасовийОрдерJSON` реалізує лише ПКО UAH; multi-currency — TODO.
- `СтворитиМаршрутнийЛистJSON` — лише шапка + 4 таб. частини; оплати 1С відновить
  з пов'язаних ПКО/РКО.
- Усі функції створюють документи у статусі **чернетка**
  (`РежимЗаписиДокумента.Запись`); проведення менеджер у 1С робить вручну.

### 5.3 Idempotency

- **Перенесена на Node-сторону.** `apps/store/lib/sync/queue-processor.ts`
  переводить успішні jobs у `status='sent'` і більше не ретраїть; помилки
  пройшли 5 спроб → `status='failed'`. Race-condition при одночасних
  cron-екземплярах теоретично можливе, але `Catalog.СинкЛог` на 1С-стороні
  ми не маємо (постачальник блокує).

### 5.4 Загальні

- Поле `code1C` у відповіді = `Документ.Номер` для документів, `Контрагент.Код`
  для довідника.
- Усі helpers мають префікс `_LTEX_` щоб уникнути конфлікту імен з рештою
  модуля веб-сервісу (а також щоб сигналізувати «не чіпати — наша вставка»).

## 6. Безпека

- **Транспорт:** усі запити з Node.js йдуть через HTTPS (у production) або
  через `localhost` SOAP (на LTEX_TEST за допомогою Apache на тій самій машині).
- **Пароль:**
  - Зберігається hardcoded у функції `_LTEX_ПеревіритиПароль` (заміна
    `LTEX_SYNC_PASSWORD_PLACEHOLDER`) і у `services/manager-sync/.env`
    як `ONEC_SOAP_PASSWORD` (Node).
  - НЕ комітити у git.
  - Ротувати раз на 90 днів або при підозрі на витік. Процедура ротації:
    1. Згенерувати новий пароль (PowerShell, як у Кроці 1).
    2. У Конфігураторі: відкрити модуль веб-сервісу, замінити значення
       `ОчікуванийПароль` на новий, зберегти, F7, перепублікувати.
    3. Негайно (<1 хв) оновити `.env` і `pm2 restart ltex-manager-sync --update-env`.
    4. У вузький проміжок Node буде отримувати `auth_failed`; cron з backoff
       відновить чергу.
- **Журналювання:** усі помилки у BSL обернені у `Попытка...Исключение
ОписаниеОшибки()` — повідомлення повертається у JSON `error.message`.

## 7. Запитання до 1С-розробника (відкриті)

Перед розгортанням рекомендовано звірити:

1. **Імена реквізитів `Catalog.Контрагенты`** у central — чи всі імена з
   `ОбновитиКлієнтаJSON` дійсно існують? (`НаименованиеТТ`,
   `ОбластьНаименование`, `ГородНаименование`, `ВідділенняНП`, `СайтURL`,
   `Геолокация`, `КонтактViber`, `СтатусДіалогу`, `МісячнийОбсяг`,
   `ЛіцензіяДо`).
2. **`ВнешнийИД`** на документах `ЗаказПокупателя` / `РеализацияТоваровУслуг` /
   `ПриходныйКассовыйОрдер` / `РасходныйКассовыйОрдер` / `МаршрутныйЛист` /
   `ПоступленнеБезналичныхДенежныхСредств`?
3. **`Документ.ПоступленнеБезналичныхДенежныхСредств`** — назва точна?
4. **`СтатусыМаршрутногоЛиста`** — імена значень
   (`Составляется` / `Отправлен` / `Завершен`)?
5. **Auto-резолв міста через `НайтиПоНайменуванню(city)`** — безпечно?
