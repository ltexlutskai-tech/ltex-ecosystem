# L-TEX OUTBOUND SYNC — BSL артефакти (Етап 2)

**Що це:** готовий пакет BSL-коду + 1С метаданих для додавання у Central-базу
1С Підприємства. Реалізує JSON-обгортки навколо існуючих ОбменАРМ-процедур,
щоб наш Node.js-проксі (`services/manager-sync`) міг писати у central без
розбирання `ХранилищеЗначения`.

Це **Етап 2** master-плану `docs/1C_INTEGRATION_PLAN.md`.

## 2. Порядок розгортання у 1С Конфігураторі

⚠ **Перед розгортанням:** зробити резервну копію центральної бази (DT-файл).

1. **Створити `Constant.СинкСистемнийПароль`**
   - Імпортувати [`СинкСистемнийПароль.xml`](%D0%A1%D0%B8%D0%BD%D0%BA%D0%A1%D0%B8%D1%81%D1%82%D0%B5%D0%BC%D0%BD%D0%B8%D0%B9%D0%9F%D0%B0%D1%80%D0%BE%D0%BB%D1%8C.xml) АБО створити вручну за інструкцією всередині файлу.
   - Згенерувати нове значення (PowerShell):
     ```powershell
     [Convert]::ToBase64String([byte[]] (1..32 | ForEach-Object { Get-Random -Maximum 256 }))
     ```
   - Виставити константу у режимі Підприємства (Усі функції → Константи).

2. **Створити `Catalog.СинкЛог`**
   - Імпортувати [`СинкЛог.xml`](%D0%A1%D0%B8%D0%BD%D0%BA%D0%9B%D0%BE%D0%B3.xml).
   - Додати 3 реквізити (`СтворенийДокумент`, `ДатаСтворення`, `ТипОперації`) — інструкції всередині XML.

3. **Створити `CommonModule.СинкВхідний`**
   - Загальні → Загальні модулі → Додати → Ім'я: `СинкВхідний`.
   - Властивості: `Server`=Так, `ServerCall`=Так, `Global`=Ні, `Privileged`=Ні.
   - Скопіювати весь вміст [`СинкВхідний.bsl`](%D0%A1%D0%B8%D0%BD%D0%BA%D0%92%D1%85%D1%96%D0%B4%D0%BD%D0%B8%D0%B9.bsl) у модуль.

4. **Додати 6 нових Export-функцій у `MobileExchange/Ext/Module.bsl`**
   - Відкрити web-сервіс `MobileExchange` → Модуль → перейти у кінець (line 163).
   - Вставити вміст [`Module.bsl.append`](Module.bsl.append).

5. **Додати 6 нових `<Operation>` у `MobileExchange.xml`**
   - У Конфігураторі: WebServices → MobileExchange → Операції → правий клік → Додати.
   - Або вручну редагувати XML — інструкції у [`MobileExchange.xml.diff`](MobileExchange.xml.diff).
   - 6 операцій: `ОбновитиКлієнтаJSON`, `СтворитиЗамовленняJSON`, `СтворитиОплатуJSON`,
     `СтворитиКасовийОрдерJSON`, `СтворитиРеалізаціюJSON`, `СтворитиМаршрутнийЛистJSON`.
   - У кожній: 2 параметри (`ПарольВхода` + `JSONДані`, обидва xs:string),
     повертає xs:string.

6. **Створити `ScheduledJob.ЧисткаСинкЛогу`**
   - Імпортувати [`ЧисткаСинкЛогу.xml`](%D0%A7%D0%B8%D1%81%D1%82%D0%BA%D0%B0%D0%A1%D0%B8%D0%BD%D0%BA%D0%9B%D0%BE%D0%B3%D1%83.xml).
   - Налаштувати розклад: щодоби о 03:30.

7. **Оновити конфігурацію БД** (F7).

8. **Перепублікувати веб-сервіс**
   - Адміністрування → Публікація на веб-сервері → Перепублікувати на Apache/IIS.
   - Або: командний рядок 1С `webinst -publish`.

## 3. Перевірка (smoke-тести)

Після перепублікації перевірити, що 6 нових операцій з'явились у WSDL:

```bash
curl -s "https://<1c-host>/<base>/ws/MobileExchange.1cws?wsdl" | grep -oE 'name="(ОбновитиКлієнтаJSON|СтворитиЗамовленняJSON|СтворитиОплатуJSON|СтворитиКасовийОрдерJSON|СтворитиРеалізаціюJSON|СтворитиМаршрутнийЛистJSON)"' | sort -u
```

Очікувано: 6 рядків (по одному на операцію).

### Smoke-тест 1: ОбновитиКлієнтаJSON (новий клієнт)

```bash
PASSWORD='<значення-з-Константа.СинкСистемнийПароль>'
URL='https://<1c-host>/<base>/ws/MobileExchange.1cws'
PAYLOAD='{"idempotencyKey":"smoke-test-001","data":{"name":"L-TEX SMOKE 001","tradePointName":"ТТ-Smoke","region":"Волинська","city":"Луцьк"}}'

cat > /tmp/req.xml <<EOF
<?xml version="1.0" encoding="utf-8"?>
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/">
  <soap:Body>
    <ms:ОбновитиКлієнтаJSON xmlns:ms="http://arm_mobile">
      <ms:ПарольВхода>$PASSWORD</ms:ПарольВхода>
      <ms:JSONДані>$(echo -n "$PAYLOAD" | sed 's/&/\&amp;/g; s/</\&lt;/g; s/>/\&gt;/g; s/"/\&quot;/g')</ms:JSONДані>
    </ms:ОбновитиКлієнтаJSON>
  </soap:Body>
</soap:Envelope>
EOF

curl -s -X POST "$URL" \
  -H "Content-Type: text/xml; charset=utf-8" \
  -H "SOAPAction: \"http://arm_mobile#MobileExchange:ОбновитиКлієнтаJSON\"" \
  --data @/tmp/req.xml | grep -oE '<[a-z]+:return[^>]*>[^<]*</[a-z]+:return>'
```

Очікувана відповідь (текст всередині `<return>`):

```json
{ "ok": true, "code1C": "000005800", "alreadyProcessed": false, "error": null }
```

### Smoke-тест 2: Idempotency (повторний виклик з тим самим ключем)

Повторити той самий запит (той самий `idempotencyKey`). Очікувано:

```json
{ "ok": true, "code1C": "000005800", "alreadyProcessed": true, "error": null }
```

(Той самий `code1C`, але `alreadyProcessed: true` — підтверджує, що
дубль НЕ створено.)

### Smoke-тест 3: Невірний пароль

Замінити `$PASSWORD` на навмисно неправильне значення. Очікувано:

```json
{
  "ok": false,
  "code1C": null,
  "alreadyProcessed": false,
  "error": { "code": "auth_failed", "message": "Невірний пароль" }
}
```

### Smoke-тест 4: Невалідний JSON

Передати `JSONДані = "не json"`. Очікувано:

```json
{
  "ok": false,
  "code1C": null,
  "alreadyProcessed": false,
  "error": { "code": "bad_json", "message": "Невірний JSON у JSONДані" }
}
```

## 4. Сумісність з Node-стороною (відомі TODO)

Існуючий код у `services/manager-sync/src/soap/client.ts` шле SOAP-запити з:

- **Імена операцій** БЕЗ суфікса `JSON`: `ОбновитиКлієнта`, `СтворитиЗамовлення`,
  `СтворитиОплату`, `СтворитиРеалізацію`, `СоздатьПКО`, `СтворитиМаршрутнийЛист`.
- **3 параметри** SOAP: `ПарольВхода` + `IdempotencyKey` + `ПакетДанних`
  (envelope budується у `services/manager-sync/src/soap/envelope.ts`).

**Розбіжності з цим BSL-кодом:**

| Аспект                | Поточний Node-код                                                                                                       | Цей BSL-код                                                                                                                                               |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Імена операцій        | `ОбновитиКлієнта`, `СтворитиЗамовлення`, `СтворитиОплату`, `СтворитиРеалізацію`, `СоздатьПКО`, `СтворитиМаршрутнийЛист` | `ОбновитиКлієнтаJSON`, `СтворитиЗамовленняJSON`, `СтворитиОплатуJSON`, `СтворитиКасовийОрдерJSON`, `СтворитиРеалізаціюJSON`, `СтворитиМаршрутнийЛистJSON` |
| К-сть SOAP-параметрів | 3 (`ПарольВхода` + `IdempotencyKey` + `ПакетДанних`)                                                                    | 2 (`ПарольВхода` + `JSONДані`, де `JSONДані = {"idempotencyKey","data"}`)                                                                                 |

**Як вирішити (вибір залишається orchestrator):**

### Опція А (рекомендована, простіше): підлаштувати Node під BSL

У наступному раунді скоригувати `services/manager-sync/src/soap/`:

1. Перейменувати operation names у `client.ts` (додати суфікс `JSON`).
2. Скоригувати `envelope.ts::buildSoapEnvelope` — замість 3-х елементів-параметрів
   (`<ms:ПарольВхода/><ms:IdempotencyKey/><ms:ПакетДанних/>`) генерувати 2
   (`<ms:ПарольВхода/><ms:JSONДані/>`), де `JSONДані` = `JSON.stringify({idempotencyKey, data: payload})`.
3. Аналогічно перейменувати у `buildSoapAction()`.
4. Оновити тести (`soap/envelope.test.ts`, `soap/client.test.ts`,
   `routes/sync-*.test.ts` — вони перевіряють operation names).

Це 1 worker-сесія ~30хв. Не потребує змін у BSL.

### Опція Б: підлаштувати BSL під Node

Зберегти 3-параметровий контракт + старі імена операцій. Тоді у цьому
BSL-коді треба:

1. Замінити сигнатуру кожної функції з `(ПарольВхода, JSONДані)` на
   `(ПарольВхода, IdempotencyKey, ПакетДанних)`.
2. Прибрати парсинг `Запит.idempotencyKey` + `Запит.data` — використати
   `IdempotencyKey` напряму + `ПарситиJSON(ПакетДанних)` як data.
3. Прибрати суфікси `JSON` з імен у XML + Module.bsl.

Це теж 1 сесія, але більше дотику у файлах.

### Опція В (status quo): нічого не міняти, BSL — як референс на майбутнє

Залишити цей BSL-код як артефакт у репозиторії, фактично не розгортати у
production до моменту узгодження. Тоді `services/manager-sync` продовжує
працювати у `SYNC_MOCK_MODE=true` бек-енд.

  TODO: звірити з 1С-розробником, які реквізити реально є у central

### 5.2 СтворитиЗамовленняJSON / СтворитиРеалізаціюJSON / СтворитиКасовийОрдерJSON / СтворитиМаршрутнийЛистJSON

### 5.3 Загальні

- Поле `code1C` у відповіді = `Документ.Номер` для документів, `Контрагент.Код` для
  довідника. Це обумовлено тим, що Номер документа — людиночитаний (наприклад

  > 50k записів додати індекс через `InformationRegister` замість Catalog;
  > поки що `Catalog` достатньо.

## 6. Безпека

- **Транспорт:** усі запити з Node.js йдуть **тільки** через HTTPS. У публікації
  веб-сервісу заборонити HTTP-варіант.
- **Пароль:**
  - Зберігається у `Константа.СинкСистемнийПароль` (1С) і в `services/manager-sync/.env`
    як `ONEC_SOAP_PASSWORD` (Node).
  - НЕ комітити у git.
  - Ротувати раз на 90 днів або при підозрі на витік.
  - При ротації: спочатку оновити константу у 1С, потім негайно (<1хв) — `.env`
    - `pm2 restart ltex-manager-sync --update-env`. У вузький проміжок Node буде
      отримувати 401, що OK (заглушка ретраїв у `process-sync-queue` опрацює).
- **Idempotency-захист:** кожен SOAP-запит має унікальний `idempotencyKey`
  (UUID v4, генерується нашою стороною у `enqueue.ts`). У `СинкЛог` записується
  на 7 днів. Це захищає від подвійних викликів при retry-storms.
- **Антишахрайство:** `СинкВхідний.ПеревіритиПароль` робить string-compare
  (case-sensitive, після `СокрЛП`) з константою. Якщо константа пуста, повертає
  Ложь (нікому не довіряємо).
- **Журналювання:** усі помилки записуються через `ЗаписьЖурналаРегистрации`
  з рівнем "Помилка" та повідомленням `ОписаниеОшибки()`. Перевіряти у
  Адміністрування → Журнал реєстрації.
