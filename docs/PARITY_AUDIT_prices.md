## Top gaps, prioritized

### 🔴 Пріоритет 1 — Критично для повноти даних при sync

1. **Per-lot ціна у регістрі (`_InfoRg5225._Fld5228RRef`)** — у 1С ціна може бути прив'язана до конкретної характеристики (`ХарактеристикиНоменклатуры`). При імпорті/синхронізації per-lot ціни з регістру треба класти в `Lot.priceEur`, а не в `Price` — адаптер імпорту це враховує, але в UI при синку через `/api/sync/prices` ця логіка відсутня (endpoint приймає тільки `productId`-рівень).

2. **Поля `Ефір` / `ЕфірНаДоставку` (`_Fld7729`, `_Fld7730`)** — у `ХарактеристикиНоменклатуры` є два boolean-поля: «вийшов в ефір» і «ефір на доставку». Вони присутні в MSSQL-схемі але відсутні у нашій `Lot` моделі. При importer/sync ці поля ігнорувались, але вони потрібні для відображення в менеджерці (які лоти вже «в ефірі»).

### 🟡 Пріоритет 2 — Функціональний gap менеджерки

3. **Повна історія цін у UI** — дані є в таблиці `Price` (є `validFrom`/`validTo`), але картка товару `/manager/prices/[id]` показує тільки поточні ціни. Відсутній UI «Таймлайн змін ціни» як в 1С. Відносно просто додати — запит `Price ORDER BY validFrom DESC` для товару.

4. **Per-seller прапор «Рекомендовані товари»** — 1С `Продавец.Рекомендации` містить список рекомендованих товарів для конкретного продавця (`ТзНоменклатура.Рекомендация = Истина`). У нас ця per-agent персоналізація відсутня. Фільтр «Рекомендовані» у прайсі 1С — часто використовуваний менеджерами.

5. **Default тип цін продажу per-manager (`Продавец.ТипЦенПродажи`)** — у 1С кожен торговий агент має свій default тип цін (зберігається у `Константах`/`ТорговыеАгенты`). Наш менеджер завжди бачить `wholesale`/`akciya` незалежно від менеджера. Потрібне поле `User.defaultPriceTypeId` FK → `MgrPriceType`.

6. **`ЦеныПоставщиков` per-manager toggle** — у 1С ціни постачальника видно тільки якщо `Продавец.ЦеныПоставщиков = true`. У нас `product-card.ts::isSupplierPriceType()` робить prefix-match і показує завжди. Потрібний per-user toggle (можна як `User` bool поле).

7. **Журнал змін броні (`ИзмененияБрониХарактеристики`)** — 1С веде регістр змін броні (хто, коли, на якого клієнта). У нас є детекція протермінованої броні і нагадування, але немає хронологічного журналу. При роботі з конфліктами броні журнал корисний.

### 🟢 Пріоритет 3 — Дрібні удосконалення

8. **Поріг заборгованості (`ПорогЗадолженостиEUR`)** — 1С має константу порогу для підсвічування боргу клієнта. У нас борг відображається у картці клієнта, але без порогового підсвічування на рівні прайсу.

9. **`ЦенаПродажиВес` vs `Ціна`** — у 1С розрізняють `ЦенаПродажиВес` (ціна за вагу для мішків) і `Цена` (базова ціна одиниці). При імпорті і синку важливо правильно обрати яке поле класти в `Price.amount` / `Lot.priceEur`. В нашому `POST /api/sync/prices` це вже враховано, але варто задокументувати в коді.

10. **`ПосиланняНаСайт` у Номенклатурі** — 1С зберігає URL-сторінки товару на сайті. Наш admin і 1С можуть розійтись (1С показуватиме старий URL). При активній синхронізації бажано оновлювати це поле зворотнім записом (наш slug → до 1С). Але для MVP — некритично.

11. **Чат-повідомлення при новій номенклатурі** — у 1С `Номенклатура.ObjectModule.bsl` при `ПриЗаписи` нова номенклатура надсилає повідомлення в `ГруппыЧата.действиеНоваяНоменклатура`. У нас такий механізм відсутній (наша нова номенклатура — через `Receiving`, а не через прямий Create в прайсі).

12. **`_Reference105._Fld6454/_Fld6455/_Fld6456` у `ТипыЦен`** — `_Reference105` має 15 колонок (крім `_Code`/`_Description` є ще `_Fld6454..6460`). Ймовірно це флаги та посилання (округлення, валюта за замовчуванням, группа цін). При імпорті типів цін ми беремо тільки code+label — цього достатньо для MgrPriceType, але деякі відтінки поведінки типів цін можуть бути втрачені.
