# Стратегія повної заміни 1С на L-TEX-app

**Створено:** 2026-05-29 (Plan-агент після виявлення Molenari OU lockdown)
**Контекст:** User розглядає повний відмов від 1С на користь власної self-hosted системи.

---

## 1. Executive Summary

### Стратегічна рекомендація: **Поетапна міграція протягом 6–9 місяців**

L-TEX-app уже покриває **~65% операційних потреб** 1С (5 з 7 ключових щоденних блоків: Замовлення, Реалізація, Оплати/Каса, Маршрутні листи, Прайс готові + 80% Картки клієнта + Нагадування + Чат-inbox). Залишилось дописати **3 склад-логістичні блоки** (Прийняття, Перепаковки, Закупки), **2 звітних** (Звіти, Платіжні доручення) та **інтегрувати Checkbox** для ПРРО — це ~4–5 місяців розробки.

**Ключове відкриття:** user НЕ використовує 1С для бухгалтерії, податкової звітності чи бухгалтерських проводок (Дт/Кт). 1С для нього = **операційна складсько-торгова система** з 18 звітами + 3 обробками. Це **знімає 80% типового страху міграції з 1С** — нам не треба відтворювати План рахунків, регламентовану звітність чи податкові накладні.

### Орієнтовний час

| Етап                                            | Тривалість  | Зміст                                                                                   |
| ----------------------------------------------- | ----------- | --------------------------------------------------------------------------------------- |
| **Фаза 0 — закриття BSL Molenari** (паралельно) | 1–2 тижні   | Звернення до Molenari OU або переїзд на варіант B (mock + паралельна робота)            |
| **Фаза 1 — склад/логістика блоки**              | 2–3 місяці  | Прийняття, Перепаковки, Заказы поставщикам, Зміна стану мішка, Потребности              |
| **Фаза 2 — звіти + Checkbox**                   | 2 місяці    | 18 звітів (з пріоритетом топ-8), інтеграція з Checkbox API для ПРРО, платіжні доручення |
| **Фаза 3 — міграція даних з 1С**                | 1 місяць    | Дамп + експорт усіх активних документів і довідників з 1С у L-TEX                       |
| **Фаза 4 — паралельна робота і cut-over**       | 1–2 місяці  | Менеджери працюють у L-TEX, 1С — read-only; потім вимкнути 1С                           |
| **Фаза 5 (опційно) — бухгалтерія**              | за потребою | Якщо колись треба: інтеграція з MeDoc/Парус/окремий аутсорс                             |
| **Разом**                                       | **6–9 міс** | До повного позбавлення від 1С                                                           |

### Ризики (топ-3)

1. **Molenari OU блокує BSL** — без розблокування sync обмежений mock-mode (user звернеться до постачальника, 30–60 хв роботи, $50–150).
2. **18 унікальних звітів** — кожен унікальний, копіювати «один-в-один» довго. Mitigation: пріоритезувати топ-8 (95% щоденного використання), інші — за потребою.
3. **Експорт активних залишків і дебіторської заборгованості з 1С** — без BSL модифікацій складно, треба парсити ХранилищеЗначения або писати JSON-обгортки (4 з 6 вже написані у M3.1, треба ще 2).

## 5. Checkbox інтеграція (детально)

### Що це

**Checkbox** (https://checkbox.ua) — український SaaS-ПРРО (програмний реєстратор розрахункових операцій), легально замінює апаратний касовий апарат для ФОП. Має REST API, інтегрується через JWT.

### Як працює зараз у 1С user-а

При створенні Реалізації менеджер додає **ТТН Нової Пошти** → 1С автоматично викликає Checkbox API → створюється **post-payment проект чеку** (чек який буде сплачено при отриманні товару — модель «накладена післяплата»). Клієнт сплачує НП → НП повідомляє Checkbox → чек проводиться → надсилається клієнту електронний фіскальний чек.

### Що треба у L-TEX

**Сценарій (точка інтеграції):** `/manager/sales/[id]` сторінка Реалізації

1. Менеджер створює Реалізацію, ставить статус «Відправлено».
2. Через кілька годин (коли НП реєструє накладну) — менеджер вводить **№ ТТН НП** у спеціальне поле.
3. UI кнопка **«Створити проект чеку у Checkbox»**.
4. Backend (`/api/v1/manager/checkbox/create-project`):
   - Auth до Checkbox (JWT з ключем з env `CHECKBOX_API_KEY`)
   - POST до Checkbox `/v1/receipts` з payload: товари, суми, ТТН НП, тип=post-payment
   - Збереження `ReceiptProject` запису з `checkboxProjectId`
5. Checkbox webhook → `/api/v1/manager/checkbox/webhook` → оновлення статусу `ReceiptProject.checkboxStatus` коли клієнт сплатив
6. QR-код фіскального чеку показується у UI (опційно — emailing клієнту)

**Estimate:** M (2 тижні) — типова REST інтеграція з webhook + 1 нова модель + 1 UI блок

**User дії перед стартом:**

- Зареєструвати акаунт у https://checkbox.ua (якщо ще не зробив)
- Згенерувати API-токен у кабінеті
- Передати `CHECKBOX_LICENSE_KEY` + `CHECKBOX_CASH_REGISTER_KEY` у environment

**Рекомендація:** не починати самостійно реалізацію без актуальної документації Checkbox (їх API еволюціонує). Перед роботою — orchestrator робить **WebFetch https://docs.checkbox.ua/** щоб взяти найсвіжіший REST contract.

### 6.2. Як експортувати

**Опція A (preferred): через нові JSON-обгортки BSL**

- Дописати 2 нові операції до `Module.bsl.append`: `ОтриматиЗалишкиJSON` + `ОтриматиБоргиJSON`
- Інші дані (Замовлення/Контрагенти/Ціни) — вже покриті готовими M3.1-M3.4 операціями (треба зняти Molenari блок)
- На стороні Node — одноразовий скрипт `scripts/migrate-from-1c.ts` що: запускає послідовно всі pull-операції з кросором, нормалізує до Prisma format, робить bulk insert/update

**Опція B (fallback, якщо BSL не модифікується через Molenari): експорт у файли**

- 1С Конфігуратор → Сервис → Выгрузка данных (стандартний механізм)
- Експорт у XML
- Парсер на Node читає XML → нормалізує → bulk insert у Prisma

**Опція C (last resort): ручне внесення менеджерами**

- Активні замовлення (~300) — менеджери знають своїх клієнтів, можуть створити заново в `/manager/orders` за тиждень-два
- Залишки складу — скан мішків, які фізично є; під час «прийняття» в нову систему

### 6.3. Open questions для user-а

1. **Чи готовий Molenari OU розблокувати BSL?** Це визначає Опцію A vs B/C для експорту.
2. **Які саме дані критичні** — тільки активні замовлення + борги, чи треба історію 5 років? (Рекомендація: тільки активне + останніх 3 міс історії; інше — архів у 1С read-only сервер на 1–2 роки)

## 9. Альтернативи перед повною міграцією

### Опція A. **Попросити Molenari OU зняти блокування — РЕКОМЕНДОВАНО**

- Звернутись на http://1c.ua/v8/RegionalSolution за зняттям обмежень з WebService.MobileExchange
- Або попросити їх самих зробити BSL вставку (готових 921 LOC) — 30–60 хв їхньої роботи
- **Очікувана вартість:** $50–150 (одноразова)
- **Очікуваний час:** 1–7 днів
- **Що даєш цим:** повну sync 1С↔L-TEX, можливість плавного cut-over без даунтайму, можливість експорту усіх даних через BSL JSON-обгортки

### Опція B. **Жити у mock-mode + паралельна робота (нинішня)**

- Менеджери дублюють дані: створюють замовлення у L-TEX **і** в 1С
- **Проблема:** double-entry, помилки, навантаження на менеджерів
- **Коли годиться:** тільки коротко (1–2 міс) поки чекаємо Molenari

### Опція C. **Поетапна міграція без обміну з 1С**

- Менеджери: **у L-TEX**, не у 1С
- Бухгалтер: **у 1С**, не у L-TEX (читає звіти, експортовані з L-TEX як CSV)
- 1С продовжує існувати, але як **read-only archive + bookkeeping**
- **Перевага:** не залежить від Molenari; cut-over швидкий
- **Недолік:** треба домовитись з бухгалтером про новий workflow (отримання даних з L-TEX замість прямого 1С)

### Опція D. **Status quo + mock-mode**

- Нічого не міняти; L-TEX-app залишається секондарним для менеджерів
- 1С залишається primary
- **Перевага:** нульовий ризик
- **Недолік:** платимо за два інструменти і блокування Molenari далі

### Опція E. **Купити стандартну УТ від 1С Company**

- Повна міграція з Molenari-кастомної на офіційну УТ 11
- **Вартість:** $500–2000 за ліцензію + кастомізація
- **Час:** 3–6 місяців
- **Недолік:** залишаємось на 1С (Linux server потрібен з конфігом, support складний); не відповідає стратегічній цілі позбутись 1С

## Підсумок

L-TEX вже на **65% шляху** до повної заміни 1С. Залишилось 4–5 місяців роботи (Фази 1–3) щоб отримати повноцінну self-hosted систему без 1С. Бухгалтерія та податкова — НЕ блокери (user не використовує 1С для них). Ключове рішення зараз — **звернутись до Molenari OU** для розблокування BSL, бо інакше міграція даних ускладнюється.

Стратегія **низького ризику + поетапна**: ніколи не залишаємо user-а без робочого інструменту. 1С Server залишається до самого cut-over, потім ще 6 місяців у read-only режимі як архів.
