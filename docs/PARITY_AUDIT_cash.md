## Top gaps, prioritized

1. **КорректировкаДолга (ОбменАРМ рядки ~4940):** `MgrCashOrder.debtCorrection` зберігається але не синхронізується з `Документ.КорректировкаДолга (СписаниеЗадолженності)` у 1С. Без цього корекція боргу без реалізації не відбивається в облікові 1С.

2. **Окремий ПКО per валюта (ОбменАРМ.СоздатьПКО цикл по валютах):** ОбменАРМ створює до 3 окремих ПКО (UAH, EUR, USD) + окреме `ПлатежноеПоручениеВходящее` для безналу. Наш 1 `MgrCashOrder` record з 4 amount-полями не відображає цієї структури. При sync-back у 1С — всі УІДГРН / ПКО_УІД / УІДUSD посилатимуться на один запис замість 3.

3. **Безнал UAH → ПлатежноеПоручениеВходящее:** Безготівкова оплата (`amountUahCashless`) у 1С — це НЕ ПКО, а окремий тип документа `Документ.ПлатежноеПоручениеВходящее`. BSL TODO зафіксовано у CLAUDE.md, але структурно невирішено.

4. **НеВідображатиВДодатку guard (Form/Module.bsl рядки ~280):** 1С приховує рахунки з `НеВідображатиВДодатку = true` з picker-а при Приход. `MgrBankAccount.hiddenInApp` поле існує, але потрібно перевірити API `/api/v1/manager/dictionaries` чи фільтрує за цим флагом у режимі income (у Расход — всі рахунки мають бути видимими).

5. **ВключитиСумуБоргу (Form/Module.bsl рядки ~180):** 1С формула: `СуммаКОплате = Борг + СумаРеалізації - СуммаОплаченноПоМЛ`. Ми лише додаємо `clientDebtEur` до довільного `baseSumToPay`, ігноруючи вже здійснені оплати по маршрутному листу.

6. **Viber Расход-шаблон (Form/Module.bsl рядки ~390–420):** 1С має окремий текст для: (а) безнал-Приход «Реквізити оплати: {опис рахунку}\nСума: {грн}»; (б) Расход «Оплата на витрати: {стаття}\nСума: {канали}». `buildPaymentReceiptText()` покриває лише Приход-варіант.

7. **ВидыОперацийПКО / РКО (7+6 типів):** ПКО ObjectModule.bsl підтримує `РозничнаяВыручка`, `ОплатаПоставщику`, `ВозвратПодотчетника`, `КредитыЗаймы` тощо. Наша сторона — лише бінарний `income/expense`. При розширенні каси (видача підзвіт, внесок у банк) — бракуватиме семантики.

8. **Курс з МаршрутныйЛист (`Общий.ЗаполнитьКурсПоМаршрутному`):** ПКО при проведенні тягне курс із МаршрутногоЛиста. Ми завжди беремо з `ExchangeRate` таблиці. При розходженні курсів (лист оформлено з іншим курсом) — різниця не компенсується.

9. **КО-1 / КО-2 print forms:** Офіційні форми МФУ України (ПКО = КО-1 2018, РКО = КО-2) з ЄДРПОУ, реквізитами організації, підписами касира та бухгалтера. Відсутні — можуть знадобитися при перевірці або переході на повну касову дисципліну.

10. **НомерОрдера per Касса:** 1С нумерує у межах рік/організація/Касса (окремо для UAH-каси, EUR-каси). `MgrCashOrder.docNumber` — глобальний лічильник. При появі Довідника «Кассы» потрібна перепартиція.

11. **ДоговорКонтрагента → Сделка у РасшифровкаПлатежа:** ПКО має рядки розшифровки платежу з прив'язкою до Договору та Сделки. Наша сторона зберігає лише `saleId` (без концепції Договору). Для повноцінного взаєморозрахунку важливо при активації 1С sync.

12. **НДС (ставка/сума на РасшифровкаПлатежа):** ПКО ObjectModule.bsl веде СтавкаНДС/СуммаНДС per рядок та пише у регістр `КнигаДоходовРасходов`. Наша сторона VAT повністю відсутній. Актуально якщо бізнес перейде на спрощену з ПДВ або загальну систему.
