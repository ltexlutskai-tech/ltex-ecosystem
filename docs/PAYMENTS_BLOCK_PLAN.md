## 0. Затверджені рішення user (10) — враховано в плані

1. **Здача — ручна в 3 валютах** (UAH/EUR/USD), як 1С: менеджер вводить решту руками, система лише
   **підказує** рекомендовані суми (`СдачаXxxРек`). (Зараз Stage-4 рахує здачу авто в одній валюті — переробляємо.)
2. **Банк. рахунки — довідник** `MgrBankAccount` (замість вільного рядка `bankAccount`).
3. **«Знижка на залишок» — відтворити**; механізм — **по рядку реалізації** (зменшує найдорожчий рядок
   `SaleItem`, ставить прапор знижки, перераховує суми); для оплати **без реалізації** — через
   `debtCorrection` (коректировка боргу) на ордері. Доступна лише коли `|залишок| ≤ поріг`.
4. **Кілька оплат на одну реалізацію — дозволено.** Анти-дубль `КонтрольОплати` 1С **НЕ відтворюємо**
   (кожне «Сформувати» створює нові ордери; борг рахується агрегацією).
5. **Стаття руху коштів — довідник** `MgrCashFlowArticle` (замість вільного рядка `cashFlowArticle`).
6. **Окремий список `/manager/payments`** — повний (Дата/Номер/Вид/Клієнт/Стаття/Сума EUR/Рахунок,
   фільтр архівних, пошук) + кнопка «Створити».
7. **Оплата без реалізації — дозволено** (погашення боргу клієнта): обираємо клієнта, `СуммаКОплате`
   за замовч. = борг клієнта.
8. **Кнопка «Вайбер» — відтворити** (текст реквізитів безналу / факт-оплати / решти + «Поділитися» через
   наявний `ShareSheet`).
9. **Довідники — порожні поки** (наповняться вручну або при обміні з 1С) + **мінімальний адмін-UI**
   керування (додати/редагувати/сховати), щоб безнал-шлях був придатний до використання.
10. **Вид руху Приход/Расход — обидва у формі** (Расход → стаття обов'язкова), як 1С. Для Приход вибір
    прихованих банк. рахунків заблоковано (гард «не можна при приході»).

**Додаткові (успадковані з Реалізації, без окремого питання):**

- Курси EUR/USD — **знімок** на ордері при створенні (нові поля `rateEur`/`rateUsd`); базова валюта зведення — EUR.
- `ПроверкаПолнойОплаты` (пошта без наложки + борг>0) — **м'яке попередження**, не жорсткий блок (як у Реалізації).
- Поріг знижки `ПорогЗадолженостиEUR` — код-константа з дефолтом **5 €** (легко винести в налаштування пізніше).
- Реальний 1С BSL (`СоздатьПКО`) — **не пишемо зараз** (на загальному етапі обмінів); робимо mock-каркас sync, як у Реалізації Етап 5.

## 1. База (моделі + міграція `20260527_mgr_payments`, additive)

### Розширити `MgrCashOrder` (`mgr_cash_orders`)

Нові поля (усі nullable / default — additive, Stage-4 дані не ламаються):

- `customerId?` (FK `Customer`) — Контрагент (для оплати без реалізації + денормалізація). Резолв через `code1C`.
- `bankAccountId?` (FK `MgrBankAccount`) — замінює вільний `bankAccount` (старий рядок лишаємо як deprecated, не використовуємо).
- `cashFlowArticleId?` (FK `MgrCashFlowArticle`) — замінює вільний `cashFlowArticle`.
- `rateEur` / `rateUsd` (Float, default 0) — **знімок курсів** (КурсEUR/КурсUSD).
- `documentSumEur` (Float, default 0) — `СуммаДокумента` (зведена сума в EUR; для колонки списку «Сума» та обміну).
- `debtCorrection` (Float, default 0) — `КоректировкаДолга` (списання залишку без реалізації).
- `correctionUid?` (String) — `УИДКоректировки`.
- `docNumber` (Int, autoincrement) — Номер документа.
- `archived` (Boolean, default false) — `Архив` (фільтр списку + сіре підсвічування).
- `uidUah?` / `uidEur?` / `uidUsd?` (String) — мультивалютні бізнес-ключі обміну (`УИДГРН`/`ПКО_УИД`/`УИДUSD`).

Здача в 3 валютах **не потребує нових полів**: ордер-розхід (`type="expense"`, `changeForId`→прихідний)
зберігає суми у наявних `amountUah`/`amountEur`/`amountUsd`. (Stage-4 `changeCurrency` лишаємо nullable, не використовуємо.)

### Новий довідник `MgrBankAccount` (`mgr_bank_accounts`)

- `id`, `code1C?` @unique, `name` (Description — IBAN/№ рахунку/назва ФОП), `description?` (Опис), `hiddenInApp` (Boolean, `НеВідображатиВДодатку`), `archived` (Boolean), `createdAt`/`updatedAt`.
- _(привʼязку рахунків до конкретного контрагента — як майбутнє уточнення; зараз плоский довідник + «показати всі».)_

### Новий довідник `MgrCashFlowArticle` (`mgr_cash_flow_articles`)

- `id`, `code1C?` @unique, `code?` (числовий код), `name` (Description), `parentId?` (self-ref, 2 рівні), `archived` (Boolean), `createdAt`/`updatedAt`.

### Sync

- enum `SyncEntityType` += `cash_order` (additive).

**Міграція:** одна idempotent additive (`ADD COLUMN IF NOT EXISTS` / `CREATE TABLE IF NOT EXISTS` /
`DO $$ … duplicate_object` для enum), за зразком `20260526_mgr_cash_orders`. **⚠️ Перед deploy: `prisma migrate deploy`.**

## 3. Етапи розробки

### Етап 1 — База + довідники + список «Оплати»

**Менеджер бачить:** замість заглушки `/manager/payments` — список усіх касових ордерів (Дата/Номер/Вид/Клієнт/
Стаття/Сума EUR/Рахунок), фільтр «архівні» (за замовч. сховані, сірий фон архіву), пошук, пагінація, кнопка «Створити».

- Міграція `20260527_mgr_payments` (поля `MgrCashOrder` + 2 довідники + enum).
- `GET /api/v1/manager/cash-orders` — список (search, type, archived, date range, ownership через `customer.code1C`).
- `dictionaries` endpoint += `bankAccounts` + `cashFlowArticles`.
- Сторінка `/manager/payments` + тулбар + таблиця + filter-state (URL) + пагінація.
- Мінімальний адмін-UI довідників (`/manager/bank-accounts`, `/manager/cash-flow-articles` або в Налаштуваннях):
  список + додати/редагувати/сховати/архівувати (admin-guard).
- Сервіс `getPaymentSummary(saleId)` (порт `ПолучитьДанныеПоОплате`): Приход−Расход у EUR, статус Борг/Передплата,
  розбивка по валютах, `codAmountUah` (наложка). Використовує лише ордери з тієї ж підстави.
- Dashboard tile «Оплати» (опц.) — лічильник за сесію.

### Етап 2 — Payment-форма (порт «Оплата») + create/discount API

**Менеджер бачить:** єдина форма оплати (зі списку «Створити» та з Реалізації «Оплата»): клієнт (+опц. реалізація),
Вид руху, До оплати (авто з боргу/реалізації), Включити суму боргу, згорнуті курси EUR/USD, 4 канали оплати
(UAH/UAH-безнал/EUR/USD) з підказками «ще треба», банк. рахунок (при безналі), стаття (при Расход), 3 поля здачі
з підказками, підсумки Оплата/Решта/Залишок (EUR), кнопки **Сформувати / Дати знижку на залишок / Вайбер**.

- Переробити `lib/manager/cash-order.ts` під формули §B аудиту (зведення в EUR, рекомендації, ВключитиСумуБоргу, ручна здача).
- `POST /api/v1/manager/cash-orders` (переробка): optional `saleId` + `customerId`, `type`, 4 канали, 3-валютна ручна
  здача, `bankAccountId`, `cashFlowArticleId`, курси-знімок, `debtCorrection`. Створює income-ордер (+ за наявності
  здачі — expense-ордер з `changeForId`). Ownership server-side. **Без анти-дубля.**
- `POST /api/v1/manager/cash-orders/discount-remainder` — знижка на залишок: є реалізація → зменшити найдорожчий
  `SaleItem` (priceEur/pricePerKg) на залишок + прапор + перерахунок `Sale.totalEur/totalUah`; нема реалізації →
  записати `debtCorrection` на ордер. Гард `|залишок| ≤ поріг`.
- Payment-форма (заміна `payment-modal.tsx`) + гард банк. рахунку при приході + м'яке попередження повної оплати.
- Wire: `/manager/payments` «Створити» + `/manager/sales/[id]` «Оплата» → одна форма (реалізація preset).

### Етап 3 — Viber + sync-каркас + БСЛ-спека

- `buildPaymentReceiptText` (порт `ПолучитьТекстСообщенияВайбер`/`ПолучитьТекстДляВайбер`): реквізити безналу /
  факт-оплата по валютах / решта / наложка — + кнопка «Вайбер» через наявний `ShareSheet`.
- `enqueueCashOrderCreate` (entityType `cash_order`, мультивалютні UUID-ключі) + `routeFor` case
  `/sync/cash-orders/:id` + manager-sync route + `createCashOrderMock` (fire-and-forget у create).
- БСЛ-спека §3.5 (`СоздатьПКО`: 1 мобільний ордер → до 3 ПКО/РКО по валютах + безнал як платіжне доручення;
  ключі UUID; прапор `Сдача`; §H аудиту) у `docs/1C_SYNC_MODULES_SPEC.md`.
