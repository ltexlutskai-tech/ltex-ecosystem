## Ухвалені рішення (2026-07-03)

1. **Новий клієнт з сайту** → менеджер за мапою **область→агент** (`MgrRegionAgent`, вже існує для
   чат-ботів) + **дефолтний менеджер** як fallback (кого саме — спитати при розробці Блоку 1).
2. **Замовлення з кошика сайту** → у системі створюється **Замовлення-чернетка** на клієнта
   (з позиціями кошика) + **нагадування/сповіщення** менеджеру. Менеджер перевіряє і проводить.
3. **Фото товарів** → керуються **в картці товару системи** (`/manager/prices/[id]`): блок фото
   (upload на `/media/*`, реордер, видалення). Вкладка «Товари» в адмінці сайту зникає.
4. **Дашборд-«відвідувачі»** → **власний легкий лічильник** візитів (без зовнішніх сервісів/кукі)
   - те, що вже є: замовлення з кошика, ліди (швидкі замовлення), продані через сайт товари,
     перегляди товарів (ViewLog).

## Блоки робіт (кожен — окремий воркер; перед кожним — питання user-у)

### Блок 0 — Знос 1С-синхронізації (механічний, питань нема)

- Адмінка: прибрати сторінки/пункти «Журнал синхр.», «Черга sync 1С», розділ «1С СИНХРОНІЗАЦІЯ»
  (sync-products/categories/lots/prices/rates сторінки), «Курси валют» (редагування — у системі).
- API: видалити `/api/sync/*` (products/categories/lots/prices/rates/orders import+export) —
  їх викликала лише 1С. Видалити `/api/admin/sync-jobs*`.
- Код: прибрати enqueue-виклики (`lib/sync/enqueue.ts`, `queue-processor.ts`, `proxy-client.ts`,
  `pull-from-1c.ts`) + крони `/api/cron/process-sync-queue`, `/api/cron/pull-from-1c`.
- Сервіс: видалити `services/manager-sync/` цілком + запис в `ecosystem.config.js`.
- Header менеджерки: прибрати індикатор «Синхронізовано Xс тому» (якщо читає sync-статус).
- БД: таблиці `mgr_sync_jobs`/`mgr_sync_state`/`sync_log` НЕ дропаємо (безпечно лишити; приберемо колись потім).
- ЗАЛИШИТИ: `apps/store/scripts/import-1c-historical.ts` + `scripts/import-1c.ps1` (офлайн-реімпорт
  історії з MSSQL — потрібен до фінальної звірки перед продакшном).
- **Деплой-нотатка user:** видалити Windows Scheduled Tasks «LTEX Process Sync Queue» і
  «LTEX Pull From 1C»; `pm2 delete ltex-manager-sync; pm2 save`.

### Блок 1 — Замовлення з сайту → система

Пайплайн: checkout кошика → (знайти/створити Customer — вже є) → **знайти/створити MgrClient**
(match по phone; новому — агент за мапою область→агент, fallback дефолтний менеджер) →
**створити Order-чернетку** в системі (позиції кошика; клієнт; canал=сайт) → **нагадування**
менеджеру (`MgrReminder`) + (опц.) Telegram-сповіщення. Те саме для швидкого замовлення (quick-order) — як лід.
Прибрати вкладку «Замовлення» з адмінки. Email-підтвердження покупцю — лишити як є.
**Питання перед розробкою:** хто дефолтний менеджер; чи бронювати лоти з кошика одразу; статус
чернетки (draft/sent); що робити з «замовленнями» без збігу телефону (дубльовані клієнти).

### Блок 2 — Клієнти з сайту → довідник

Реєстрація на сайті → авто-створення MgrClient (агент за тією ж мапою) + таймлайн-запис.
Прибрати вкладку «Клієнти» з адмінки. (Технічно спільний код із Блоком 1 — resolve-or-create.)

### Блок 3 — Товари/Категорії/Фото в системі

- Картка товару менеджерки: **блок «Фото»** (upload/reorder/delete → `/media/*`, sharp-пайплайн
  з адмінки перевикористати) + можливість створити НОВИЙ товар у системі (форма в менеджерці).
- Категорії: CRUD у системі (перенести функціонал `/admin/categories` у `/manager/...`).
- Прибрати вкладки «Товари», «Лоти», «Категорії» з адмінки.
- **Питання перед розробкою:** хто з ролей може створювати/редагувати товари й фото (усі менеджери /
  лише admin+owner+warehouse?); чи потрібні лоти вручну з системи (зараз — Поступлення).

### Блок 4 — Дашборд сайту

- Легкий лічильник візитів: таблиця агрегатів на день (visits/uniques по днях, без PII/кукі —
  hash IP+UA на день) + middleware/route-hook. Нова невелика міграція.
- Плитки: візити (день/тиждень), замовлення з кошика (к-сть/сума), ліди quick-order, продані
  через сайт товари, топ переглядів (ViewLog).
- Прибрати з дашборду все 1С/склад/борги (це в системі).

## Порядок виконання

Блок 0 (одразу) → Блок 1+2 (спільний пайплайн; після відповідей) → Блок 3 → Блок 4.
Кожен блок: спека → питання user → воркер → перевірка → merge → деплой-нотатки.
