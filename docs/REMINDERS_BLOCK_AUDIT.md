# Аудит блоку «Нагадування» (M3.0)

Дослідження старого 1С-функціоналу (MobileAgentLTEX) + інвентар того, що вже є в
нашій менеджерці, перед написанням плану реалізації.

Джерела: `docs/1c-export-mobile/MobileAgent/` (Catalog.Нагадування,
DataProcessor.Нагадування, InformationRegister.ЗапланированыеУведомления, enums
`Періоди` + `ТипыДействийПриНапоминании`, спільний модуль авто-створення),
`docs/1c-export/` (центральна база), наша поточна реалізація M1.3c.

---

## 1. Призначення блоку

Особистий «органайзер» менеджера на пристрої: нагадування про дзвінки, оплати,
дії з клієнтами + **автоматичні** системні нагадування (з'явилось відо на товар,
що його чекає клієнт; минула бронь мішка; є протерміновані замовлення).

## 2. Архітектура в 1С (3 рівні)

| Рівень | Об'єкт                                      | Роль                              |
| ------ | ------------------------------------------- | --------------------------------- |
| Дані   | `Справочник.Нагадування`                    | Зберігає налаштування нагадування |
| Черга  | `РегистрСведений.ЗапланированыеУведомления` | Заплановані спрацювання + статус  |
| UI     | `Обработка.Нагадування` (3 форми)           | Список/створення + 2 попапи       |

## 3. ⚠️ Обмін з центральною базою — ВІДСУТНІЙ

У центральній базі (`docs/1c-export/`) **немає** жодного об'єкта
`Нагадування`/`ЗапланированыеУведомления` і **жодної згадки** в обмінах
(`ОбменАРМ`, ExchangePlans). У файлах блоку — **нуль** згадок
обмен/выгрузка/HTTP/SOAP. Блок **повністю локальний на пристрої**. Нагадування
генеруються самим агентом на основі **вже синхронізованих** даних (замовлення,
броні, відео на товарах). Тобто реалізуємо повністю на нашому боці, без 1С-обміну.

## 4. Справочник «Нагадування» — поля

| Поле                        | Тип                               | Призначення                  |
| --------------------------- | --------------------------------- | ---------------------------- |
| `Нагадування`               | Строка(256)                       | Текст/опис нагадування       |
| `ДатаНагадування`           | Дата/час                          | Коли спрацювати              |
| `Періодичність`             | enum `Періоди`                    | Режим повторення             |
| `Активне`                   | Булево                            | Активне (не виконане)        |
| `ДеньНедели`                | Число                             | Авто з дати (для «Щотижня»)  |
| `Контрагент`                | → Контрагенты                     | Клієнт нагадування           |
| `НагадатиПроТовар`          | Булево                            | Тип «Для товарів»            |
| `ІзЗаказа`                  | Булево                            | Згенероване із замовлення    |
| `ЗаказВідео`                | Булево                            | Прапорець «замовлення відео» |
| `КонтрагентДляДействій`     | → Контрагенты                     | Клієнт для авто-дії          |
| `ХарактеристикаДляДействій` | → ХарактеристикиНоменклатуры      | Лот/мішок для авто-дії       |
| `КонтрагентВідео`           | → Контрагенты                     | Клієнт для відео-дії         |
| `НоменклатураВідео`         | → Номенклатура                    | Товар для відео-дії          |
| `ТипДействяПриНапоминании`  | enum `ТипыДействийПриНапоминании` | Тип авто-дії                 |

**Таблична частина `Товары`** (для типу «Для товарів»):
`Номенклатура` (артикул), `Кількість`, `Виконано` (булево по рядку).

## 5. Перечислення

**`Періоди`** (6 значень, усі є в UI — підтверджено скріном форми):
`НеПовторювати`, `Щодня`, `Щотижня`, `Щомісяця`, `Щороку`, `ПоПодії`.

**`ТипыДействийПриНапоминании`** (3 значення):

- `ЗакрытиеСтарыхЗаказов` — є протерміновані замовлення → відкрити «Закриття старих замовлень»;
- `ПродолжениеБрониТовара` — минула бронь → відкрити форму встановлення броні;
- `ОтправкаСообщенияВайберОВідео` — з'явилось відео → відкрити картку для Viber-повідомлення.

## 6. Форма створення/список (Обработка.Форма)

- Перемикач типу зверху: **«Звичайне»** ↔ **«Для товарів»** (драйвер `НагадатиПроТовар`) — змінює сторінку панелі.
- Поля: **Опис** (текст), **Дата нагадування** (дата/час), радіо-група **періодичності** (6 варіантів), чекбокс **«Заказ відео»**.
- Панель **«Для товарів»**: пікер **Контрагент** (із «Показати всі» + «+»), кнопка **«Додати»**, таблиця **Товар / Кількість** (+ `Виконано`).
- Динамічні **підказки** під датою (за режимом), дослівно з BSL:
  - Щодня: `Кожен день о ГГ:хх`
  - Щотижня: `Кожен <день тижня> о ГГ:хх`
  - Щомісяця: `Кожен <N> день місяця о ГГ:хх`
  - Щороку: `Кожен рік <d MMMM> о ГГ:хх`
- Велика кнопка внизу **«Встановити нагадування»** → `ВстановитиНагадування()`.

## 7. Фільтри списку (Catalog.ФормаСписка)

- **«Завершені»** — за замовч. вимкнено → показує лише активні (`Виконано=Ложь`); увімкнено → і виконані.
- **«Заказ відео»** — показує лише `ЗаказВідео=Истина`.
- Колонки: Опис (`Нагадування`) + Дата (`дд.ММ.рррр ГГ:хх`) + позначка виконання (зелена «галочка»).
- Жовтим підсвічується активний (ще не виконаний) запис.

## 8. ФормаПерегляду (попап при спрацюванні)

Read-only: дата + текст. Радіо-група **«Дія»**:
`0` = Виконано; `N` хв = Відкласти на N хв; `1` = Відкласти до нового надходження (лише для «Для товарів»).
Кнопка **«Дія по попередженню»** — лише якщо заповнений `ТипДействяПриНапоминании` (виконує відповідну дію). OK: дія 0/1 → закрити/обробити; дія >1 хв → відкласти (+ перепланувати локальний push).

## 9. ФормаОтриманихНагадувань (пакетний попап)

Заголовок «Активні нагадування». Таблиця всіх прострочених нагадувань (вибір →
деталі) + кнопка «Дія по попередженню» (маршрутизує за типом дії: ЗакриттяСтарихЗамовлень
→ форма закриття; Viber-відео → картка характеристики; ПродовженняБроні → форма броні)

- OK (виконати/відкласти). Показується коли спрацювало кілька нагадувань одночасно.

## 10. Регістр ЗапланированыеУведомления

Поля: `УИД_Уведомления` (ключ), `ОбъектУведомления` (→ Нагадування),
`ДатаУведомления`, `ДанныеУведомления` (серіалізована структура
Заголовок/Текст/Данные), `Выполнено`, `Отложено`. Індекси `(Выполнено, ДатаУведомления)`
для опитування черги + `(ОбъектУведомления, Выполнено)` для антидублю.

## 11. Авто-генеровані нагадування (`СоздатьНапоминаниеСДействием`) — 3 типи

Усі: періодичність `НеПовторювати`, дата = `ТекущаяДата()+60` (майже одразу),
антидубль = запит «чи є вже активне (не виконане) нагадування з тим самим типом
дії + тим самим клієнтом/характеристикою».

1. **Закриття старих замовлень** — текст: `По контрагенту <X> є протерміновані активні замовлення. Закрити їх?` (дія `ЗакрытиеСтарыхЗаказов`).
2. **Viber про відео** — текст: `По артикулу <арт> мішку <лот> для контрагента <X> з'явилось відео. Відправити у вайбер повідомлення?` (дія `ОтправкаСообщенияВайберОВідео`).
3. **Продовження броні** — текст: `По артикулу <арт> мішку <лот> для контрагента <X> була знята бронь. Перенести бронь?` (дія `ПродолжениеБрониТовара`).

**Push:** на мобільному (`#Если МобильноеПриложениеКлиент`) — локальний OS-push;
на десктопі — попап `ФормаПерегляду`. **Повторювані** режими (Щодня/…) push **НЕ**
реєструють — лише запис у черзі.

## 12. Нагадування «Для товарів» — логіка

Один запис може містити кілька товарів; виконується **порядково** (`Виконано` по
рядку). Коли всі рядки виконані → запис `Активне=Ложь`. Антидубль при додаванні:
якщо для клієнта вже є активне нагадування по цьому артикулу — не додавати повторно
(в журнал: «Нагадування по артикулу <X> для клієнта <Y> вже існує…»).

## 13. Що в нас уже є (M1.3c) + GAP-аналіз

**Є зараз:**

- Модель `MgrReminder` = `{ clientId (обов'язк.), ownerUserId, body, remindAt, completedAt, snoozedUntilAt }`.
- API `/api/v1/manager/clients/[id]/reminders` (GET/POST) + `[rid]` (PATCH/DELETE) — дії complete/uncomplete/snooze/edit.
- UI: вкладка «Нагадування» на картці клієнта (форма + 4-бакетне групування) + дзвіночок у шапці (polling 60с, top-10 прострочених).
- Standalone `/manager/reminders` — **заглушка `UnderConstruction`**.

**Це ≈ лише тип «Звичайне», прив'язаний до клієнта, без повторень і авто-дій.**

**Бракує проти 1С:**
| # | Розрив | Потрібно |
|---|---|---|
| G1 | Standalone-екран | Реальний список усіх моїх нагадувань + фільтри «Завершені»/«Заказ відео» + «Створити» |
| G2 | Нагадування без клієнта | `clientId` → **опційний** (звичайне нагадування може бути без клієнта) |
| G3 | Періодичність | enum `Періоди` + поле + (повторення Щодня/Щотижня/Щомісяця/Щороку/По події) |
| G4 | Тип «Для товарів» | дочірня таблиця `MgrReminderItem` (товар/к-сть/виконано) + форма |
| G5 | Прапорець «Заказ відео» | поле `orderVideo` + фільтр |
| G6 | Спрацювання-попап | модалка при спрацюванні (Виконано / Відкласти N хв / Дія) + пакетний показ |
| G7 | Авто-генеровані (3 типи) | детектори подій (відео/бронь/старі замовлення) + дії |
| G8 | «Закриття старих замовлень» | окремий екран (є в 1С окремим пунктом меню) — потрібен для дії №1 |

**Залежності G7/G8 від інших систем:**

- Viber-відео: маємо `videoUrl` на товарах/лотах + замовлення → подію можна обчислити.
- Бронь: маємо бронювання лотів (блок Прайс: `reserved*` на `Lot`) → закінчення броні можна детектувати.
- Закриття старих замовлень: **окремий блок/екран, якого в нас немає** (відповідає пункту меню 1С «Закриття старих замовлень»).

## 14. Кандидатні етапи (чернетка для плану)

- **Етап 1 — Standalone «Звичайне» + список + фільтри.** Замінити заглушку реальним екраном (мої нагадування по всіх клієнтах + standalone), фільтри «Завершені»/«Заказ відео», пошук, «Створити». Схема: `clientId` опційний + поле періодичності + `orderVideo` + тип. Реюз наявного CRUD.
- **Етап 2 — Періодичність.** Радіо 6 режимів + підказки + перепланування наступного спрацювання для повторюваних (через наявний cron-механізм).
- **Етап 3 — Тип «Для товарів».** `MgrReminderItem` (товар/к-сть/виконано) + форма (клієнт + додавання товарів + таблиця) + порядкове виконання + антидубль.
- **Етап 4 — Спрацювання-UX.** Попап при спрацюванні (Виконано / Відкласти N хв) + пакетний попап + розширення дзвіночка.
- **Етап 5 (залежний) — Авто-генеровані.** Детектори: відео-з'явилось / бронь-минула (маємо дані) + дії (Viber / продовжити бронь). «Закриття старих замовлень» — після окремого екрана.

## 15. Відкриті питання (обсяг) — рішення user

1. **Обсяг авто-нагадувань (G7/G8):** робити зараз manual-частину (Етапи 1-4) і відкласти авто-генеровані + «Закриття старих замовлень» на пізніше (вони перетинаються з бронями/відео + фазою 1С-обмінів)? Чи включити доступні авто-типи (відео/бронь) одразу?
2. **Повторення (G3):** потрібні справжні повторювані нагадування (Щодня/Щотижня перестворюються) вже в v1, чи достатньо одноразових + збереження режиму як мітки, а повторення — follow-up?
3. **UX спрацювання (G6):** модалка-попап при спрацюванні (як у 1С) чи поки достатньо наявного дзвіночка + списку?
