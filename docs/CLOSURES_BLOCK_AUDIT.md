# Аудит блоку «Закриття старих замовлень» (M3.x)

Дослідження старого 1С-функціоналу (MobileAgentLTEX + **центральна база**) + інвентар
нашого коду. **Висновок наперед: блок принципово зав'язаний на обмін із центральною
1С** (на відміну від Нагадувань/інших блоків, що працювали на локальних даних).

Джерела: `docs/1c-export-mobile/MobileAgent/DataProcessors/ЗакрытиеСтарыхЗаказов/`,
`docs/1c-export-mobile/Central/CommonModules/ОбменАРМ/Ext/Module.bsl`, наш код
(`Order`/`Sale`, sync-проксі, заглушка `/manager/closures`).

---

## 1. Призначення

По кожному клієнту в центральній 1С може бути лише **одне актуальне замовлення**.
Щоб створити нове — старі мають бути закриті. Інструмент: (а) **актуалізує стан** —
закриває в центральній базі виконані/непотрібні замовлення; (б) **аналіз+перенос** —
показує що замовляли / що фактично відвантажено («Продано») і одним кліком переносить
позиції в нове замовлення.

## 2. ⚠️ Архітектура: ВИКЛЮЧНО через центральну базу (SOAP)

- **Список незакритих** — серверний метод `ПолучитьДанныеНезакрытыхЗаказов(пароль, GUID_контрагента)`:
  запит до `РегистрНакопления.ЗаказыПокупателей.Остатки` (КоличествоОстаток > 0,
  фільтр по контрагенту+торговому агенту). Повертає `ХранилищеЗначения(ТаблицаЗначений)`:
  `Количество, Сумма, Заказ_уид, Номенклатура_уид, Дата, НоменклатураПредставление, СтатусЗамовлення, Продано`.
- **«Продано»** — обчислюється на сервері: SUM кількості з `Документ.РеализацияТоваровУслуг.Товары`
  по тому ж товару+контрагенту, де дата реалізації > дати замовлення. **Не зберігається**,
  суто інформативне (скільки відвантажено від моменту замовлення).
- **«Закриття»** — `ЗакрытьСтарыеЗаказы(пароль, Товары, GUID_контрагента)`: центральна
  база створює і **проводить** `Документ.ЗакрытиеЗаказовПокупателей` (ВидОперації=ЗакрытиеЗаказов),
  що обнуляє залишки регістру. Мобільний отримує лише рядок успіх/помилка — **локального
  ефекту немає**.
- Мобільний агент **не працює офлайн** — без живого SOAP список порожній.

## 3. ⚠️ SOAP-інтерфейс використовує `ХранилищеЗначения`

Існуючі методи приймають/повертають `ХранилищеЗначения` (бінарна 1С-серіалізація).
Наш Node-проксі (`services/manager-sync`) **не вміє** будувати/читати ValueStorage.
Тому **переюзати існуючі методи напряму з нашого застосунку не можна** — потрібні
**нові JSON-дружні web-методи** на стороні центральної 1С (як у решті обмінів —
пишемо BSL-спеку в `1C_SYNC_MODULES_SPEC.md`), які всередині викликають ту саму
логіку (запит до регістру / створення документа закриття). Серверна **логіка вже є** —
треба лише JSON-обгортки.

## 4. Форма (мобільна) — макет

- Реквізит **Контрагент** (вибір з довідника; при зміні таблиця очищується).
- Кнопка **«Заповнити»** → завантажує незакриті (SOAP).
- Таблиця `Товары` (read-only, **не можна додавати/видаляти рядки вручну**):
  колонки Замовлення(дата) / Номенклатура / Статус / **Додати в замовлення** (чекбокс,
  єдине редаговане) / Цена / Замовлено(Количество) / Сумма / **Продано** (зелена
  підсвітка рядка коли `Продано ≥ Количество`).
- Кнопка **«Закрити замовлення»** → SOAP-закриття + запис в історію клієнта
  (`РегистрСведений.РаботаСКлиентом`, авто-запис) + якщо є відмічені «Додати в
  замовлення» → авто-створення нового `Документ.Заказ` і його відкриття.
- **Створення нового замовлення:** Контрагент, ПризначитиПродажТорговому=Истина,
  ВивантажуватиВ1С=Истина, Дата=зараз, Доставка=Доставка, ТипЦен=з контрагента,
  СкладКонтрагента; по рядках Номенклатура/Количество/Цена/Сумма + `ЦенаПродажиВес = Цена/20`.

## 5. Інтеграція

- **Авто-відкриття:** при створенні замовлення `ПоискАктуальныхЗаказов` питає центральну
  (`ПолучитьАктуальныеЗаказы`) — якщо у клієнта є проведене замовлення зі статусом
  «Новий» і його нема локально → форма закриття **відкривається автоматично** з
  підставленим клієнтом.
- **Нагадування:** enum `ТипыДействийПриНапоминании.ЗакрытиеСтарыхЗаказов` існує; при
  обміні система мала б створювати нагадування «…є протерміновані активні замовлення.
  Закрити їх?» з дією, що відкриває цю форму з клієнтом. (У нашому блоці Нагадувань ми
  цю 3-тю дію свідомо відклали — саме сюди.)

## 6. Наш код — інвентар + GAP

**Є:**

- Заглушка `/manager/closures` (`UnderConstruction`), пункт меню «Закриття старих замовлень».
- `Order` (`isActual`/`archived`/`status` draft/sent/posted/cancelled, `code1C`, `customerId`)
  - `OrderItem` (productId/lotId/quantity/weight/priceEur). `Sale`/`SaleItem` (реалізації).
- Створення замовлення: `POST /orders` + `createOrderWithItems` (але `/orders/new` **не**
  приймає prefill-параметри). `getMyClientCodes1C` (ownership). `MgrClientTimelineEntry`
  (історія). sync-проксі (mock-режим), `SyncEntityType` (без `closures`).
- `MgrReminderAction` — без `close_orders`.

**GAP проти 1С:**
| # | Розрив |
|---|---|
| G1 | Список незакритих = з центрального **регістру залишків**, якого ми не дзеркалимо. Наші локальні `Order` ≠ центральні залишки. |
| G2 | «Продано» точне = з центральних реалізацій (по даті). Локально — лише приблизно (нема зв'язку замовлення↔реалізація). |
| G3 | «Закриття» = проведення центрального документа. Локально — лише `archived=true`. |
| G4 | SOAP-методи на `ХранилищеЗначения` → потрібні нові JSON-методи (BSL) + жива конекція. |
| G5 | `/orders/new` без prefill; sync без `closures`; reminder-дія `close_orders` відсутня. |

## 7. Варіанти підходу (рішення user)

- **A. Реальний обмін з 1С (цей блок — пілот обмінів):** повна UI + наш API + операції
  у sync-проксі + **BSL-спека** 2 нових JSON-методів для центральної (логіка там уже є).
  Дані справжні, «закриття» справжнє. **Але:** live тільки коли на сервері 1С задеплоять
  BSL + ми налаштуємо конекцію; з пісочниці не тестується (нема доступу до 1С) — тестує user/1С-розробник.
- **B. Локальна наближена версія зараз:** з наших `Order`/`Sale`; «Продано» приблизне;
  «закриття» лише локально (НЕ в 1С). Працює одразу, але не відображає реальний стан 1С
  і не закриває насправді — слабша/потенційно оманлива версія.
- **C. Відкласти** до загального етапу обмінів з 1С (бо блок усе одно вимагає живого SOAP).

**Супутні рішення (для A або B):** (1) перенос у нове замовлення — авто-створити+відкрити
чи лише закрити; (2) доступ — лише менеджери чи +admin; (3) чи вмикаємо тепер reminder-дію
`close_orders` (з блоку Нагадувань).
