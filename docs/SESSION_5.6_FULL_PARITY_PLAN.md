# Сесія 5.6+ — Повний паритет з 1С: усі довідники, регістри, документи + перенос 100% даних

> **Мета (рішення user 2026-06-17):** відтворити у власній системі **кожен** довідник, регістр і
> документ, яким реально користувались у 1С, і **перенести всю історичну інформацію** за весь
> період роботи. Функціонал можна робити по-новому (обмінів з 1С уже немає), але дані —
> на 100%, зі схожими звітами/документами/довідниками.
>
> **Скоуп = «що реально використовувалось»:** уся мобільна конфіга (`docs/1c-export-mobile-full/`)
>
> - ті об'єкти центральної УТ, що **містять дані L-TEX** (борг, продажі, собівартість, каса/банк,
>   залишки, ціни, курси, повернення). Стандартний баласт УТ без даних L-TEX (виробництво, ОЗ,
>   ПДВ-книги, роздріб, бюджетування) — НЕ відтворюємо (там немає що переносити).
>
> Повний інвентар об'єктів зі статусом — `docs/1C_REGISTRY_INVENTORY.md`.

## Архітектурний принцип (єдиний для всього)

1С тримає підсумки у **регістрах накопичення** (рухи: прихід/розхід по вимірюваннях). Ми вже
довели цей патерн на боргу: `MgrDebtMovement` (таблиця рухів + idempotent-ключ
`(kind, sourceType, sourceId)` + recompute + hooks проведення документів). **Тиражуємо його на всі
регістри:**

- **Кожен регістр накопичення 1С → наша таблиця рухів** (`*Movement`) з вимірюваннями-колонками
  - ресурсами + `occurredAt` + idempotent-ключем джерела.
- **Історичні дані** імпортуємо **прямо з 1С MSSQL `_AccumRg…`** (авторитетні підсумки) —
  як зробили для боргу (`--entity debt`, 78к рухів). Не реплеїмо документи — беремо готові рухи.
- **Нові документи** при проведенні пишуть рухи у відповідні регістри (hooks, як у боргу).
- **Звіти** читають із таблиць рухів (швидко, як 1С-звіти «Залишки/Обороти»).
- **Довідники** — звичайні таблиці + admin-CRUD; історія з 1С через `--entity dictionaries`.
- **Розширення імпортера:** кожна фаза додає `--entity <name>` у `import-1c-historical.ts`
  (idempotent upsert по `code1C`=hex / по idempotent-ключу руху).

Новий розділ сайдбара **«Довідники та регістри»** (як 1С «Все функції»): список усіх довідників
(CRUD) + усіх регістрів (read-only переглядач рухів із фільтрами) + розділ **«Звіти»**.

## ФАЗИ (кожна — самостійно постачувана; user обирає порядок/обсяг)

### ФАЗА 0 — Каркас «Довідники та регістри» + розділ «Звіти»

- Пункт сайдбара «Довідники та регістри» → сторінка-хаб зі списком усіх об'єктів (як 1С).
- Загальний компонент **переглядача регістру** (таблиця рухів: період, реєстратор, вимірювання,
  ресурси, прихід/розхід; фільтри по даті/вимірюваннях; CSV/XLSX-експорт).
- Розділ «Звіти» — хаб усіх звітів (наявні sales-by-client/supplier/debts + нові).
- **Без міграцій даних.** Інфраструктура для решти фаз.

### ФАЗА 1 — Закрити прогалини довідників (+ перенос даних)

Окремі таблиці + admin-CRUD + історичний імпорт для:

- **Одиниці виміру** (`ЕдиницыИзмерения` + `КлассификаторЕдиницИзмерения`) — шт/кг/пара/коеф.;
  додати `unit` на `OrderItem`/`SaleItem`/`Lot` (зараз ніде не пишеться).
- **Міста / Області / Торгові агенти** — окремі довідники (зараз рядки/мапа); FK з клієнтів.
- **Контакти Viber / Групи чату / Учасники чату** — таблиці підписників і груп.
- **Категорії об'єктів, Робочий стіл, Сховище дод. інформації** — службові, за потреби.
- Дані: новий `--entity dictionaries2` (або розширити наявний `dictionaries`).
- Міграція: `2026xxxx_dictionaries_full`.

### ФАЗА 2 — Регістри-обороти + базові звіти (ядро запиту)

Матеріалізувати 4 ключові регістри + переглядачі + звіти:

- **2a. Продажі** (`AccumRg Продажи`) → `SalesMovement` (Номенклатура, Характеристика, Клієнт,
  Агент, ДокументПродажу / Кількість, Вага-кг, Виручка, ВиручкаБезЗнижок).
  Звіти: **виручка/кг по клієнтах · по товарах · по агентах · за період · сезонність · ефект знижок.**
- **2b. Рух коштів / ДДС** (`AccumRg ДвиженияДенежныхСредств`) → `CashFlowMovement` (Каса/Рахунок,
  Стаття, Прихід/Розхід, Контрагент / Сума). Звіт: **cash-flow по статтях; залишок каси/рахунків на дату.**
- **2c. Залишки товарів шт + кг** (`AccumRg ОстаткиТоваров` + `_AccumRg6608` у вазі) → `StockMovement`
  (Склад, Номенклатура, Характеристика, Якість / Кількість, Вага). Звіти: **залишки складу шт/кг ×
  якість/категорія; тонни на складі; неліквіди; вільне = склад − резерв** (з регістром резервів).
- **2d. Залишки замовлень** (`AccumRg ЗаказыПокупателей`) → звіт **«до відвантаження»** (доповнює «Потреби»).
- Борг (`ВзаиморасчетыСКонтрагентами`) — ✅ вже є (`MgrDebtMovement`); додати переглядач+звіт «Акт звірки».
- Дані: `--entity sales-reg, cashflow-reg, stock-reg, orders-reg`. Міграції на кожен регістр.

### ФАЗА 3 — Собівартість і маржа (найцінніше для власника)

- Імпорт `AccumRg ПродажиСебестоимость` → собівартість у `SalesMovement.cost`.
- Звіт **«Валовий прибуток / Маржа»** = Виручка − Собівартість, по товарах/клієнтах/агентах/період.
- (Опц.) Партіонні залишки `ПартииТоваров` → вартість запасу складу.
- Дані: `--entity cost-reg`. **Без цього маржа неможлива** (історичної собівартості в нас немає).

### ФАЗА 4 — Історичні курси валют

- Імпорт `InfoRg КурсыВалют _InfoRg4655` (EUR/USD→UAH помісячно/подобово) → `ExchangeRate` (ряд дат).
- Перерахунок грн-сум у звітах/документах за курсом дати (зараз fallback 43 спотворює суми).
- Дані: `--entity rates`. Дешево, відчутний ефект на всі грошові звіти.

### ФАЗА 5 — Документи руху товару (нові, + перенос)

Кожен: модель(+ТЧ) → проведення пише рухи у `StockMovement`/`MgrDebtMovement` → UI → імпорт історії.

- **Повернення від покупця** (`ВозвратОтПокупателя`) — мінус-реалізація, **прямо коригує борг**.
- **Повернення зі складу / постачальнику** (`Возврат`, `ВозвратПоставщику`).
- **Перепаковка** (`Перепаковка`) — розбір/комплектація мішків (специфіка секонду; 2 ТЧ + норма втрат).
- **Списання / Оприбуткування / Інвентаризація / Переміщення між складами.**
- Дані: `--entity returns, repack, writeoff, inventory, transfer`. Міграції на кожен документ.

### ФАЗА 6 — Фінансові документи: банк/каса розширення (+ перенос)

- **Платіжне доручення вхідне/вихідне** (безнал), або легше — поле «спосіб оплати
  готівка/картка/банк» + банк-реквізити на `MgrCashOrder`.
- **Переміщення готівки / Інкасація (внесок) / Грошовий чек** — рух каса↔каса, каса↔банк.
- **Акт звірки взаєморозрахунків** — read-only звіт по `MgrDebtMovement` за період (дебет/кредит/сальдо).
- **Рахунок на оплату** — окремий документ-інвойс (зараз лише друк з `Order`).
- Дані: `--entity bankdocs, cashmoves`.

### ФАЗА 7 — Звіти: повний пакет + експорт

- Хаб «Звіти» з усіма звітами фаз 2-6 + наявними; єдині фільтри (період/клієнт/агент/товар/склад).
- **XLSX-експорт** (зараз CSV) для всіх звітів.
- Друк документів, де ще немає.

### ФАЗА 8 — Дрібне / службове (за бажанням)

- Тайм-трекінг дня агента (`СтатусДня` Початок/Кінець), Норми запасів (`НормыЗапасов`),
  реакції/видалені повідомлення чату, надійність постачальників, історія статусів клієнта.

## Перенос даних — спосіб

Усе через `apps/store/scripts/import-1c-historical.ts` (читає 1С **MSSQL** `LEGACY_1C_DB_URL`,
idempotent). Кожна фаза = новий `--entity`. Регістри-обороти беремо з `_AccumRg…` напряму
(авторитетні рухи), документи — з `_Document…` по `code1C`=hex. Після кожної фази — прогін
відповідного `--entity` на проді + звірка кількостей із 1С-звітами.

> ⚠️ Залежність: для частини регістрів потрібен доступ до 1С MSSQL (`ltex` база) — він уже
> налаштований (`LEGACY_1C_DB_URL=mssql://ltex_app_reader@localhost:1433/ltex`). Якщо база 1С
> уже знята/недоступна — частину `_AccumRg…` можна взяти з дампу `docs/1c-export-2026-06-02/`
> лише як метадані (структуру), але **самі дані рухів — лише з живої MSSQL**. Це треба підтвердити.

## Рекомендований порядок (за цінністю/залежностями)

**0 → 4 (курси, дешево, ефект всюди) → 2 (регістри+звіти) → 3 (маржа) → 1 (довідники) → 5
(документи руху) → 6 (фінанси) → 7 (звіти+XLSX) → 8 (дрібне).**

Кожна фаза — окремий пакет: схема(міграція) + `--entity` імпорт + UI + звіт + перевірки
(typecheck/тести/prettier/build) + деплой. Орієнтовно 1 фаза ≈ 1 робоча сесія воркерів.
