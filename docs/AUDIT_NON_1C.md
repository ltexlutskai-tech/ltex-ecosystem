# Аудит не-1С товарів / лотів / штрихкодів

Скрипт `apps/store/scripts/audit-non-1c.ts` — **read-only**. Рахує, скільки в БД
сутностей створено НЕ з 1С (наш Excel-каталог S71 / документи «Поступлення»), і
скільки з них можна безпечно почистити vs заблоковано історією продажів.

> **Сесія 5.7, п.7 плану** (`docs/SESSION_5.7_DICTIONARIES_PLAN.md`).
> Рішення user: **поки ЛИШЕ аудит** — нічого не видаляти/не архівувати.
> Скрипт жодного `create/update/delete` не робить.

## Запуск

Скрипт читає `DATABASE_URL` напряму (бойова база). На відміну від
записувальних скриптів (`import-1c-historical.ts`, `recompute-client-debt.ts`),
тут **немає** гарду `IMPORT_TARGET_DB_URL` / `--confirm-prod` — бо скрипт нічого
не пише, читати бойову базу безпечно.

```powershell
# DATABASE_URL береться з apps/store/.env
pnpm --filter @ltex/store exec tsx scripts/audit-non-1c.ts

# машинний вивід (JSON для подальшої обробки)
pnpm --filter @ltex/store exec tsx scripts/audit-non-1c.ts --json
```

> Це **НЕ** запуск через `import-1c.ps1` — той скрипт для записувального
> імпорту з MSSQL. Тут лише читання нашого PostgreSQL.

## Дискримінатори (як визначаємо «не-1С»)

| Сутність    | «не-1С»                                   | Примітка                                                                                                                                                                             |
| ----------- | ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Product** | `code1C IS NULL`                          | ⚠️ Excel-каталог S71 теж `code1C NULL` (легітимний). Тому ділимо на «з images/prices» (каталог) vs «голі».                                                                           |
| **Lot**     | `receivingId IS NOT NULL`                 | Лот, створений нашим документом «Поступлення». `status='archived'` = ознака 1С-імпорту. Lot ще не має `code1C` (надійніший дискримінатор зʼявиться після реімпорту `--entity lots`). |
| **Barcode** | належить лоту з `receivingId IS NOT NULL` | —                                                                                                                                                                                    |

## Колонки виводу

**PRODUCTS**

- **Усього / З code1C / Без code1C** — загальний поділ 1С vs не-1С.
- **з images/prices (легітим. каталог S71)** — не-1С товари, що мають хоч одне
  зображення АБО ціну. Ознака легітимного Excel-каталогу — **лишати**.
- **«голі»** — не-1С без images і без prices. Перші кандидати на чистку.
- **Не-1С заблоковані історією** — мають хоч один `OrderItem` / `SaleItem` / `Lot`.
  FK-Restrict блокує фізичне видалення — **лишити** (або спершу архів).
- **Не-1С без посилань (кандидати на видалення)** — не-1С без жодних посилань;
  фізично видаляються без FK-помилок.

**LOTS**

- **receivingId != null** — наші не-1С лоти («Поступлення»).
- **status='archived'** — ознака 1С-імпорту.
- **решта** — інші лоти (1С-імпорт без `archived` тощо).
- **Не-1С (receiving) заблоковані історією** — не-1С лоти з `OrderItem`/`SaleItem`.
- **Не-1С (receiving) без посилань (кандидати)** — безпечні для видалення.

**BARCODES**

- **Усього / На не-1С лотах** — штрихкоди, привʼязані до не-1С лотів
  (`Barcode.lot.receivingId != null`). Видаляються каскадом разом з лотом
  (`Barcode → Lot onDelete: Cascade`).

## Наступні кроки (ПІСЛЯ рішення user — окремо)

1. **Лишити** не-1С товари з images/prices (легітимний Excel-каталог S71).
2. **Мʼякий архів** (зворотний) для «голих» не-1С товарів / не-1С лотів —
   окремий скрипт, ще не написаний.
3. **Фіз. видалення** — лише для сутностей «без посилань» (кандидати), за
   окремим підтвердженням на кожен батч. Заблоковані історією — не чіпати
   (FK-Restrict від `OrderItem`/`SaleItem`).
4. Опційно: реімпорт `--entity lots` з новим `Lot.code1C` для надійнішого
   дискримінатора не-1С лотів (наразі — лише `receivingId`).
