## ХВИЛЯ 1 — швидкі перемоги з великою віддачею

### Блок 1. Інтеграція Нова Пошта API (створення ТТН з картки)

**Навіщо:** зараз менеджер вбиває № відділення й ТТН **руками** в реалізації. Усі конкуренти друкують ТТН у 1 клік прямо із замовлення, з авто-підстановкою даних. Це найбільша щоденна втрата часу менеджерів. Бенчмарк — KeyCRM/KeepinCRM (масові ТТН, реєстри, авто-статуси).

**Що робимо (етапами):**

- **1.1. Довідники НП (кеш API):**
  - Підтягування міст/відділень/поштоматів через `Address.getCities` / `getWarehouses` НП API.
  - Автопідстановка отримувача з картки клієнта (`MgrClient.novaPoshtaBranch` → нормальний вибір відділення зі списку, а не текст).
  - _Технічно:_ новий сервіс `lib/delivery/nova-poshta.ts` (обгортка над `https://api.novaposhta.ua/v2.0/json/`), ключ у `.env` `NOVA_POSHTA_API_KEY` (уже частково є в мобільному). Кеш довідників у БД (`NpCity`, `NpWarehouse`) з оновленням раз на добу.

- **1.2. Створення ЕН/ТТН з картки реалізації:**
  - Кнопка «Створити ТТН» на `Sale` → `InternetDocument.save`: відправник (наші реквізити), отримувач (клієнт), вага/місця/опис — **підтягуються авто** з реалізації, післяплата (COD) авто на суму боргу/замовлення.
  - Номер ТТН зберігається на `Sale` (нове поле `Sale.npTtn` замість вільного тексту), трек-код копіюється клієнту одним кліком.
  - _Технічно:_ `lib/delivery/create-waybill.ts`; додати поля `Sale.npRef` (внутрішній ref НП), `Sale.npTtn`, `Sale.npCost`, `Sale.deliveryStatusNp`. Міграція additive.

- **1.3. Друк і масові операції:**
  - Друк маркування (наліпка 100×100) та експрес-накладної (PDF з НП API `printDocument`).
  - **Масове створення ТТН** зі списку реалізацій (виділив N → «Створити ТТН») + друк **реєстру** (`getDocumentList` / `ScanSheet`).

- **1.4. Авто-трекінг статусів:**
  - Cron раз на N хвилин `TrackingDocument.getStatusDocuments` → оновлює `Sale.deliveryStatusNp`; статус НП **каскадом** може міняти статус реалізації/маршруту (як у KeepinCRM: «Прибуло» → сповіщення менеджеру/клієнту).
  - Використати наявну модель `Shipment` (уже є трекінг у мобільному) — розширити на менеджерку.

**Результат:** менеджер оформляє відправлення за секунди, а не хвилини; борг по післяплаті рахується авто.

### Блок 2. Воронка-канбан для лідів і угод

**Навіщо:** зараз ліди — це **проста таблиця статусів** (Новий/Контакт/Конвертований/Відхилений). Менеджер не бачить «дошки», де кожен потенційний клієнт — картка, яку тягнеш по етапах. Це стандарт усіх CRM (Pipedrive, KeyCRM, Sitniks) і головний інструмент, щоб **не губити потенційних покупців**.

**Що робимо:**

- **2.1. Модель воронки:**
  - Нові моделі `Pipeline` (воронка) + `PipelineStage` (етапи, з порядком і кольором) + прив'язка `MgrLead.stageId` (і за потреби `Order.stageId`).
  - Стартові етапи (редаговані): _Новий лід → Взято в роботу → Прорахунок/пропозиція → Очікує оплату → Виграно / Програно._ **5–7 етапів, не більше.**

- **2.2. Канбан-дошка (UI):**
  - Сторінка `/manager/leads/board` — колонки-етапи, картки-ліди, **drag-and-drop** між колонками (бібліотека типу `@dnd-kit`).
  - На картці **лише 3–4 поля**: клієнт, сума/обсяг, канал-джерело (Instagram/Telegram/сайт), **«днів у стадії»**.
  - У **заголовку колонки** — підсумок: скільки лідів + на яку суму (міні-прогноз).
  - **Підсвічування «застрягання»**: якщо лід висить у стадії довше N днів — червона позначка (аналог Pipedrive «Deal Rotting»).
  - Фільтри дошки (по агенту, каналу, сумі) — не виходячи з канбану.

- **2.3. Автоматизація переходів:**
  - Перше вхідне повідомлення в чаті → авто-створення ліда в колонці «Новий».
  - Створення замовлення з ліда → авто-перехід у «Виграно» + конвертація в клієнта (у нас уже є `markLeadsConverted` — під'єднати).

**Результат:** власник і менеджери бачать «живу картину» продажів; жоден потенційний клієнт не губиться; видно, де «вузьке місце».

## ХВИЛЯ 2 — потік нових клієнтів і гроші

### Блок 3. Instagram Direct + добудова WhatsApp (єдиний чат-інбокс)

**Навіщо:** ваш бізнес продає через соцмережі, але система **не бачить Instagram** — найбільший канал лідів для second-hand/моди. У нас уже є єдиний інбокс (Telegram+Viber) і «заготовки» під WhatsApp/Instagram — треба їх **реально під'єднати**. Планка — KeyCRM/Sitniks (Instagram Direct + коментарі в одному вікні).

**Що робимо:**

- **3.1. Instagram Direct (через Meta API):**
  - Підключення Instagram Professional акаунта через Meta (Facebook) Graph API + Messenger Platform (webhook на `messages`, `comments`).
  - Вхідні Direct → у наш `ChatConversation` (платформа `instagram`, у нас уже є в enum), вихідні — через Send API.
  - **Коментарі під постами** → в ту саму стрічку (як Sitniks/KeepinCRM), відповідь публічно або переведення в Direct.
  - _Технічно:_ активувати `getPlatformSender()` для `instagram` (зараз mock), новий `app/api/instagram/webhook/route.ts`, дзеркалить наявні telegram/viber вебхуки. Потрібен бізнес-верифікований Meta App + токени.

- **3.2. WhatsApp Business:**
  - Через WhatsApp Cloud API (Meta) — активувати наявну заготовку `whatsapp` у `platform-send.ts`.
  - Шаблони повідомлень (WhatsApp вимагає pre-approved templates для першого контакту).

- **3.3. Покращення інбоксу під нові канали:**
  - Уніфіковані **швидкі відповіді/шаблони з фото товару зі складу** (як Sitniks — надіслати картку товару з наявністю прямо в чат).
  - Авто-створення ліда + картки клієнта з першого повідомлення (є база — розширити на нові канали).

> **Facebook Messenger** — «безкоштовний бонус» цього блоку (той самий Meta API), додати за наявності часу. **TikTok** — повноцінного DM-API немає навіть у лідерів (Sitniks має лише TikTok Ads/ліди-форми); лишаємо на потім.

**Результат:** усі повідомлення з усіх каналів — в одному вікні; Instagram-ліди більше не губляться в телефоні менеджера.

### Блок 4. Банк-виписки (Monobank + ПриватБанк) + автозвірка боргу

**Навіщо:** рух коштів зараз **вводиться руками**. KeepinCRM тягне виписки Приват/Моно **кожні 7 хвилин**, KeyCRM робить **автозвірку оплати прямо в замовленні**. У нас сильний облік боргів — і це **ідеально лягає**: автоматично бачити, хто заплатив, і закривати борг.

**Що робимо:**

- **4.1. Monobank (найпростіше — почати з нього):**
  - Monobank API для бізнесу/ФОП (`https://api.monobank.ua/`, персональний токен): підтягування виписки `statement`.
  - Нові транзакції → у наш регістр ДДС (`CashFlowMovement`) + у розділ «Банківські транзакції».
  - _Технічно:_ `lib/banking/monobank.ts`, cron `api/cron/pull-bank-statements`, нова модель `BankTransaction` (сирі транзакції з банку) з прив'язкою до `BankPaymentIncoming`.

- **4.2. ПриватБанк (Приват24 для бізнесу / автоклієнт-банк):**
  - API виписок ПриватБанку (merchant + token). Аналогічна логіка.

- **4.3. Автозвірка з боргом клієнта:**
  - У картці клієнта / реалізації: показати непов'язані вхідні платежі з виписки → менеджер (або авто за призначенням платежу/сумою) прив'язує платіж → **автоматично гаситься борг** (`MgrDebtMovement`).
  - Це наш козир: у конкурентів звірка з «замовленням», у нас — з **живим регістром боргу**.

**Результат:** менеджер бачить оплати клієнтів автоматично; борги закриваються без ручного вводу; звіт ДДС наповнюється сам.

## ПАРАЛЕЛЬНО (наскрізно) — Блок 5. Освіження інтерфейсу (UI/UX)

Робимо **точково**, не переписуючи систему. Кожен пункт — окрема невелика задача, впроваджуємо поступово.

- **5.1. 3-панельний чат-інбокс:** зліва список діалогів+фільтри | центр переписка | **справа — картка клієнта (борг, останні замовлення, історія)**. Зараз у нас 2 панелі — додати праву панель контексту (головний прискорювач за оглядами Intercom/HubSpot).
- **5.2. Швидкий пошук Ctrl+K** (командна палітра): один вхід з клавіатури — знайти клієнта/замовлення/**лот за штрихкодом**, стрибнути на документ. У шапці вже є пошук/сканування — зробити повноцінну палітру.
- **5.3. Картка замовлення/реалізації — менше кліків:** кнопка ТТН прямо на картці (з Блоку 1), inline-редагування ключових полів, **прогресивне розкриття** (головне спершу, історія/документи — за вкладками).
- **5.4. Порожні стани (онбординг):** там, де список порожній — пояснення + **одна очевидна кнопка** («Створити перше замовлення»). Критично для нетехнічного користувача.
- **5.5. Дашборд-полір:** 3 ряди (KPI → воронка по стадіях → активність) + **лідерборд торгових агентів** по продажах (у нас є рольові дашборди — додати воронку й лідерборд).
- **5.6. Загальна візуальна модернізація:** поступово відходити від «важкого 1С-вигляду» вкладок до чистіших списків/карток (не ламаючи звичну логіку для тих, хто працював у 1С).

## ХВИЛЯ 3 — умовно (за вашим рішенням)

### Блок 6. Checkbox / ПРРО (фіскальні чеки)

**Навіщо / питання до вас:** усі укр. CRM мають Checkbox (авто-чек за подією). **АЛЕ** для **гуртового B2B** (продаж від 10 кг юрособам/ФОП за безготівкою й рахунками) фіскальний чек часто **не потрібен** — потрібен рахунок/видаткова накладна (це у нас **уже є**, з друком). Checkbox критичний, якщо ви приймаєте **готівку/картку від кінцевих покупців** і зобов'язані видавати фіскальний чек.

**Рішення потрібне від вас:** чи є у вас операції, що вимагають ПРРО-чека? Якщо так — робимо:

- **6.1.** Інтеграція Checkbox API: каса/касир/зміна, фіскалізація оплати з картки реалізації, чек клієнту в месенджер/email.
- **6.2.** Авто-фіскалізація за тригером (статус «Оплачено»/«Отримано») + авто-повернення чека.
- _Технічно:_ `lib/fiscal/checkbox.ts`, поля `Sale.fiscalReceiptId`/`fiscalStatus`. Бенчмарк — LP-CRM (керування змінами) + SalesDrive (автофіскалізація при отриманні на НП).

_Якщо ПРРО не потрібне — цей блок пропускаємо, це нормально для чистого гурту._

## ХВИЛЯ 4 — довші проєкти

### Блок 7. Полір обліку + нативний мобільний застосунок

- **7.1. Матриця маржа×оборотність** (запозичити в KeepinCRM): звіт, де видно товари в осях «маржа × швидкість обороту» — що приносить гроші, що «мертвий вантаж». У нас є всі дані (є `CostMovement`, `StockMovement`) — це новий звіт поверх наявних регістрів.
- **7.2. Резерв-картка «В наявності / Заброньовано / Доступно»** на товарі/лоті (як KeepinCRM/Sitniks) — у нас є бронювання лотів, зробити наочний зведений показник.
- **7.3. Спрощення звітів:** наші звіти сильні, але «важкі як 1С» — додати простіші візуальні зведення (графіки, а не лише таблиці) для власника.
- **7.4. Нативний мобільний застосунок для менеджера** (Sitniks — головна фішка): обробка замовлень, чати, створення ТТН, **офлайн-сканування мішків на складі** (зберегти локально → синк при зв'язку). Зараз у нас адаптивний веб; нативний застосунок — окремий великий проєкт (є база `apps/mobile-client` для клієнтів — можна відштовхнутися).

## Підсумок — з чого почати

**Рекомендований порядок першого кварталу:**

1. **Нова Пошта API** (Блок 1) — найбільша щоденна економія часу, чіткий обсяг.
2. **Воронка-канбан** (Блок 2) — не губимо потенційних клієнтів, видима цінність для власника.
3. Паралельно — **UI-пункти 5.1–5.4** (права панель чату, Ctrl+K, порожні стани) — швидкі, помітні покращення.

Далі — **Instagram Direct** (Блок 3) і **банк-виписки** (Блок 4).

**Що НЕ робимо:** не мігруємо на готову CRM і не переписуємо ядро. Наша сила — ваговий гурт + управлінський облік + маршрути — залишається, а по інтеграціях і вигляду ми поступово стаємо на рівень (а місцями вище) українських лідерів, під наш унікальний бізнес.

### Технічні нотатки для розробки (звід)

- **Нові моделі (additive міграції):** `NpCity`, `NpWarehouse`, `BankTransaction`, `Pipeline`, `PipelineStage`; нові поля на `Sale` (`npRef/npTtn/npCost/deliveryStatusNp`, за потреби `fiscalReceiptId/fiscalStatus`), `MgrLead.stageId`.
- **Нові сервіси/lib:** `lib/delivery/nova-poshta.ts` + `create-waybill.ts`; `lib/banking/{monobank,privatbank}.ts`; `lib/fiscal/checkbox.ts`; активувати `instagram`/`whatsapp` у `lib/chat/platform-send.ts`.
- **Нові вебхуки:** `app/api/instagram/webhook`, `app/api/whatsapp/webhook` (дзеркалять telegram/viber).
- **Нові cron:** `pull-bank-statements`, `track-np-status` (як наявні cron-задачі, той самий `CRON_SECRET`).
- **UI:** канбан на `@dnd-kit`; командна палітра Ctrl+K; права панель у чат-інбоксі.
- **Секрети (`.env`):** `NOVA_POSHTA_API_KEY`, `MONOBANK_TOKEN`, `PRIVAT_*`, `META_APP_*` (Instagram/WhatsApp), `CHECKBOX_*`.
- **Принцип незмінності:** усі зміни additive; магазин, історичний імпорт, ядро обліку — не чіпаємо.
