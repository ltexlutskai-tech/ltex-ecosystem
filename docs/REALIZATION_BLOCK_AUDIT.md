# Блок «Реалізація» — повний аудит 1С (перед розробкою)

> **Документ 1С:** `Document.РеализацияТоваровУслуг` (Реалізація товарів і послуг / sales act).
> **Маршрут у нас:** `/manager/sales` (зараз stub `UnderConstruction session="M1.6"`).
> **Рішення user (зафіксовано):** Реалізація — **окремий документ** (не породжується із Замовлення; кнопка «Додати» одразу створює нову), що фіксує **факт продажу/відвантаження** (товар відвантажено клієнту → формує борг клієнта, списує зі складу).
> **Джерела:** мобільна форма (`MobileAgent/Documents/РеализацияТоваровУслуг/*`, форма документа 1452 рядки логіки + 1581 XML), центральна база (`Central/Documents/РеализацияТоваровУслуг/Ext/ObjectModule.bsl` 5891 рядок + схема), модулі обміну `Central/CommonModules/ОбменАРМ` (7635 рядків).

## 0. Що таке Реалізація і її місце в системі

Реалізація — це **точка-продажу/акт відвантаження** менеджера: менеджер у клієнта (контрагента) сканує штрихкоди мішків, ставить ціни, зберігає продаж. У центральній 1С це стандартний документ УТ «Реализация товаров и услуг», який при проведенні:

- **списує товар зі складу** (партійний облік, FIFO);
- **збільшує борг клієнта** (взаєморозрахунки);
- записує асортимент покупця;
- реєструє контрагента для зворотного обміну (щоб документ повернувся на пристрій агента).

У старій 1С мобільній Реалізація зазвичай **створюється з маршрутного листа** (поля `Сделка`=Замовлення, `МаршрутныйЛист`, `Неплановая`, `Наложка` приходять як параметри), а кнопка **Підбір** тягне товари, фізично завантажені в авто для цього маршруту. **У нас (рішення user) — окремий документ**, тож зв'язок з маршрутом/замовленням робимо як необов'язкові посилання, а Підбір — через прайс (як у Замовленнях), доки не буде блоку Маршрутів та реальних обмінів.

**Межа блоку:** глибокі **Оплати/Каса** (`КассовыйОрдер`, здача в 3 валютах, прихідні ордери) — це **окремий наступний блок**. Тут, у Реалізації, лише поверхневі елементи оплати: прапор **Наложка** (післяплата) + обчислення **суми післяплати в грн**, а кнопки «Створити оплату / Відкрити оплати» делегуються блоку Оплати.

**Підсумки футера:** Кількість (Підсумок) + Сума (Підсумок).
**Додавання рядків:** лише через **скан ШК / Підбір / швидкий ШК** — НЕ можна друкувати у порожній рядок (`BeforeAddRow` скасовано). Колір тексту — зелений.
**Контекст-меню:** «Повторити ціну» — копіює ціну/кг поточного рядка на всі рядки тієї ж номенклатури.

**Перерахунок (на зміну/збереження):**
```
для кожного рядка (крім Знижка):
   вага = Характеристика.Вага  (0 → парс(Характеристика) → 20)
   Ціна = ЦінаЗаКг × вага
   Сума = Ціна × Кількість
Сума_документа = Σ Сума
```

## 3. Логіка цін

1. **Ціна за кг** береться з регістру цін за ключем `(Номенклатура, Характеристика, ТипЦін)`; якщо для конкретної характеристики нема — fallback на ціну з **порожньою** характеристикою (= базова ціна товару). У нас: `Price` за `MgrPriceType.code === Price.priceType`, характеристика ≈ лот.
2. **Вага** — з характеристики (мішок). 0 → парс назви → **дефолт 20 кг**.
3. Похідні: `Ціна = ЦінаЗаКг × вага`; `Сума = Ціна × Кількість`; `Сума_док = Σ Сума`.
4. **Зміна типу цін** перераховує `ЦінаЗаКг` усіх рядків.
5. **Валюта:** усі ціни — **EUR**. Курс EUR/USD — знімок на документі. Грн рахується для показу/післяплати: `грн = EUR × курсEUR`.
6. **Округлення:** післяплата — до цілих грн. Рядки/сума — 2 знаки. Кроку ціни (0.05) **тут немає** (на відміну від Замовлень).
7. **Контроль відхилення ціни** (при збереженні): MIN ціна з регістру за типом цін; якщо `ЦінаЗаКг` відхиляється > **0.20 EUR** — попередження «ціна має бути X, є Y» + «Дійсно зберегти продаж?».

## 5. Скан ШК — потік

1. `ПолучитьДанныеТовараПоШК(ШК, ТипЦін)` → за штрихкодом знаходить {Номенклатура, Характеристика, Одиниця, Ціна за кг, Залишок}. Порожньо → «Не найден товар».
2. **Перевірка броні** `АктивнаБроньМішка`: якщо мішок заброньовано **не мною** і бронь активна → «Активна бронь мішка до …» + відмова. (Своя бронь не блокує.)
3. **(тільки десктоп)** перевірка залишку: `Залишок ≤ 0` → «Товару немає на залишку». На мобільному пропущено.
4. `ДодатиТовар(...)`: якщо рядок Номенклатура+Характеристика вже є → «Товар уже добавлен!» (без злиття). Інакше — новий рядок, вага з характеристики, ціна з типу цін.

## 6. Список (ФормаСписка)

- Колонки: **Дата** + **Номер** (групою), **Контрагент**, **Місто** (= `Контрагент.Город`), **Сума**.
- Тулбар: чекбокс **«Відображати архівні»** + кнопка **«Додати»** (одразу нова Реалізація).
- За замовч. фільтр `Архів = false` (архівні приховані). Архівні рядки — фон **світло-сірий**.
- У центральній базі список багатший: Номер, Дата, Проведено, Організація, Контрагент, **Торговий агент**, Маршрут, Сума, **Доставка**, Відповідальний, **Експрес-накладна** (ТТН). Червоні рядки — непроведені/проблемні.

## 7. Viber/чат — повідомлення (3 кнопки)

Усі три спершу `ПеревіркаПовноїОплати` (якщо Доставка=Почта і НЕ Наложка і є борг → «створити реалізацію без повної оплати?»), перераховують суми, будують текст `ПолучитьТекстСообщенияВайбер`:

- **Контрагенту** (клієнту): ім'я + регіон/місто/тел; рядок доставки; позиції `[Назва] к-ть×вага×ціна/кг = сума`; підсумки EUR+грн; оплачено/решта; борг/передплата; післяплата.
- **В групу** (внутрішнє): з артикулом, ШК (для Почти), сектором складу, коментарем, датою. **Вмикає експорт у 1С + зберігає.**
- **У чат** (бот-вихідні): запис у регістр `ИсходящиеСообщения` (отримувач — чат із `ГруппыЧата`).

## 8. Збереження/проведення

**Зберегти** = провести+закрити. Валідації перед записом:
1. `ПеревіритиРекомендації` — рекомендовані товари не додані → «Записати?».
2. Контроль відхилення ціни (>0.20 EUR).
3. Перевірка рядків: усі {Номенклатура, Характеристика, Кількість, Ціна} заповнені; порожні рядки авто-видаляються, частково заповнені → помилка.
4. `Архів = false`.

**Гейт експорту** (`ПриЗаписи`): якщо `ВивантажуватиВ1С` → реєструє документ у регістрі обміну (наступний sync завантажить у центральну 1С); інакше — знімає реєстрацію.

## 9. Центральна база — схема й проведення

**Стандартний УТ «Реализация товаров и услуг»** + кастом L-TEX. Ключові кастом-поля шапки: `МаршрутныйЛист`, **`ТорговийАгент`**, **`КурсEUR`/`КурсUSD`**, **`Наложка`**, **`Доставка`** (enum), **`НомерВідділенняНП`**, **`Проверена`**, **`НомерЭкспрессНакладной`** (ТТН), **`ВидОплаты`**, **`СумаОплатиНаложкою`**. Рядок `Товары` додатково має **`ЦенаПродажиВес`** (ціна/кг), **`Знижка`**, **`ЗаказПокупателя`** (рядкове посилання на замовлення).

**Проведення:** рух по регістрах — взаєморозрахунки (борг клієнта ↑), товари організації, **списані товари (склад ↓, партійний FIFO)**, ПДВ-продаж; + запис асортименту покупця + реєстрація контрагента для обміну.

**Зв'язок Замовлення→Реалізація** (у центральній існує, але **у нас не використовуємо** за рішенням user): через `Сделка` (шапка) + `Товары.ЗаказПокупателя` (рядок); central авто-зливає в наявну реалізацію того ж замовлення за 7 днів.

**Призначення агента** на імпорті: `ТорговийАгент = ?(ПризначитиПродажТорговомуКонтрагента, агент_клієнта, поточний_продавець)`.

## 10. Обмін (sync) — контракт

> **Критично:** транспорт центральної 1С — **бінарний `ValueStorage`** (deflate-9), НЕ JSON/XML. Веб-клієнт напряму так не вміє → потрібен JSON-шар на боці 1С (BSL пишемо самі **на етапі обмінів, наприкінці**, як і вирішено стратегічно). Зараз — як у Замовленнях: черга `MgrSyncJob` + mock-проксі.

- Бізнес-ключ/ідемпотентність — **GUID документа** (`Ссылка`/`УІД`) + токен `ИД` з регістрації обміну.
- Реалізація передається як набір рядків (групуються по `Ссылка`). Поля рядка/шапки — у звіті вище (Номенклатура, Характеристика, Кількість, Ціна, ТипЦін, Курс EUR/USD, Наложка, Доставка, СумаОплатиНаложкою, ПризначитиПродажТорговомуКонтрагента, тощо).
- **Лот у central не приймається точно** для замовлень — у Реалізації рядок несе характеристику (мішок), але оскільки наш central поки не готовий, пишемо **загальні позиції** (`lotId=null`), точний лот — після обмінів (узгоджено з інваріантами проекту).

## 11. Що вже є на `main` (для перевикористання)

## 12. Ключові відмінності Реалізація vs Замовлення (для плану)

| Аспект                 | Замовлення (готово)                | Реалізація (новий блок)                                                                 |
| ---------------------- | ---------------------------------- | --------------------------------------------------------------------------------------- |
| Суть                   | намір купити                       | **факт продажу/відвантаження** (борг, списання складу)                                  |
| Курс валют             | один `exchangeRate`                | знімок **EUR + USD**; блок створення якщо курс порожній                                  |
| Знижка/рядок           | нема                               | **`Знижка`** (ручна ціна зберігається при перерахунку)                                   |
| Призначення агента     | «призначити продаж торговому»      | те саме, але дефолт **TRUE** (агент клієнта)                                             |
| Експорт у 1С           | `exportTo1C` дефолт TRUE           | у 1С дефолт **FALSE**, вмикається після Viber-відправки                                  |
| Оплата                 | окремо                             | **Наложка** + сума післяплати грн вбудовані; «Створити/Відкрити оплати» → блок Оплати    |
| Скан ШК                | не акцентувалось                   | **центральний спосіб додавання рядків** (мішок за штрихкодом)                            |
| Підбір                 | за прайсом                         | у 1С — з маршрутного листа; **у нас — за прайсом** (до блоку Маршрутів)                  |
| Експрес-накладна (ТТН) | нема                               | поле `НомерЭкспрессНакладной` (+CheckBox ETTN у central — окрема інтеграція, не зараз)   |
| Viber 3 кнопки         | —                                  | Контрагенту / В групу / У чат (генерація тексту + «Поділитися», як у Прайсі)             |
