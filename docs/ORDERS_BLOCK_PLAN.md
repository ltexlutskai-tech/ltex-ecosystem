# Блок «Замовлення» — план розробки (менеджерська програма)

> **Мета:** відтворити функціонал екрана «Замовлення» (Document.Заказ) зі старого
> 1С-додатку, новим інтерфейсом у `/manager/orders/*`. Працюємо на `main`.
>
> **Статус:** ЧЕРНЕТКА на ревʼю user-а.
>
> **Особливість блоку:** частина вже існує (M1.4 список+перегляд, M1.5b створення),
> тож план — переважно **доповнення** наявного під 1С, а не з нуля.
>
> Інвентар старого 1С — у листуванні (форми `Documents/Заказ/Forms/{ФормаДокумента,ФормаСписка,...}`).

---

## 0. Уточнення user (враховано в плані)

1. **«Архів»** = замовлення, які **відправлені й проведені в 1С** (статус після успішного обміну), а не «видалене». Поки обмін у mock-режимі — архівних не буде; заповниться, коли підключимо реальний обмін.
2. **«Скопіювати»** — **НЕ робимо** (прибрано з плану).
3. **«Призначити продаж торговому контрагенту»** = зарахувати продаж **іншому торговому агенту** (не тому менеджеру, що веде клієнта). Тобто поле — вибір агента, кому йде продаж.
4. **«Нагадування»** на рядку товару = автоматично створити нагадування, **коли товар наступного разу прийде на склад** (з документа «Приход»). Це окремий складніший механізм (приход → нагадування) — **ВІДКЛАДАЄМО**, зробимо пізніше окремим блоком.

## 2. Що треба додати в базу

Розширюємо `Order` новими полями (необов'язкові, магазину не заважають):

- **тип цін** (priceTypeId — з довідника `MgrPriceType`),
- **доставка** (deliveryMethod — Доставка/Пошта/Самовивіз),
- **наложка** (cashOnDelivery — так/ні),
- **торговий агент продажу** (assignedAgentUserId — кому зараховано продаж),
- **вивантажувати в 1С** (exportTo1C — так/ні, дефолт так),
- **архів** (archived — проведено в 1С; ставиться при успішному обміні, поки завжди false),
- **актуальний** (isActual — активне замовлення).

(Маршрутний лист — поле додамо разом з блоком «Маршрути».)

## 3. Етапи розробки

### Етап 1 — Доповнити форму створення замовлення під 1С

**Що бачить менеджер:** та сама форма `/manager/orders/new`, але з повним набором полів як у 1С.

> ✅ **ГОТОВО (2026-05-21).** До `Order` додано 7 полів (priceTypeId/deliveryMethod/cashOnDelivery/assignedAgentUserId/exportTo1C/archived/isActual, міграція `20260524_order_manager_fields`). Форма: блок «Параметри замовлення» — Тип цін (select, перерахунок рядків через `lib/manager/order-pricing.ts`: `MgrPriceType.code`===`Price.priceType`), Доставка, Наложка, «Призначити продаж торговому» + агент (дефолт=поточний менеджер), «Вивантажувати в 1С». Вибір клієнта підтягує тип цін/борг/доставку з `MgrClient`. +25 тестів, сьют 1146 pass, typecheck чистий.

- Додати **тип цін** (вибір; при зміні — **перерахунок цін усіх рядків**).
- Додати **доставку**, **наложку**, **«призначити продаж торговому»** (вибір агента) + **«вивантажувати в 1С»**.
- При **виборі клієнта** — автоматично підтягнути його тип цін, борг (показати), склад, спосіб доставки.
- Підбір товарів/лотів — є; за потреби покращити (вимога вибрати клієнта+тип цін перед підбором, як у 1С).
- Підсумки грн + € — є.

### Етап 2 — Редагування замовлення + статуси

- Зробити сторінку `/manager/orders/[id]` **редагованою** (зараз лише перегляд): міняти товари/кількість/поля, зберігати.
- **Статуси** замовлення: чернетка → відправлено в 1С → проведено (=архів). Поки обмін mock — статуси ставляться вручну/каркасно; реальний перехід «проведено» — на етапі обмінів.

### Етап 3 — Доповнити список замовлень

- Колонки **Місто** (з картки клієнта) + **Актуальний**.
- Фільтр **«Відображати архівні»** (за замовчуванням архівні = проведені в 1С приховані).
- **Пошук по товарах** усередині замовлень (які замовлення містять товар за артикулом/назвою) + по клієнту.
- Кольори: архівні — приглушені.
- (Без «Скопіювати» — за рішенням user.)

### Етап 4 — ВІДКЛАДЕНО: нагадування з приходу

Окремий механізм: документ «Приход» товару → коли позначений товар знову надходить на склад, автоматично створюється нагадування менеджеру. Потребує: модель приходу + звʼязок «нагадати про товар» + тригер при надходженні. **Зробимо пізніше окремим блоком** (user підтвердив). Колонку «Нагадування» на рядку замовлення додамо разом із цим механізмом.

## 5. Рішення для підтвердження (або зміни)

1. **Тип цін** при зміні **перераховує ціни всіх рядків** автоматично (як 1С) — ок?
2. **Торговий агент** для «призначити продаж торговому» — вибір зі списку менеджерів (`User`). Дефолт — поточний менеджер. Ок?
3. **Доставка** — три варіанти (Доставка / Пошта / Самовивіз), як 1С. Ок?
4. **Маршрутний лист** у замовленні — додамо **разом із блоком «Маршрути»** (поки поле в базі є, у формі прихово/заглушка). Ок?
5. **Статуси замовлення** — пропоную: «Чернетка» / «Відправлено в 1С» / «Проведено (архів)» / «Скасовано». Назви ок чи свої?
6. **Мінімум «від 10 кг»** — у 1С НЕ перевіряється. Лишаємо без перевірки, чи додати попередження? (Рекомендую без жорсткої перевірки, як 1С.)
